In the last section, we discussed the concept of scalability and how it relates to the ability of a system to handle increased load.

But, a system that can handle a million of users is not very useful if it goes down every other day.

This is where the concept of 'availability' comes in.

Simply put, 'Availability' is the measure of a system's uptime, or how often it is operational and accessible to users. A system is called highly available if it continues to function even in the face of failures, such as hardware malfunctions, software bugs, or network issues.

Now, do not confuse 'Availability' with 'Reliability'. While they are related, they are not the same thing.

'Reliability' refers to the ability of a system to perform its intended function consistently without failure over time, while 'Availability' focuses on the system's uptime and accessibility.

A system can be highly available but not necessarily reliable. For example, a website that is online but serves frequent errors. In this case, the website is available but not reliable.

In the las section, we saw how we can measure 'Scalability' as there are various metrics like 'Throughput', 'Latency', and 'Resource Utilization'.

What about 'Availability'? How do we measure it?

There is actually a formula to calculate the availability of a system -

    Availability = (Uptime / (Uptime + Downtime)) * 100%

Where 'Uptime' is the amount of time the system is operational and accessible, and 'Downtime' is the amount of time the system is not operational or accessible.

So basically, we are calculating the percentage of time the system is available to users.

So, if a system was up for 300 days and down for 5 days, the availability would be -

    Availability = (300 / (300 + 5)) * 100% = 98.36%

You might've heard of terms like Three nines or Five nines when it comes to availability. These terms refer to the percentage of uptime a system has.

Availability is often described in terms of 'nines' to indicate the level of uptime.

    - 'Two nines' (99% availability) means the system is down for about 3.65 days per year.
    - 'Three nines' (99.9% availability) means the system is down for about 8.76 hours per year.
    - 'Four nines' (99.99% availability) means the system is down for about 52.56 minutes per year.
    - 'Five nines' (99.999% availability) means the system is down for about 5.26 minutes per year.
    - 'Six nines' (99.9999% availability) means the system is down for about 31.5 seconds per year.

So, the more nines, the higher the availability and the less downtime the system experiences.

# AVAILABILITY IN SERIES VS PARALLEL

When designing a system for high availability, it's important to consider how components are arranged. There are two common configurations: series and parallel.

In a series configuration, components are arranged in a linear fashion, where the failure of any single component can lead to the failure of the entire system. For example, if you have a web server, a database server, and an application server all in series, and the database server fails, the entire system will be unavailable.

This means, in a series, the availability of the system is the product of the availability of each component. If each component has an availability of 99%, then the overall availability of the system would be:

    Overall Availability = 0.99 * 0.99 * 0.99 = 0.970299 or 97.03%

So, this mean each component in the chain is reducing the overall availability of the system. Even though each component has a high availability, the overall system availability can be significantly lower due to the series configuration.

In a parallel configuration, components are arranged in such a way that if one component fails, another can take over its function. This is often achieved through redundancy. For example, if you have two web servers in parallel, and one fails, the other can continue to serve requests, thus maintaining the availability of the system.

# HOW DO SYSTEMS FAIL?

Systems can fail in various ways, and understanding these failure modes is crucial for designing highly available systems.

The most common type is the hardware failures. For example, a hard drive might fail, a power supply might go out, or a network card might stop working. These failures can lead to downtime if there are no redundant components in place.

Now, while hardware failures are random, software failures can be more predictable. They can occur due to bugs in the code, memory leaks, or unhandled exceptions. For instance, if a web application has a memory leak, it might consume all available memory over time and crash, leading to downtime.

Then, there are also network failures, which can occur due to issues with routers, switches, or even the internet service provider. These failures can lead to loss of connectivity and make the system inaccessible to users.

Finally, we have human errors, which can be a significant cause of system failures. This can include misconfigurations, accidental deletions, or even mistakes during maintenance. For example, if an administrator accidentally deletes a critical database, it can lead to significant downtime.

# REDUNDANCY

Redundancy is the practice of adding extra components or resources to a system to ensure that it can continue to function even if one or more components fail.

Redundancy is like the foundation of high availability. By having redundant components, we can ensure that if one component fails, another can take over its function, thus maintaining the availability of the system.

## ACTIVE-PASSIVE REDUNDANCY (STANDBY)

In an active-passive redundancy configuration, one component is active and handling the workload, while the other component is passive and on standby. If the active component fails, the passive component can take over its function.

For example, in a web server setup, you might have two servers where one is active and serving requests, while the other is passive and ready to take over if the active server fails. This setup can provide high availability, but it may not be as efficient since the passive component is not utilized until a failure occurs.

Also, in case of failover, there might be a brief downtime as the passive component takes over, which can affect the availability of the system.

But, the benefit of this approach is that it is simpler to implement.

This is a good approach for systems that require consistent data state and some downtime is acceptable during failover.

A standby can be configured to be -

    1. Cold standby: The passive component is not running and is only started when the active component fails. This can lead to longer downtime during failover. So, the passive component is literally powered off and needs to boot up when the active component fails. This one incurs the longest downtime during failover since the passive component needs to boot up and initialize before it can take over the workload.

    2. Warm standby: The passive component is running and ready to take over, but it is not actively handling the workload. This can lead to shorter downtime during failover compared to cold standby. This allows for a quicker failover compared to cold standby since the passive component is already running and can take over more quickly when the active component fails.

    3. Hot standby: In this configuration, the passive component is already running and actively handling the workload, so failover can occur with minimal downtime since the passive component can take over almost immediately when the active component fails. The data is synchronized between the active and passive components in real-time, ensuring that there is no data loss during failover. This approach provides the highest level of availability but can be more complex and costly to implement due to the need for real-time synchronization and resource utilization.

So, cold standby is the cheapest but it is also the slowest.

Hot standby is the most expensive but it provides the best availability. For example, in case of databases, we might have a primary database that is active and a secondary database that is passive. The secondary database can be configured to replicate data from the primary database in real-time, ensuring that it is always up-to-date and ready to take over if the primary database fails.

Warm standby is a middle ground between the two.

## ACTIVE-ACTIVE REDUNDANCY

Here, both components are active and handling the workload simultaneously. If one component fails, the other can continue to handle the workload without interruption.

Let's say we have three web servers in an active-active configuration. All three servers are actively serving requests, and if one server fails, the remaining two servers can continue to handle the workload without interruption. This approach can provide higher availability compared to active-passive redundancy since there is no downtime during failover.

But, the issue is that this is complex to implement and can be more expensive since all components are actively handling the workload. Additionally, it requires careful load balancing and synchronization to ensure that all components are working together effectively.

It is usually used for stateless services where each request can be handled by any of the active components without the need for synchronization. For example, in a web application, you might have multiple web servers in an active-active configuration, where each server can handle incoming requests independently. This allows for high availability and scalability, as the workload can be distributed across multiple servers without the need for failover.

If services are stateful, then active-active redundancy can be more challenging to implement since it requires synchronization of state between the active components. In this case, you might need some short of shared storage (like database or redis) or sticky sessions to ensure that requests from the same user are consistently handled by the same active component. This can add complexity and overhead to the system, but it can still provide high availability if implemented correctly.

## GEOGRAPHIC REDUNDANCY

Well, what if the entire data center goes down due to a natural disaster or power outage? This is where geographic redundancy comes in.

Geographic redundancy involves having redundant components or resources located in different geographic locations. This can help ensure that if one location experiences a failure, the other location can continue to provide service.

For example, in AWS, there is a concept of 'Availability Zones' (AZs) and 'Regions'. An AZ is a physically separate data center within a region, and a region is a collection of AZs. By deploying resources across multiple AZs or regions, you can achieve geographic redundancy and improve the availability of your system.

If you have a global application, you might want to deploy your resources in multiple regions to ensure that users can access the application even if one region experiences a failure. This can also help reduce latency for users by serving them from the closest region.

One important thing to note about redundancy is that it gets harder as you move down the stack. For example, it is easier to have redundant web servers than it is to have redundant databases. This is because databases often require synchronization and consistency, which can be more complex to achieve in a redundant configuration. Moreover, Redundancy is not free. It can add cost and complexity to the system, so it's important to carefully consider the trade-offs when designing for high availability. If the added cost is not justified by the benefits of increased availability, it may not be worth implementing redundancy in certain components of the system.

# HIGH AVAILABILITY PATTERNS

There are some common patterns that are used to achieve high availability in systems. Let's discuss a few of them.

## LOAD BALANCER WITH MULTIPLE SERVERS

This is probably one of the most common patterns for achieving high availability. In this pattern, you have a load balancer that distributes incoming traffic across multiple servers. If one server fails, the load balancer can redirect traffic to the remaining healthy servers, ensuring that the system remains available to users.

A load balancer continuously monitors the health of the servers and can automatically remove any failed servers from the pool of available resources. This allows for seamless failover and helps maintain high availability.

But, now you can also say that the load balancer itself is a single point of failure. To mitigate this, you can have multiple load balancers in an active-passive or active-active configuration to ensure that if one load balancer fails, the other can take over its function.

## DATABASE REPLICATION

Unlike web-servers, databases are stateful and require synchronization to ensure data consistency. We have to carefully replicate data between the primary and secondary databases to ensure that they are in sync.

The replication can either be synchronous or asynchronous.

In synchronous replication, the primary database waits for the secondary database to acknowledge that it has received and applied the changes before it considers the transaction complete. This ensures strong consistency but can lead to increased latency. If your replica is in a different region, the latency can be even higher due to the distance between the primary and secondary databases.

In asynchronous replication, the primary database does not wait for the secondary database to acknowledge the changes. This can lead to lower latency but may result in data loss if the primary database fails before the changes are replicated to the secondary database.

There is also semi-synchronous replication, which is a middle ground between the two. In this approach, the primary database waits for at least one secondary database to acknowledge the changes before it considers the transaction complete. This can provide a balance between consistency and performance.

## QUEUES

Message queues like Kafka, SQS, or RabbitMQ can also be used to achieve high availability. By using a queue, you can decouple the components of your system and allow them to communicate asynchronously.

This pattern is used commonly for bursts of traffic. For example, if you have a web application that receives a high volume of requests, you can use a queue to buffer the incoming requests and process them asynchronously. This can help prevent your application from becoming overwhelmed and ensure that it remains available to users.

## CIRCUIT BREAKER

A circuit breaker is a design pattern that helps prevent cascading failures in a system. It works by monitoring the health of a component and 'tripping' if it detects a failure. When the circuit breaker is tripped, it can redirect traffic to a fallback mechanism or return an error response to the user.

This is important because if a dependency fails, if we keep calling it, we are simply wasting resources and potentially making the situation worse. By using a circuit breaker, we can prevent this and allow the system to degrade gracefully instead of failing completely.

A circuit breaker can either be: 

    - closed: where the circuit is functioning normally and allowing traffic to pass through.
    - open: where the circuit is tripped and blocking traffic from passing through.
    - half-open: where the circuit is in a testing state, allowing a limited number of requests to pass through to check if the underlying issue has been resolved.

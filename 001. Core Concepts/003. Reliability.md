In the last section, we talked about 'Availability' which is a measure of how often a system is up and running.

A lot of times, we confuse 'Availability' with 'Reliability'. But they are not the same thing.

A system is considered 'Reliable' if it can perform its intended function without failure for a specified period of time.

So, if with 'Availability' we answer the question "Is the system up?", with 'Reliability' we answer the question "Does the system work correctly?".

An example of a system that is highly available but not reliable could be a website that is always up but frequently returns errors or incorrect data. Conversely, a system that is reliable but not available could be a service that works perfectly when it's up, but it goes down frequently.

An unreliable system can lead to a poor user experience, loss of trust, and potential financial losses. Therefore, it's crucial to design systems that are both highly available and reliable to ensure a positive user experience and maintain the integrity of the service.

A system might aggressively cache data to improve performance and availability, but if the cached data is stale or incorrect, it can lead to reliability issues. On the other hand, a system that prioritizes reliability might implement strict data validation and error handling, but if it doesn't have proper redundancy and failover mechanisms, it could suffer from availability issues.

# MEASURING RELIABILITY

So, how do we measure reliability?

There are some common metrics used to measure reliability.

## 1. MEAN TIME BETWEEN FAILURE (MTBF)

As the name implies, MTBF is the average time between failures of a system. It is calculated by dividing the total operating time by the number of failures -

    MTBF = Total Operating Time / Number of Failures

For example, if a system operates for 1000 hours and experiences 5 failures, the MTBF would be:

    MTBF = 1000 hours / 5 failures = 200 hours

This means, the system will continue to operate without failure for an average of 200 hours. In other words, you will see a failure every 200 hours on average.

## 2. MEAN TIME TO RECOVERY (MTTR)

MTTR is the average time it takes to recover from a failure. So, the lower the MTTR, the more reliable the system is. It is calculated by dividing the total downtime by the number of failures -

    MTTR = Total Downtime / Number of Failures

For example, if a system experiences 5 failures and the total downtime is 50 hours, the MTTR would be:

    MTTR = 50 hours / 5 failures = 10 hours

This means, on average, it takes 10 hours to recover from a failure.

## 3. ERROR RATE

Error rate is the percentage of requests that result in an error. It is calculated by dividing the number of errors by the total number of requests and multiplying by 100 to get a percentage -

    Error Rate = (Number of Errors / Total Number of Requests) * 100

For example, if a system receives 1000 requests and 50 of them result in errors, the error rate would be -

    Error Rate = (50 errors / 1000 requests) * 100 = 5%

## 4. DATA CORRECTNESS

Data correctness is a measure of how often the system returns correct data. It can be calculated by dividing the number of correct responses by the total number of responses and multiplying by 100 to get a percentage -

    Data Correctness = (Number of Correct Responses / Total Number of Responses) * 100

This metric is often overlooked, but it's crucial for ensuring that the system is not only available but also reliable in terms of the quality of the data it provides. A system may have 99.99% availability, but if even 1% of successful responses are incorrect, it can lead to significant issues for users and stakeholders.

# WHY UNRELIABILITY HAPPENS

So, why do systems become unreliable? There are several reasons for this, including -

    1. Hardware failures: Physical components of a system can fail due to wear and tear, manufacturing defects, or environmental factors. For example, a hard drive can fail, causing data loss and system downtime.

    2. Software bugs: Software can have bugs that cause it to crash or produce incorrect results.

    The most dangerous bugs are the ones that don't cause crashes but lead to incorrect data or behavior, as they can go unnoticed for a long time.

    3. Configuration errors: Incorrect configuration of a system can lead to reliability issues. For example, if a database is not properly configured, it can lead to data corruption or loss.

    4. Human error: Mistakes made by users or administrators can lead to reliability issues. For example, accidentally deleting important files or mis-configuring a system can cause it to become unreliable.

# GRACEFUL DEGRADATION

There is a term called 'Graceful Degradation' which refers to the ability of a system to continue functioning even when some components fail. This is an important aspect of reliability, as it allows the system to maintain some level of functionality even in the face of failures.

For example, if a web application has a feature that allows users to upload files, and the file upload service fails, the application can still allow users to view and interact with other features. This way, the system can continue to provide value to users even when certain components are not working.

It is not good if a system completely fails when a single component fails. Instead, it should be designed to degrade gracefully, allowing users to continue using the system with reduced functionality rather than experiencing a complete outage. This can help improve the overall user experience and maintain trust in the system even during failures.

# IDEMPOTENCY

Idempotency is a property of certain operations in which performing the same operation multiple times has the same effect as performing it once. This is an important concept in reliability, as it allows systems to handle retries and failures without causing unintended consequences.

For example, consider a payment processing system. If a user submits a payment request and the system experiences a failure before it can confirm the payment, the user may retry the request. If the payment processing operation is idempotent, the system will recognize that the same payment request has already been processed and will not charge the user multiple times.

For this, the system can generate a unique identifier (known as an idempotency key) for each payment request, and if it receives a request with the same identifier, it can simply return the result of the original request instead of processing it again. This way, even if the user retries the payment request multiple times, they will only be charged once, ensuring reliability and a positive user experience.

So, that is the basic idea of reliability. It is crucial to design systems that are not only available but also reliable, ensuring that they can perform their intended functions correctly and consistently over time. By understanding the metrics for measuring reliability and the reasons why systems can become unreliable, we can take steps to improve the reliability of our systems and provide a better experience for our users.

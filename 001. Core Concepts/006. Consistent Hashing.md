Before we talk about 'Consistent Hashing', we should first understand what is 'Hashing' in general.

In computer science, hashing is a technique used to uniquely identify a specific object from a group of similar objects. It is commonly used in data structures like hash tables, where it allows for efficient data retrieval.

A hash function takes an input (or 'key') and returns a fixed-size string of bytes, typically a hash code. The output is usually a number that represents the original data.

So, how does Hashing work in the context of distributed systems?

As we know, a distributed system is a collection of independent computers that appear to the users of the system as a single computer. In such systems, we often need to distribute data across multiple nodes (computers) for load balancing and fault tolerance.

Let's say we have multiple servers and we want to distribute user data across these servers. How do we efficiently route user requests to the correct server?

A common approach is to hash the request and then use the hash value to determine which server should handle the request.

For example, if we have 4 servers, we can take the hash value of the request and mod it by 4 to get a number between 0 and 3, which corresponds to one of the servers. In general terms, if we have 'N' servers, we can use the formula -

    hash(request) % N
     
to determine which server should handle the request.

But, this method is dependent on the number of servers which means that if the number of servers or N changes, this will lead to a lot of rehashing and redistribution of data across the servers.

# THE PROBLEM WITH TRADITIONAL HASHING

When we add or remove a server, the hash values of the keys will change, and we will need to rehash all the keys to determine their new server locations. This can lead to a significant amount of data movement and can cause performance issues.

For example, let's say we have 4 servers and requests are assigned to servers based on the hash value mod 4. The hash value is calculated by simply taking the sum of bytes in the request. If we add a new server (making it 5 servers), the hash value mod 5 will change for all requests, and we will need to rehash all the requests to determine their new server locations. This means doing this can cause session loss, cache loss, and other performance issues.

# THE SOLUTION: CONSISTENT HASHING

This is where 'Consistent Hashing' comes into play.

Consistent Hashing is a special kind of hashing that minimizes the number of keys that need to be remapped when a hash table is resized (i.e., when servers are added or removed).

But, how does Consistent Hashing work?

In Consistent Hashing, a 'Circular Hash Space' or 'Hash Ring' is used. There is a large and constant has space and both the nodes (servers, caches, databases) and the keys (data items) are mapped to positions on this hash ring using a hash function.

When the number of nodes change, only a certain portion of the keys need to be remapped to different nodes, which significantly reduces the amount of data movement and improves performance. This number is typically k/n where 'n' is the number of nodes and 'k' is the number of keys.

So, how this 'Hash Ring' is constructed?

Well, we usually use a large space ranging from 0 to 2^32 - 1 (e.g. for a 32-bit hash function) and this creates a circular structure where the values wrap around from the maximum back to zero.

We use a hash function to assign a position for each node (e.g. server) on the hash ring.

Now, when a key is added, we hash the key to get its position on the hash ring. For example, let's say a request comes and let's say we use the IP address of the request as the key. We hash this key to get a position on the hash ring.

Now, after we get the position, we simply move clockwise on the hash ring until we find the first node (server) that is greater than or equal to the key's position. This node will be responsible for handling the request.

Let's say we had three servers S1, S2, and S3, and they are placed on the hash ring at positions 10, 20, and 30 respectively.

Now, a new server S4 is added and it is placed at position 25 on the hash ring.

This means, all the requests that were previously handled by S3 (which is at position 30) and have hash values between 20 and 30 will now be handled by S4 (which is at position 25).

This means, only a small portion of the requests (those between 20 and 30) will be remapped to S4, while the rest of the requests will continue to be handled by their original servers (S1 and S2). This is the key advantage of Consistent Hashing - it minimizes the number of keys that need to be remapped when a server is added or removed.

Similarly, let's say we remove server S2 (which is at position 20). Now, all the requests that were previously handled by S2 will be reassigned to the next available server on the hash ring. Again, only a small portion of the requests will be remapped.

# VIRTUAL NODES

To further improve the distribution of keys and reduce the impact of adding or removing servers, we can use 'Virtual Nodes' in Consistent Hashing.

In basic consistent hashing, each server is represented by a single point on the hash ring. However, this can lead to uneven distribution of keys if the servers are not evenly spaced on the hash ring.

For example, if the servers are placed at positions 10, 20, and 30, and we have a large number of keys that hash to values between 20 and 30, then server S3 (at position 30) will be overloaded while servers S1 and S2 will be underutilized.

Virtual Nodes solve this problem by allowing each physical server to be represented by multiple points on the hash ring. This means that each server can have multiple virtual nodes, and each virtual node will be assigned a position on the hash ring.

Each server is hashed multiple times to different locations and so, when a request (key) comes in, and it is hashed, it will be assigned to the closest virtual node on the hash ring, which in turn will be mapped to the physical server.

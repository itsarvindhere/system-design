Fault tolerance in system design means a system can keep working, possibly in a reduced mode, even when some part of it fails. So, it is the ability of a system to handle outages and errors without any loss of functionality.

In real systems, failures are normal. Servers crash, databases slow down, networks drop packets, disks fail, APIs time out, deployments introduce bugs, and cloud regions can become unavailable. A fault-tolerant system assumes these things will happen and is built to survive them.

For example, imagine a food delivery app. If the payment service is temporarily slow, the app should not completely crash. It might retry the payment request, show a pending payment state, or let the user try again. If the recommendation service fails, users should still be able to browse restaurants and place orders.

Fault tolerance is the overall design philosophy: “How do we keep the system useful when something breaks?”

Common fault tolerance techniques include:

    1. Redundancy: running multiple servers instead of one.
    2. Replication: storing copies of data in multiple places.
    3. Retries: trying an operation again after a temporary failure.
    4. Timeouts: not waiting forever for a broken service.
    5. Circuit breakers: temporarily stopping calls to a service that is failing.
    6. Bulkheads: isolating failures so one bad component does not bring down the whole system.
    7. Graceful degradation: disabling non-critical features while keeping core features alive.
    8. Monitoring and alerts: detecting problems quickly.

For example, let's say an e-commerce site has these features - Product Search, Cart, Checkout, Payment, Recommendations and Reviews.

If the reviews service fails, the website can still show products, allow cart updates, and complete checkout. Reviews may be hidden temporarily. That is fault tolerance.

The system did not necessarily switch to another reviews service. It simply handled the failure in a way that preserved the important parts of the user experience.

In the last section, we learnt what is 'Failover'. Do not confuse between Failover and Fault Tolerance.

Failover is a specific mechanism used to recover from a failure by switching from a failed component to a backup component. For example, suppose your application uses Database A as the primary database. You also have Database B as a standby replica. If Database A fails, the system switches traffic to Database B. That switch is called failover.

Fault tolerance is the broader goal. Failover is one technique that helps achieve that goal.

Let's say you have a Video Streaming app. The system has Login service, Video catalog service, Streaming service, Recommendations service, Payment service, User history service 
etc.

Suppose that Recommendations fail. The app still lets users log in, browse popular videos, and watch content. It simply hides personalized recommendations. That is fault tolerance through graceful degradation. No failover may have happened.

Let's say the primary database fails. The system detects the failure and switches to a replica database. That is failover. This failover helps the system be fault-tolerant because users can still browse and watch videos.

So, a system can use failover as part of fault tolerance, but fault tolerance is not only failover. Fault tolerance is the system’s ability to keep working when failures happen. Failover is the act of switching to a backup component after a failure.

Failover is one tool used to build fault-tolerant systems, but fault tolerance also includes retries, timeouts, fallbacks, queues, isolation, replication, graceful degradation, and monitoring.

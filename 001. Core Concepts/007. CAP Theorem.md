One of the fundamental concepts in distributed systems is the CAP theorem where 'CAP' stands for Consistency, Availability, and Partition Tolerance.

This theorem states that in the presence of a network partition, a distributed system can only guarantee either consistency or availability, but not both. So, this theorem basically describes the trade-offs that distributed systems must make when it comes to these three properties.

## CONSISTENCY

Consistency ensures that every read receives the most recent write or an error. In other words, all nodes see the same data at the same time.

For example, let's say you have a distributed database with multiple replicas. If you write data to one replica, consistency ensures that all other replicas will reflect that change immediately. So, if you read from any replica, you will get the most up-to-date data.

It makes sense that for applications where data integrity is crucial, such as financial transactions or inventory management, consistency is a key requirement. In these cases, you would want to ensure that all nodes have the same data at all times.

## AVAILABILITY

Availability ensures that very request receives a response, without guarantee that it contains the most recent write. This means that the system is operational and responsive, but it may not always return the most up-to-date data.

For example, if you have a distributed system that is designed for high availability, it may allow reads from any replica even if some replicas are not up-to-date. This means that you might get stale data, but the system will still respond to your request.

For applications that should remain operational even in the face of failures, such as social media platforms or e-commerce websites, availability is a critical requirement. In these cases, you would want to ensure that the system can still respond to user requests even if some nodes are down or not up-to-date.

## PARTITION TOLERANCE

Finally,  partition tolerance means that the system continues to operate despite an arbitrary number of messages being dropped (or delayed) by the network between nodes. In other words, the system can still function even if there are network partitions that prevent communication between some nodes.

For example, if you have a distributed system that is designed to be partition tolerant, it may allow nodes to continue operating even if they cannot communicate with each other. This means that the system can still function and respond to requests even if there are network issues.

# DESIGN STRATEGIES

## EVENTUAL CONSISTENCY

One common design strategy for distributed systems is to use eventual consistency. This means that the system allows for temporary inconsistencies between nodes, but eventually, all nodes will converge to the same state.

For example, in a distributed database that uses eventual consistency, when you write data to one replica, it may take some time for that change to propagate to all other replicas. During this time, if you read from another replica, you may get stale data. However, eventually, all replicas will reflect the change and converge to the same state.

## STRONG CONSISTENCY

Another design strategy is to use strong consistency. This means that the system ensures that all nodes have the same data at all times, even in the presence of network partitions.

For example, in a distributed database that uses strong consistency, when you write data to one replica, it will immediately propagate to all other replicas. This means that if you read from any replica, you will get the most up-to-date data. Financial applications and inventory management systems often use strong consistency to ensure that all nodes have the same data at all times.

## TUNABLE CONSISTENCY

Finally, some systems may choose to use tunable consistency, which allows you to configure the level of consistency based on your application's requirements.

Systems like Cassandra offer tunable consistency, where you can choose between strong consistency, eventual consistency, or something in between based on your application's needs.

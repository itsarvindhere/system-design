A checksum is a small value calculated from data to help check whether that data changed or got corrupted.

So Checksums are like quick 'data fingerprints' that are used to detect errors.

For example, if you send a file over a network, the sender can calculate a checksum before sending it. The receiver calculates the checksum again after receiving it. If both checksums match, the data is probably correct. If they do not match, the data was likely changed, damaged, or transmitted incorrectly.

Let's say you send these numbers -

    4,7,2,9

A very simple checksum could be the sum of these numbers -

    4  + 7 + 2 + 9 => 22

So, the sender sends -

    Data: 4,7,2,9
    Checksum: 22

The receiver adds the numbers again and if the receiver gets the checksum as 22, the data is probably okay.

Checksums are used to detect accidental errors, such as:

    - network transmission errors
    - corrupted files
    - damaged storage data
    - incomplete downloads
    - memory or disk errors

When data travels over a network, it is split into packets. Each packet may include a checksum. The receiver checks the packet checksum to see whether the packet was damaged during transmission.

For example:

    - TCP uses checksums
    - UDP uses checksums
    - IP headers can use checksums in IPv4

If a packet fails the checksum test, it is usually discarded. With TCP, the lost or bad packet can be retransmitted. With UDP, the bad packet may simply be dropped.

# CHECKSUM vs HASH

Checksums and hashes are related, but they are not always the same thing.

A checksum is usually designed to detect accidental errors.

A cryptographic hash is designed to detect changes and resist tampering.

So, some examples of checksum can be simple sum (adding the data values), parity bit (whether the number of 1 bits is even or odd), CRC32, TDP checksum, UDP checksum etc.

Example of cryptographic hashes can be SHA-256, SHA-3, BLAKE3 etc.

If you only want to know whether a file was accidentally corrupted, a checksum may be enough.

If you want to verify that nobody maliciously changed a file, use a cryptographic hash like SHA-256.

Do note that Checksums do not usually fix errors by themselves. They mostly detect errors. Some systems combine checksums with error correction methods, but the checksum alone usually just says:

    This data looks okay

OR

    This data has changed

They cannot tell you who created the data, was the checksum itself changed, is the data authorized, is the algorithm secure against an attacker etc.

# CRC CHECKSUMS

CRC means Cyclic Redundancy Check. CRC is a common type of checksum used to detect accidental changes in data. It is stronger than a simple sum because it can catch many more types of errors.

It treats data like a binary number and performs a kind of polynomial division. You do not need to deeply understand the math at first. The key idea is:

    CRC is designed to catch common transmission errors very well.

CRC can detect many errors that simple sums miss, such as:

    - single-bit errors
    - many burst errors
    - accidental changes in blocks of data

CRC is used in Ethernet, ZIP Files, Storage devices, and communication protocols. CRC32 is a well-known example. It produces a 32-bit checksum value.

# LIMITATIONS OF CHECKSUMS

Checksums are useful, but they are not perfect. They can sometimes miss errors. Two different pieces of data can produce the same checksum. This is called a collision. Simple checksums have a higher chance of collisions. Stronger checksums and cryptographic hashes have a much lower chance.

Also, normal checksums are not secure against attackers. A malicious person could change the data and create a new matching checksum if the algorithm is simple and not protected.

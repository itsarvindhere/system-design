HTTP and HTTPS are both ways for your browser and a website to talk to each other. The difference is that HTTP talks in the open, while HTTPS talks through an encrypted, identity-checked channel.

# HTTP (HyperText Transfer Protocol)

HTTP is the rule system your browser and a web server use to exchange web pages, images, data, forms, API responses, and so on.

When your browser uses HTTP, it sends a request like:

    GET /home HTTP/1.1
    Host: example.com

The server then sends back a response -

    HTTP/1.1 200 OK
    Content-Type: text/html

    <html>...</html>

So, the core idea is -

    Client asks, Server answers

HTTP is stateless. That means each request is treated as independent. The server does not automatically remember the previous request just because it came from the same browser.

So how do websites remember you are logged in?

They add state on top of HTTP, usually with cookies or tokens. After login, the server sends back a cookie. Your browser stores it. On later requests, the browser sends that cookie again.

For example -

    Cookie: session_id=abc123

Now the server can say, “This request has session_id abc123, so this is some specific User.” The memory is not built into HTTP itself; it is created using extra data sent with each request.

HTTP is built around resources. A resource is anything the server exposes through a URL. This could be:

    /home
    /users/17
    /images/logo.png
    /api/orders/991

The URL identifies what you want, and the HTTP method describes what you want to do with it.

Common methods include -

    GET     read/fetch a resource
    POST    submit data or create something
    PUT     replace something
    PATCH   partially update something
    DELETE  remove something

There is another important idea: safe and idempotent methods.

## SAFE AND IDEMPOTENT METHODS

GET is supposed to be safe, meaning it should only fetch data, not change server state. Visiting a page should not delete your account.

PUT and DELETE are supposed to be idempotent, meaning if you send the same request multiple times, the end result should be the same as sending it once. Deleting the same item twice should still leave it deleted. Replacing a profile with the same data twice should still produce the same final profile.

POST is usually not idempotent. If you submit the same order twice, you might create two orders.

Now, HTTP by itself does not mean encrypted. Original HTTP sends these messages in a way that can be read by machines in the middle of the network. That is why HTTPS exists.

# HTTPS

HTTPS is HTTP over TLS (Transport Layer Security). The 'S' stands for 'Secure.'

The HTTP part is still the same: methods, paths, headers, bodies, status codes, cookies, caching, statelessness. HTTPS adds a secure layer underneath before the HTTP messages are exchanged.

## TLS HANDSHAKE

A TLS handshake is the setup conversation that happens before HTTPS starts sending real web data.

Its job is to let your browser and the server agree on three things:

    1. Are you really the website I wanted?
    2. What encryption method are we going to use?
    3. What secret keys will we use for this connection?

After the handshake finishes, normal HTTP traffic can travel securely.

Suppose you visit:

    https://example.com
    
Before your browser sends:

    GET / HTTP/1.1
    Host: example.com

it first does the TLS handshake.

In simplified form, it goes like this.

Your browser says hello to the server:

    ClientHello

This message says, 'I want to make a secure connection. Here are the TLS versions and encryption algorithms I support. Here is some random data from me.'

The server replies -

    ServerHello

This means, 'Okay, we will use this TLS version and this encryption algorithm. Here is some random data from me.'

Then the server sends a certificate.

The certificate is like a digital ID card for the website. It says, 'This public key belongs to example.com.' The certificate is signed by a trusted Certificate Authority.

Your browser checks the certificate. It asks:

    Is this certificate for example.com?
    Is it expired?
    Was it issued by a trusted authority?
    Has it been tampered with?

If the certificate fails, the browser shows a warning.

If it passes, the browser and server create shared encryption keys. The important part is that they can agree on a secret key even though someone might be watching the network. Modern TLS usually does this using a key exchange method called ECDHE.

Once both sides have the same shared secret, they switch to encrypted communication.

From then on, the actual HTTP request is protected:

    GET / HTTP/1.1
    Host: example.com

That request still exists, but it is now inside the encrypted TLS connection.

# HTTP VERSIONS

HTTP versions matter as they matter because they change how the same HTTP ideas travel over the network.

The core HTTP model stays mostly the same across versions:

    request -> response
    methods like GET/POST
    headers
    body
    status codes
    cookies
    caching

What changes is the transport behavior: speed, connection handling, multiplexing, compression, and whether it runs over TCP or UDP.

## HTTP/0.9

HTTP/0.9 was the earliest simple version. It was barely a full protocol by modern standards. A client could basically ask for a document with a very simple request, and the server returned HTML. No headers, no status codes, no content types. It was enough for the very early web, but not enough for images, APIs, caching, authentication, or modern pages.

## HTTP/1.0

HTTP/1.0 made HTTP look more like what we recognize today. It added status codes, headers, content types, and more formal request/response structure. Now the server could say things like 200 OK, 404 Not Found, or Content-Type: text/html. But a major limitation was connection handling. In typical HTTP/1.0 behavior, each request often used a separate TCP connection. If a web page needed HTML, CSS, JavaScript, and ten images, the browser might need many connections. That is expensive because creating TCP connections takes time.

## HTTP/1.1

HTTP/1.1 improved this a lot. It became the long-lived classic version of the web. It added persistent connections by default, meaning the browser could reuse the same TCP connection for multiple requests instead of opening a new one every time. It also added better caching controls, chunked transfer encoding, the required Host header for serving multiple domains from one IP address, and more practical features.

But HTTP/1.1 still has an important limitation called head-of-line blocking at the HTTP level.

Over one connection, responses generally have to come back in order. If one slow response is ahead, it can delay others behind it. Browsers worked around this by opening multiple TCP connections to the same site, but that is still not ideal.

## HTTP/2

HTTP/2 was designed to fix much of that. It keeps the same HTTP meaning, like GET /page, headers, status codes, and bodies, but changes how messages are encoded and sent. Instead of plain text messages traveling one at a time, HTTP/2 uses a binary framing format. More importantly, it supports multiplexing, which means many requests and responses can share one TCP connection at the same time.

So with HTTP/1.1, the browser might effectively be stuck waiting on ordered responses per connection. With HTTP/2, the browser can send many requests together, and the responses can be interleaved. A large image does not have to block a tiny CSS file in the same way. HTTP/2 also compresses headers using HPACK, which helps because modern web requests often repeat lots of header data, especially cookies.

HTTP/2 usually runs over TLS in browsers, so in practice people often see it as part of HTTPS, though the protocol itself can technically exist without TLS in some contexts.

## HTTP/3

HTTP/3 is the newer major version. The biggest change is that it runs over QUIC, which uses UDP instead of TCP. QUIC includes encryption using TLS 1.3 concepts and handles reliability, streams, and congestion control itself.

Why move away from TCP?

Because HTTP/2 fixed head-of-line blocking at the HTTP layer, but TCP still has head-of-line blocking underneath. If one TCP packet is lost, the entire TCP connection may wait until that packet is retransmitted, even if other HTTP/2 streams are ready. Since HTTP/2 multiplexes many streams over one TCP connection, packet loss can slow everything on that connection.

HTTP/3 avoids that by using QUIC streams. If one stream has packet loss, it does not necessarily block every other stream in the same way. This is especially useful on mobile networks, Wi-Fi, and unstable connections. QUIC can also reconnect more smoothly when your network changes, such as moving from Wi-Fi to mobile data.

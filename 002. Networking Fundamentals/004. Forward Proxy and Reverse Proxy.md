You might've heard the term 'Proxy' a lot. A Proxy is a middleman.

So, instead of one computer talking to another computer, it talks through a proxy.

    Client -> Proxy -> Server

The proxy receives the request, forwards it, receives the response, and sends the response back.

Now the important part is -

    Who is the proxy representing?

And this gives us the difference between a 'Forward Proxy' and a 'Reverse Proxy.'

# FORWARD PROXY

A normal 'Proxy' usually means a 'Forward Proxy'. A Forward Proxy sits in front of a client.

    Client -> Forward Proxy -> Internet Server

For example -

    Your laptop -> Company proxy -> google.com

From Google's point of view, the request may appear to come from the company proxy, not directly from your laptop.

So the forward proxy is used by clients to reach servers.

When a client requests a resource from the internet, the client's request is first directed to a Forward Proxy and then the Forward Proxy forwards the request to the appropriate server. This server responds to the Forward Proxy and then the response is sent back to the client by the Forward Proxy.

But what is the use?

Common uses of Forward Proxies are -

    - Hide or mask the client’s real IP
    - Control which websites employees can access
    - Log outgoing traffic
    - Cache common content
    - Bypass network restrictions, depending on policy
    - Add security filtering before traffic leaves the network

For example, in an office network, employees may not access the internet directly.

    Employee laptop -> Corporate proxy -> Internet

The company proxy can block certain sites, scan traffic, or keep logs.

# REVERSE PROXY

A Reverse Proxy sits in front of the server. It represents the server. So, a client does not directly talk to the server but the request is first received by the Reverse Proxy and then forwarded internally to the server.

What is the use?

Common uses of Reverse Proxies include -

    - Load balancing
    - SSL/TLS termination
    - Hiding backend servers
    - Routing requests to different services
    - Caching responses
    - Compression
    - Rate limiting
    - Authentication
    - Security filtering
    - Serving static files
    - Centralized logging

Reverse proxies are extremely common in system design. Some examples of Reverse Proxies are Nginx, Apache HTTPD, Cloudflare, AWS Application Load Balancer etc.

Cloudflare acts like a reverse proxy. Users hit Cloudflare first, and Cloudflare forwards allowed traffic to your origin server.

Suppose you built an app with three backend servers:

    App Server 1
    App Server 2
    App Server 3

You do not want users to know or directly access those servers.

So, you can put a reverse proxy in front.

    Users -> Reverse Proxy / Load Balancer --> App Server 1 and 2 and 3

So, users only know 

    https://myapp.com
    
They do not know -

    10.0.1.10:3000
    10.0.1.11:3000
    10.0.1.12:3000

The reverse proxy chooses which backend server should handle each request.

So in simple terms -

    Forward proxy hides/protects clients.
    Reverse proxy hides/protects servers.

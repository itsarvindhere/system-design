In one of the previous sections where we learnt the OSI Model, we understood that in the transport layer, there are two main protocols - TCP and UDP. Since they are part of the Transport Layer, both solve the same problems which is the delivery of data between processes using ports.

For example, your browser, a video call app, an online game, or an email app may use TCP or UDP depending on what matters most: reliability or speed.

Think of the internet as a road system. IP is responsible for getting packets from one device to another, like writing the destination address on an envelope. TCP and UDP decide how the data should be sent.

TCP is careful and reliable. UDP is fast and lightweight.

TCP checks that data arrives correctly, in order, and completely. UDP sends data quickly without checking much afterward.

# TCP - Transmission Control Protocol

TCP is used when accuracy matters more than raw speed. It creates a reliable connection between two devices before sending data. This is why TCP is called connection-oriented.

Before TCP sends actual data, it performs a setup process called the three-way handshake:

    1. The client sends a request to start a connection.
    2. The server replies and agrees.
    3. The client confirms.

    After that, data transfer begins.

TCP keeps track of every piece of data it sends. If something is lost, TCP sends it again. If packets arrive out of order, TCP rearranges them. If the receiver is overwhelmed, TCP slows down. This makes TCP very dependable.

Due to its reliability, TCP is used for things like web browsing with HTTP/HTTPS, Email, File Downloads, Online banking, Database connections, Remote logins sch as SSH etc.

TCP is a good choice when missing or incorrect data would be a serious problem. For example, when downloading a PDF, software update, or bank statement, every byte must arrive correctly.

The downside is that TCP has more overhead. Because it checks, confirms, and retransmits data, it can be slower than UDP in situations where real-time speed matters.

# UDP - User Datagram Protocol

UDP is used when speed matters more than perfect reliability. It does not create a formal connection before sending data. This is why UDP is called connectionless.

UDP simply sends packets, called datagrams, toward the destination. It does not guarantee that they arrive. It does not guarantee they arrive in order. It does not automatically resend lost packets.

That sounds risky, but it is very useful for real-time applications.

Some common uses of UDP are video calls, voice calls, online gaming, live streaming etc.

For example, during a video call, if one tiny packet is lost, it is usually better to skip it and keep the call moving. If the app waited to resend every missing packet, the call could freeze or become delayed. UDP helps keep things real-time.

The downside is that UDP is not reliable by itself. If an application needs reliability while using UDP, the application must add its own checking, ordering, or retransmission logic.

# EXAMPLE

Imagine sending a book to someone.

TCP is like sending the book page by page and asking the receiver to confirm each page arrived. If page 24 is missing, you send page 24 again. At the end, the receiver has the complete book in the correct order.

UDP is like throwing pages across quickly and hoping most of them arrive. Some may be missing or out of order. But it is much faster, which is useful if the information becomes old quickly.

For a file download, TCP is better.

For a live video call, UDP is often better.

Most websites use TCP through HTTP or HTTPS. When you open a webpage, your browser needs HTML, CSS, JavaScript, images, and other files. If part of the webpage data is missing, the page may break or display incorrectly. TCP makes sure all required data arrives properly.

HTTPS also uses TLS encryption, and traditionally TLS runs over TCP. Modern web protocols like HTTP/3 use QUIC, which runs over UDP, but QUIC adds reliability and security features at a higher layer. So even when UDP is used, the application may build TCP-like reliability on top of it.

So to summarize -

    TCP and UDP are both transport-layer protocols.

    TCP is reliable, ordered, and connection-based. It is best when data must be complete and correct.

    UDP is fast, simple, and connectionless. It is best when low delay matters more than perfect delivery.

Load balancing means spreading incoming traffic or work across multiple servers/resources so no single server gets overloaded.

Instead of every user going to one server, a load balancer sits in front and decides where each request should go.

But, why do we need Load Balancing?

Imagine an online shopping website.

If all users connect to only one server, that server may become slow or crash when traffic increases. With load balancing, traffic is shared across many servers.

Load balancing helps with:

    - Performance: requests are handled faster
    - Availability: if one server fails, others can continue
    - Scalability: more servers can be added as traffic grows
    - Reliability: users are less affected by failures
    - Maintenance: servers can be updated one at a time without stopping the whole app

Suppose one server can handle 1,000 users.

If 5,000 users visit your app, one server is not enough. You can use five servers and place a load balancer in front of them. The user does not need to know which server handled the request. To the user, it feels like one website.

When a request arrives, the load balancer checks its rules and forwards the request to one backend server. The backend server processes the request and sends the response back.

There are two common response paths:

    1. The response goes back through the load balancer.
    2. The response goes directly from the backend server to the user.

Most web applications use the first approach because it gives better control, security, logging, and traffic management.

For HTTPS traffic, encryption is handled with TLS. A load balancer can perform TLS termination, meaning it decrypts HTTPS traffic before forwarding it to backend servers.


    Client --HTTPS--> Load Balancer --HTTP or HTTPS--> Backend Server

This way the backend servers do less encryption work and certificates are managed at one place.

# TYPES OF LOAD BALANCING

If we go by the OSI Model, there are two types of Load Balancing.

## 1. LAYER 4 (L4) LOAD BALANCING

Layer 4 load balancing works at the transport layer. It makes decisions using information such as:

    1. IP address
    2. TCP or UDP port
    3. Protocol

It does not deeply inspect the actual application data.

For example -

    Request to IP 203.0.113.10 on port 443
    Load balancer forwards it to Server 2

Layer 4 load balancing is fast because it looks at less information.

Some common uses are TCP traffic, UDP traffic, High-performance systems etc.

## 2. LAYER 7 (L7) LOAD BALANCING

Layer 7 load balancing works at the application layer. It understands application-level data, such as HTTP. It can make decisions based on the URL path, Hostname, HTTP Headers, Cookies, Request Method, Content type and more.

For example -

    /api/users     -> API servers
    /images        -> image servers
    /admin         -> admin servers

Layer 7 load balancing is smarter but usually has more overhead than Layer 4.

Some common uses are Websites, APIs, Microservices etc.

Imagine you own shop.com.

    You have:

    3 web servers
    2 API servers
    2 image servers

A Layer 7 load balancer can route requests like this:

    shop.com/             -> web servers
    shop.com/api/orders   -> API servers
    shop.com/images/item  -> image servers

If one API server fails, health checks remove it from rotation. Users continue using the site through the other API server.

If traffic increases during a sale, you add more API servers. The load balancer starts sending traffic to them.

# LOAD BALANCING ALGORITHMS

A load balancing algorithm is the rule used to choose the next server.

## 1. ROUND ROBIN

Requests are sent to servers one by one in order.

    Request 1 -> Server A
    Request 2 -> Server B
    Request 3 -> Server C
    Request 4 -> Server A

This is simple and common. This is base when servers are similar in power and requests take similar effort.

## 2. WEIGHTED ROUND ROBIN

Servers receive traffic based on assigned weights.

    Server A weight: 5
    Server B weight: 3
    Server C weight: 2

This means Server A gets more traffic than B or C. This is best when servers have different capacities.

## 3. LEAST CONNECTIONS

Traffic goes to the server with the fewest active connections. This is the best when user experience depends heavily on speed.

## 4. LEAST RESPONSE TIME

Traffic goes to the server with the fastest response time. This is the best when user experience depends heavily on speed.

## 5. IP HASH

The load balancer uses the user’s IP address to choose a server. This can help send the same user to the same server repeatedly.

    User 1 always -> Server B
    User 2 always -> Server A

This is useful for session persistence, but it can become uneven if many users come from the same network.

What are 'Sticky Sessions?'

Sometimes an application needs the same user to keep going to the same server. This is called session persistence or Sticky sessions or session affinity.

For example -

    User logs in on Server A
    Next request also goes to Server A

This can be useful if session data is stored in server memory.

However, sticky sessions are not always ideal. A better design is often to store session data outside the app server, such as in:

    1. Redis
    2. Database
    3. Shared session store

That way, any server can handle any request.

## 6. RANDOM

The load balancer randomly picks a server.

This is simple and can work surprisingly well at scale, especially when combined with health checks.

A health check means that a Load Balancer must know which servers are healthy. It regularly checks backend servers using health checks.

If Server B stops responding, the load balancer removes it from rotation.

    Before failure:
    A, B, C receive traffic

    After B fails:
    A and C receive traffic

When Server B becomes healthy again, it can be added back.

Health checks are extremely important. Without them, a load balancer might continue sending users to broken servers.

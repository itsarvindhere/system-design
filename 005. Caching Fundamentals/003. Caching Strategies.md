Caching strategies are the different ways an application decides how to read from the cache, write from the cache, and keep the cache sync with the real source of the data.

Different systems use different strategies depending on whether they care most about Read Speed or Write Speed or Freshness or Simplicity or Consistency.

Not every system uses cache in the same way. For example -

    1. A product page may be read a lot
    2. A session store may need fast lookups
    3. A metrics dashboard may tolerate slight staleness
    4. A payment system may need stronger correctness
    5. A feed system may need frequent refreshes

So, one cache strategy does not fit all. The right strategy depends on whether the system is read-heavy, write-heavy, freshness-sensitive, or latency-sensitive.

# CACHE STRATEGIES

## 1. CACHE-ASIDE

This is the most common caching strategy. In this pattern, the application is responsible for managing the cache. The flow is -

    1. App checks cache
    2. If hit, return data
    3. If miss, fetch from database
    4. Store result in cache
    5. Return result

So, the application talks to the cache first, and if the cache missed, it falls back to the database.

Let's say we want to get product details -

    GET /product/123

The Cache-aside flow is -

    App -> Cache: Do you have product 123?
    Cache -> App: No

    App -> Database: Fetch product 123
    Database -> App: Here it is

    App -> Cache: Store product 123
    App -> Client: Return product 123

So, the next time -

    App -> Cache: Do you have product 123?
    Cache -> App: Yes

    App -> Client: Return product 123

This way, the second request is much faster since it the response is served from the cache and no database query is required.

Since it is simple and flexible, Cache-Aside is popular. Also, in this strategy, the application controls when to cache, when to fetch from the database, and when to update or invalidate the cache.

It is commonly used for Product data, User profiles, API responses, Computed results, and session data. This is the best for read-heavy workloads where the application can absorb the cache logic.

But, it is not perfect. The downside is that the application has to manage the cache logic. It has to handle -

    1. Cache miss
    2. Cache invalidation
    3. Cache update after writes
    4. Stale data risk

## 2. READ-THROUGH

In the read-through caching, the cache itself is responsible for fetching data from the database when there is a miss. The application only 'asks' the cache.

So, the flow is -

    1. App asks cache
    2. If hit, return data
    3. If miss, cache fetches from database
    4. Cache stores the data
    5. Cache returns it to app

So, the app does not directly talk to the database for cached reads.

Suppose, the app asks for profile data -

    App -> Cache: user:123

If the cache has it, it returns immediately.

If not, the cache itself loads it from the database and stores it. So, the app gets the result without manually handling the miss path.

The benefit is that it simplifies the application code since the application does not have to implement Cache Miss logic, database fallback logic, and the cache population logic.

This is good for systems where many services share the same read pattern and benefit from a single loader implementation.

CDNs commonly use Read-Through strategy where when a request arrives, the CDN checks its local 'edge' server. If it has the file, it serves it directly (Cache Hit). Otherwise, it fetches the content from your origin server, caches a copy, and forwards it to the user (Cache Miss).

The downside if that if can hide complexity inside the cache layer and can be less flexible than Cache-Aside. Also, the cache now becomes more involved in the read behavior. This can be useful in some systems, but not always the simplest choice.

## 3. WRITE-THROUGH

In write-through caching, every write goes to the cache and the database at the same time (synchronously).

The flow is -

    1. App writes to cache
    2. Cache writes to database
    3. App gets success only after both are updated

This means the cache always has the latest value.

Suppose, we update the user profile -

    PATCH /users/123
    name = Arvind Rana

Write-through flow will be -

    App -> Cache: Update user:123
    Cache -> Database: Persist user:123
    Cache -> App: Success
    App -> Client: Success

Now, the future reads can come from the cache immediately.

The benefits are that the cache stays fresh, reads after writes are easier, data is persisted immediately, and the cache and database stay more in sync.

This is good when freshness matters.

The biggest downside is that the write is slower because it must update two layers - cache and database.

So, if the database is slow, every write becomes slower. Moreover, what if the cache write succeeds while the database write fails? That requires us to handle all these edge cases as well.

It is not an ideal strategy when writes are extremely frequent and low latency is critical.

It is best for those workloads where read after writes are common and write latency is tolerable.

## 4. WRITE-BEHIND (WRITE-BACK)

In the write-behind strategy, the application writes to the cache first, and the cache writs to the database later asynchronously. So, the flow is -

    1. App writes to cache
    2. App returns success quickly
    3. Cache flushes changes to database later

This improves the write performance. For example, suppose we update a counter -

    view_count = 1000

The app updates the cache immediately.

    Cache: view_count = 1001

Later, the cache writes the update to the database in the background. This is fast because the user does not wait for the database write.

This strategy is useful when writes are frequent, the immediate persistence is not always required, low latency is important, and temporary staleness is acceptable.

Some common use cases are social media counters, analytics, metrics, non-critical event aggregation, some logging systems, etc.

The write-behind strategy has a big risk. What if the cache fails before the data is written to the database? This means the data may be lost. So, write-behind is not ideal for critical data like Payments, Bank balances, Orders, Inventory reservation and more.

It is better suited for data where some delay or loss is acceptable.

## 5. WRITE-AROUND

In write-around strategy, the application writes directly to the database. So, the cache is not used for writes. Cached data is updated only on the next read miss, typically through a cache-aside fill.

If the same key was already cached when the write happens, that cached value becomes stale until it expires or is explicitly invalidated.

Write-around avoids caching data that is never read again, keeps writes simple by touching only one storage layer, and lets the cache stay focused on actually-read data.

But, it has downsides. The first read after the write is always a cache miss, existing cache entries can be left stale unless explicitly invalidated, and read-after-write behavior is weaker than write-through.

This is best for write-heavy workloads where written data is rarely read soon after, like bulk imports, event ingestion, and large append-only logs.

## 6. REFRESH-AHEAD

In the refresh-ahead, the system proactively refreshes the cache before it expires. The flow is -

    1. Cache entry is about to expire
    2. Background process refreshes it
    3. Users keep getting fresh data

This reduces cache misses for popular data.

Suppose, a homepage is requested very often.

If the cache expires exactly when the traffic is high, many requests may fall through to the database. Refresh-ahead updates teh cache before expiry, so the users keep getting fast responses. This is especially useful for hot keys.

# CACHE-ASIDE VS READ-THROUGH

They are related but different. In both cases, if the data is not found in the cache, it is fetched from the database and then also stored in the cache.

    Cache-aside:
        App manages cache explicitly

    Read-through:
        Cache manages database load on miss

So, cache-aside gives more control to the application while read-through gives more abstraction.

# WRITE-THROUGH AND WRITE-BEHIND

There is a simple difference between the two -

    Write-through:
        Write reaches database immediately

    Write-behind:
        Write reaches database later

So, Write-Through is safer and fresher but has slower writes. Write-Behind has faster writes but has a risk of data loss.

# EXAMPLES

Let's say a product page is viewed more often. Cache-aside may work well -

    1. App checks Redis
    2. If miss, load from database
    3. Store product data in Redis

If the product price changes, we invalidate or update the cache. This is a very common approach.

Session storage usually uses a 'Write-Through' cache pattern. For example -

    session:abc123 -> user_id = 42

The session should be available quickly and usually stays fairly fresh.

For analytics count, 'Write-Behind' can make sense. For example page_views, click_counts, impression counts etc. This can often tolerate a small delay.

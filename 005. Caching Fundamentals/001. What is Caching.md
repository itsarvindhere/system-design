Caching is a technique used to store frequently accessed data in a faster storage layer so future requests can be served more quickly.

So, the basic idea is -

    Expensive to fetch once
    Cheaper to reuse many times

For example, let's say an application needs the same product details again and again. Instead of asking the database every time, we store that data in a cache -

    Client -> App -> Cache -> Database

If the data is in cache, the app can return it quickly. If not in cache, the app fetches it from the database, stores it in the cache, and then returns it.

# WHY DO WE NEED CACHING?

Databases and external services are often slower than the memory. For example -

    1. Database query
    2. Network call
    3. Complex computation
    4. File read
    5. Third-party API call

If the same data is requested repeatedly, doing the same operation again and again is wasteful.

Caching helps by keeping a copy of the result close to the application.

The benefit is that it improves the Latency, throughput, user experience, backend load, and the cost efficiency.

Suppose the user opens a product page. The app needs product details. Without cache -

    User -> App -> Database

Every request is going to hit the database. With cache -

    First request:
        User -> App -> Cache miss -> Database -> Cache -> Response

    Second request:
        User -> App -> Cache hit -> Response

The second request is much faster because the app did not need to go to the database again.

# WHAT CAN BE CACHED?

Many kinds of data can be cached such as Database query results, Product details, User profiles, Session data, API responses, Computed results, Config values, Leaderboards, Authentication tokens, Static assets and so on.

The best cache candidates are data that are read frequently, expensive to compute or fetch, not changing so frequently, and acceptable to be slightly stale in some cases.

# WHAT IS A CACHE?

A cache is a temporary, fast storage layer.

It usually stores copies of data from a slower source like a database or an external API. For example -

    Database -> Source of truth
    Cache -> Fast copy for quick access

A cache is usually much faster than the database because it is memory-based.

## CACHE IS NOT THE SOURCE OF TRUTH

This is important. The cache usually does not own the real data. The database usually does.

For example -

    Database says:
        product price = Rs. 1000

    Cache says:
        product price = Rs. 1000

If the database chances, the cache may still have an old copy of the data. So, the cache is usually treated as a performance layer, not the source of truth.

## WHY IS CACHE FAST?

Cache is fast because it is often stored in memory. Memory access is much faster than disk access. Also, cache systems usually use simple key-value based lookups like -

    product:123 -> product details

This is much faster than a complex database query or a join.

# CACHE KEY AND CACHE VALUE

A cache key is a unique name that is used to store and retrieve cached data. The application uses this key to find the cached value. For example -

    product:123
    user:456
    order:789
    homepage:popular

For example -

    GET product:123

If the key exists, it is a cache hit, otherwise it is a cache miss.

The cache value is the actual data that we store against the key. For example -

    product:123 -> {
        "name": "Phone",
        "price": 50000
    }

The value may be simple text, JSON, serialized object, or compressed binary data depending on the system.

# CACHE HIT AND CACHE MISS

A cache hit means that requested data is found in the cache, whereas the cache miss means the data is not found in the cache.

Cache hit is good because it is fast. Cache miss is slower because it required fallback to the database or other source.

So, a simple flow can be -

    1. Request comes in
    2. App checks cache
    3. If found, return data
    4. If not found, fetch from database
    5. Store result in cache
    6. Return result to user

## NOT EVERYTHING HAS TO BE CACHED!

We cannot cache everything forever. There are several reasons for it. For example, the memory is limited, the data can change, not all data is read often, maintaining cache freshness is not easy, and storing everything in the cache can be expensive.

# BENEFITS AND DOWNSIDES

Caching has many benefits like -

    1. Faster response time
    2. Lower database load
    3. Better throughput
    4. Lower infrastructure cost
    5. Better user experience
    6. Can absorb traffic spikes
    7. Helps reuse expensive computation results

There are also downsides like -

    1. Stale data
    2. Cache invalidation problems
    3. More moving parts
    4. Memory limits
    5. Cache misses
    6. Consistency issues
    7. Extra logic in application

So, use caching when -

    1. The same data is requested repeatedly
    2. Data is expensive to compute or fetch
    3. Small delay in freshness is acceptable
    4. Read traffic is high
    5. Backend systems are getting overloaded

Avoid caching when -

    1. Data changes very frequently
    2. Freshness is critical at all times
    3. Access is rare
    4. The data is already cheap to fetch
    5. Cache consistency would be too hard to manage

For example, very sensitive real-time data is not a good cache candidate.

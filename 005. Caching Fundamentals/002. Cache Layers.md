In the last section, we understood what is caching in general. Now, Caching can happen at different places.

Cache Layers mean the different places where the data can be cached, from the closest to the user to the closest to the database/application.

The main idea is -

    Cache as close as possible to where the data is needed.

Different kinds of data are required in different ways. For example -

    1. Static images and CSS files
    2. API responses
    3. Database query results
    4. Session data
    5. Computed values

Not all of these should be cached in the same place. Some data should be cached near the user, while some should be cached near the application. Some should be cached near the database.

So, cache layers help us choose the right caching location for the right workload.

Let's say the user opens a product page. The request may be served through multiple layers.

    Browser Cache -> CDN -> Reverse Proxy -> App Cache -> Database

If the image is already in the browser cache, we do not need to go further. If not, maybe th CDN has it. If not, maybe the Reverse Proxy has it. So, this layered approach reduces load at each level.

# CACHE LAYERS

## 1. BROWSER CACHE

This is the cache inside the user's browser. It stores things like Images, CSS Files, JavaScript files, Static assets, and sometimes API responses if allowed.

For example, if the browser already downloaded a logo image, it may reuse it instead of downloading it again. This improves page load speed and reduces bandwidth.

Browser cache is useful because many web pages reuse the same static files. For example logo.png, app.css, main.js and more. If every page load downloaded these files again, the website would be slower.

So, browser caching helps the same user reload pages faster.

## 2. CDN CACHE

A CDN stands for Content Delivery Network. A CDN caches content closer to the user geographically.

For example -

    User in India
    User in USA
    User in Europe

So, instead of all the users going to one central server, a CDN serves content from nearby edge locations. This reduces latency.

CDNs are great for images, videos, static assets, public web pages, and downloadable files. They help with faster delivery, lower origin load, better availability during traffic spikes, and better global performance.

For globally distributed apps, CDN caching is very important.

## 3. REVERSE PROXY CACHE

A reverse proxy sits in front of the application servers.

It can cache resources and serve them without forwarding each request to the backend.

Reverse proxies like NGINX or Varnish can cache content. This is useful for public pages, API responses, frequently requested content, and reducing app server load.

Reverse proxy cache sits closer to the application than the CDN cache. It can cache dynamic content with short TTLs. For example, home page, popular product list, public profile page, trending content, etc.

## 4. APPLICATION CACHE

This is used inside the backend application. For example, the app may cache -

    1. Database query results
    2. Computed values
    3. User profile data
    4. Permissions
    5. Configuration
    6. Feature flags

A common tool here is 'Redis.'

If the app needs the same data repeatedly, it can use Redis cache for that. Application cache is one of the most common layers in the backend systems.

Application cache is useful when the app frequently asks the database for the same data. Instead of hitting the database every time, the app checks Redis first.

This reduces database load and improves API latency.

## 5. DATABASE CACHE

Some databases also keep internal caches. For example, they may cache frequently accessed pages, index blocks, query results, and buffer pool pages.

This is usually handled internally by the database engine. For example, PostgreSQL and MySQL, keep data in memory buffers to speed up repeated access. This layer is lower level and usually not maintained directly by the application.

# CACHE LAYER ORDER

A request may pass through cache layers in this order -

    1. Browser cache
    2. CDN cache
    3. Reverse proxy cache
    4. Application cache
    5. Database cache

The closer the cache is to the user, the faster the response can be. but, the closer the cache is to the user, the less context it may have.

So, each layer has its role.

## WHAT SHOULD BE CACHED AT EACH LAYER?

### 1. BROWSER CACHE

This is good for Static files, Images, JavaScript, CSS, and sometimes public API responses

### 2. CDN CACHE

This is good for Static assets, Media files, Public content, and Global traffic.

### 3. REVERSE PROXY CACHE

This is good for public HTML pages, Read-heavy API responses, and short-lived dynamic content.

### 4. APPLICATION CACHE

This is good for database results, sessions, profiles, permissions, and computed values.

### 5. DATABASE CACHE

This is good for table pages, index pages, frequently accessed internal structures and more.

Using multiple cache layers gives us better performance at different stages -

    Browser cache:
        Helps the same user reload faster

    CDN cache:
        Helps users around the world

    Reverse proxy cache:
        Helps reduce app server load

    Application cache:
        Helps reduce database load

# CACHE INVALIDATION ACROSS LAYERS

With multiple cache layers, invalidation is more important. For example, if the product price changes -

    Database updated
    App cache invalidated
    Proxy cache invalidated
    CDN cache maybe invalidated
    Browser cache may expire later

Each layer may need different invalidation behavior. This is one reason cache design can get tricky.

Different cache layers often use different TTL (Time To Live). For example -

    Browser cache:
        Long TTL for static files

    CDN cache:
        Medium TTL

    Reverse proxy cache:
        Short TTL

    Application cache:
        Short or medium TTL

This is because each layer has different freshness requirement. Static assets can live longer in the cache whereas dynamic data usually needs shorter TTLs.

# LOCAL CACHE VS DISTRIBUTED CACHE

Cache layers can also be split by scope.

## 1. LOCAL CACHE

Local cache lives inside one application instance. For example -

    App Server 1 has its own memory cache

This is very fast because it is local memory. But, it is not shared across servers. If there are many app servers, each one may have a different cache view.

## 2. DISTRIBUTED CACHE

It is shared across app instances.

For example -

    All app servers -> Redis Cluster

This keeps cache data centralized and shared. But, it is slower than the local memory cache. It is more consistent across servers.

# CACHE LAYER TRADEOFFS

Each layer has its tradeoffs.

    Browser cache:
        Great for client-side reuse, but not shared between users

    CDN cache:
        Great globally, but best for public content

    Reverse proxy cache:
        Great for reducing backend load, but needs careful headers and invalidation

    Application cache:
        Great for business data, but freshness and consistency are tricky

    Distributed cache:
        Great for sharing across servers, but adds network latency and operational complexity

Some common mistakes with cache layers are -

    1. Caching the wrong data in the wrong layer
    2. Using very long TTLs for rapidly changing data
    3. Forgetting to invalidate lower layers
    4. Assuming browser cache and CDN cache behave the same
    5. Treating cache as the source of truth
    6. Using local cache when multiple servers need shared state
    7. Caching user-specific data in shared public layers without care

One common mistake is -

    Cache at one layer gets updated but another layer still serves stale data.

This is why layered caching needs a clear strategy.

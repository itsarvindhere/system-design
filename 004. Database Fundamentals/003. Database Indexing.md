Database Indexing is a technique used to make database queries faster.

An index is like a shortcut that helps the database find data without scanning the entire table.

For example, if we have a 'users' table with 10 Million rows -

    users
        id
        name
        email
        city
        created_at

Now, if we run this query -

    SELECT * FROM users
    WHERE email = 'arvind@example.com';

Without an index, the database may need to check each row. This is called a full-table scan.

    Check row 1
    Check row 2
    Check row 3
    ...
    Check row 10,000,000

With an index on 'email', the database can quickly jump to the matching row -

    email index -> find arvind@example.com -> fetch user row

So, indexing improves read performance.

# WHY DO WE NEED INDEXING?

Databases often store huge amounts of data. If every query scans the whole table, the system becomes slow. For example -

    Find user by email
    Get orders for user
    Search products by category
    Fetch latest messages in a chat
    Get transactions by account ID

These queries may run many times per second. Indexing helps the database find matching rows quickly.

Without indexes, even simple queries can become expensive as data grows.

It is like indexes in a book. If you want to find a topic in a book, you could read every page from the beginning. This is like a full table scan.

But, if a book has an index at the back, you can find the page number quickly -

    "Transactions" -> page 120

Database indexing works in the same way. They store searchable values with pointers to the actual rows -

    email index:

        arvind@example.com -> row 101
        rahul@example.com  -> row 205
        test@example.com   -> row 890

Now the database does not need to scan every user. It can search the index first.

# HOW INDEX WORKS?

An index stores selected column values in a stricture that is fast to search. For example, if we create an index on 'email' - 

    CREATE INDEX idx_users_email ON users(email);

The database maintains a separate data structure for email values. When we query by email -

    SELECT * FROM users
    WHERE email = 'arvind@example.com';

The database can use the index. So, the flow is -

    1. Search email in index
    2. Find pointer to matching row
    3. Fetch row from table
    4. Return result

So, the index does not replace the table. It is an additional structure maintained by the database.

Indexes exist in both SQL and NoSQL databases. In NoSQL databases like MongoDB, we can create indexes on document fields like -

    db.orders.createIndex({ userId: 1, createdAt: -1 });

In ElasticSearch, data is indexed for search by default. In Cassandra and DynamoDB, data modeling is more access-pattern driven. Indexing exists, but we often design partition keys and sort keys around queries instead of adding arbitrary indexes later.

So the exact indexing model depends on the database.

# B-TREE INDEX

The most common types of database index is a B-tree index. Most relational databases use a B-tree index by default. For example - PostgreSQL, MySQL, Oracle, SQL Server etc.

A B-tree keeps the values sorted. This makes it useful for queries like -

    1. Equality Search like 
    2. Range search
    3. Sorting
    4. Prefix matching in some cases

For example -

    WHERE email = 'a@example.com'
    WHERE age > 25
    WHERE created_at BETWEEN date1 AND date2
    ORDER BY created_at

B-tree indexes are general purpose and work well for many common queries.

# INDEX ON PRIMARY KEY

Most databases automatically create an index on the primary key. For example -

    users table:
        id PRIMARY KEY

The database creates an index on the 'id'.

So, this query is fast -

    SELECT * FROM users
    WHERE id = 123;

Primary key indexes are important because rows are often fetched by ID. Foreign keys are also commonly indexed, especially when used in joins.

# UNIQUE INDEX

A unique index makes sure duplicate values are not allowed. For example -

    CREATE UNIQUE INDEX idx_users_email ON users(email);

This ensures that two users cannot have the same email. So, unique index helps with both query performance and data integrity. For example -

    User 1 email = arvind@example.com
    User 2 email = arvind@example.com

The database rejects the second insert. This is better than checking uniqueness only in application code because the database can enforce it safely under concurrency.

# COMPOSITE INDEX

A composite index is an index on multiple columns. For example -

    CREATE INDEX idx_orders_user_status ON orders(user_id, status);

This index helps in queries like -

    SELECT * FROM orders
    WHERE user_id = 123
    AND status = 'paid';

Remember that the order of columns matters here. An index on -

    (user_id, status)

is not the same as -

    (status, user_id)

The database can usually use the leftmost columns of a composite index.

For example, if we have an index -

    (user_id, status, created_at)

This can help with a query that has -

    WHERE user_id = 123

AND 

    WHERE user_id = 123 AND status = 'paid' ORDER BY created_at

But, it may not help much with only -

    WHERE status = 'paid'

because user_id is the first column.

## LEFTMOST PREFIX RULE

The leftmost prefix rule means a composite index is most useful when the query uses the first column or first few columns in order. For example, an index -

    (user_id, status, created_at)

is useful for -

    WHERE user_id = 123

    WHERE user_id = 123
    AND status = 'paid'

    WHERE user_id = 123
    AND status = 'paid'
    AND created_at > '2026-01-01'

It is less useful for -

    WHERE status = 'paid'

because the query skipped "user_id". So when designing composite indexes, we should think carefully about query patterns.

# INDEX AND ORDER BY

Indexing can also help with sorting. Let's say we often run -

    SELECT * FROM orders
    WHERE user_id = 123
    ORDER BY created_at DESC;

An index like this can help -

    CREATE INDEX idx_orders_user_created
    ON orders(user_id, created_at DESC);

The database can use the index to find orders for the user already sorted by 'created_at'.

Without this index, the database may need to find matching rows, and then sort them separately.

# INDEX AND JOINS

Indexing is also useful for join. For example -

    SELECT * FROM orders
    JOIN users ON orders.user_id = users.id;

If 'user.id' is a primary key, it already has an index. But, 'orders.user_id' should also often be indexed.

    CREATE INDEX idx_orders_user_id ON orders(user_id);

This helps the database quickly find all orders for a user. Foreign key columns are commonly indexed because they are frequently used in joins and lookups.

# COVERING INDEX

A covering index is an index that contains all the columns needed by a query. For example -

    SELECT email FROM users
    WHERE id = 123;

If the database has an index containing both the 'id' and the 'email', then it may answer the query using only the index and it may not need to read the table row separately. For example -

    CREATE INDEX idx_users_id_email ON users(id, email);

This can be faster because the database reads less data.

This is sometimes called an index-only scan.

The downside is that these indexes use more memory since they are wider. So, do not use them blindly.

# SELECTIVITY

It means how well an index narrows down the data. A highly selective column has many unique or rare values. For example -

    user_id
    email
    order_id

An index on 'email' is usually useful because one email matches one user.

A low selectivity column has few possible values. For example -

    gender
    is_active
    status

If 90% of users have 'is_active' as 'true' then an index on 'is_active' may not help much. The database may decide that scanning the table is cheaper.

Good indexes usually filter out a large portion of rows.

# CARDINALITY

It means the number of distinct column values. High cardinality is there for columns like email, phone_number, user_id, order_id etc.

Low cardinality columns can be 'status', 'gender', and boolean fields.

High-cardinality columns often make better indexes. But low-cardinality columns can still be useful in composite indexes.

For example -

    CREATE INDEX idx_orders_status_created
    ON orders(status, created_at);

This may help if we frequently query recent paid orders -

    WHERE status = 'paid'
    ORDER BY created_at DESC

So, cardinality matters, but query pattern matters more.

# DOWNSIDES OF INDEXING

Indexing is useful, but there are downsides.

The first downside is storage. Indexes take extra disk space. If a table has many indexes, storage usage increases.

The second downside is slower writes. Whenever we insert, update, or delete rows, the database must also update index.  So too many indexes can slow down writes.

The third downside is maintenance overhead. Indexes can become bloated or fragmented depending on the database. They may need maintenance like vacuuming, rebuilding, or analyzing statistics.

So, the rule is -

    Indexes improve reads,
    but they add cost to writes and storage.

We should not blindly add index on each column. This may look helpful but it can hurt performance. Every write will now update many indexes and the database query planner may have more choices but not necessarily better performance.

Indexing should be based on real query patterns -

    1. Which queries are slow?
    2. Which columns are used in WHERE?
    3. Which columns are used in JOIN?
    4. Which columns are used in ORDER BY?
    5. How many rows does the query return?
    6. How often does this query run?

# EXPLAIN PLAN

To know whether a database is using an index, we can use an Explain plan. For example -

    EXPLAIN
    SELECT * FROM users
    WHERE email = 'arvind@example.com';

The database shows how it plans to execute the query. It may show -

    Index Scan using idx_users_email

OR

    Seq Scan on users

A sequential scan means the database is scanning the table.

An explain plan helps us understand the query performance.

# B-TREE INDEX

A B-tree index is a database index that stores values in a sorted tree-like structure so the database can find rows quickly.

Most relational databases use B-tree indexes as the default index type.

The main idea is -

    Keep indexed values sorted, so searching does not require scanning the whole table.

B-tree stands for 'Balanced Tree'. Balanced means the tree is kept roughly even, so searching does not become slow as data grows.

The database does not store the index as a simple list but stores in levels -

    Root Node
        -> Internal Nodes
            -> Leaf Nodes

The root helps decide where toe search. The internal nodes guide the search and the leaf nodes contain the actual indexed values and pointers to the table rows.

Let's say we create an index on names -

    Alice
    Bob
    David
    Karan
    Rahul
    Zoya
        
A simplified B-tree might be like -

                    [Karan]
                    /       \
        [Alice Bob David]   [Rahul Zoya]

Now, if we search for 'Rahul' -

    1. Start at root: Karan
    2. Rahul is greater than Karan
    3. Go right
    4. Find Rahul

The real database structure is more complex, but the idea is the same. The tree helps the database eliminate large parts of data quickly.

The most important property of a B-tree index is that it keeps values sorted. And since the values are sorted, B-tree indexes are useful for exact lookups, range queries, sorting, and prefix searches in some cases.

B-trees are fast because each node can contain many keys and many child pointers. This means the tree is wide and shallow. So the database can eliminate huge portions of data at each level. This is very disk-friendly.

A normal binary search tree has at most two children per node. But a B-tree node can have many children and this makes them much shorter.

Databases prefer B-trees because data is stored on disk in pages or blocks. Reading one page from disk is expensive. So, it is better to store many keys in one page and reduce the number of page reads.

# CLUSTERED INDEX

A clustered index determines the physical order of data in a table.

Think of it like a dictionary where words are physically arranged alphabetically so if you know a word, you can quickly jump to the right section. The actual data is stored in that order.

The same idea applies to a clustered index. So, if we have a table with an 'id' and a 'name', without a clustered index, the rows may be stored in any order on disk.

But, if we create -

    CREATE CLUSTERED INDEX idx_id
    ON users(id);

The database physically stores row in order based on 'id'. Now, searching for something like 'id = 3' is very efficient as data itself is ordered.

If a table is clustered by the primary key, rows are stored based on primary key order. This can make range scans very efficient like -

    WHERE id BETWEEN 1000 AND 2000

Usually, a table can have only one clustered index because data can only be physically ordered one way.

Different databases handle this differently.

For example, SQL Server has explicit clustered indexes. MySQL InnoDB stores table rows organized by primary key. PostgreSQL does not maintain automatic clustered order in the same way after updates unless explicitly clustered.

# NON-CLUSTERED INDEX

A non-clustered index is a separate structure from the table data. It stores indexed column values and pointers to actual rows.

For example -

    email -> pointer to row

A table can have many non-clustered indexes. Most secondary indexes are non-clustered. The database first searches the index, then fetches the row from the table.

# HASH INDEX

A hash index uses a hash-table like structure. It is useful for exact-match lookups like -

    WHERE email = 'arvind@example.com'

But hash indexes are usually not useful for range queries like -

    WHERE created_at > '2026-01-01'

B-tree indexes are more general because they support equality, ranges, and ordering.

Hash indexes are useful in specific cases depending on the database.

# FULL-TEXT INDEX

It is used for a text search like -

    Search products containing "wireless headphones"

A normal B-tree index is not good for searching words inside long text. Full-text indexes break text into tokens and make searching faster. They are useful for things like Comments, Documents, Articles, Product descriptions etc.

Databases like PostgreSQL and MySQL support full-text indexing. Search engines like Elasticsearch are specialized for this kind of indexing.

Full-text search commonly uses an inverted index that maps words to documents or rows that contain those words.

For example -

    "database" -> doc1, doc3, doc8
    "indexing" -> doc3, doc9
    "transactions" -> doc2, doc7

This makes word search fast. Search engines like Elasticsearch, Solr, and Lucene use inverted indexes heavily.

# PARTIAL INDEX

A partial index indexes only some rows. For example, if we often query active users -

    SELECT * FROM users
    WHERE is_active = true;

We can create a partial index -

    CREATE INDEX idx_active_users
    ON users(email)
    WHERE is_active = true;

Notice that we created an index only for active users. Partial indexes reduce storage and maintenance cost when we only care about a subset of data.

They are quite useful when a filter is very common.

# INDEXING STRATEGY

A good indexing strategy starts from queries, not columns. We should look at the most important queries. For each query, we we should check -

    1. WHERE conditions
    2. JOIN columns
    3. ORDER BY columns
    4. GROUP BY columns
    5. LIMIT usage
    6. How often the query runs
    7. How many rows it returns

For example, if we often run -

    SELECT * FROM orders
    WHERE user_id = 123
    ORDER BY created_at DESC
    LIMIT 20;

A good index may be -

    CREATE INDEX idx_orders_user_created
    ON orders(user_id, created_at DESC);

This supports filtering by user and sorting by time.

# COMMON MISTAKES

Some common indexing mistakes are -

    1. Not indexing foreign keys
    2. Adding indexes on every column
    3. Ignoring composite index order
    4. Indexing low-selectivity columns without a clear query reason
    5. Forgetting indexes slow down writes
    6. Not using EXPLAIN plans
    7. Creating duplicate indexes
    8. Not indexing ORDER BY patterns
    9. Using functions on indexed columns in a way that prevents index usage
    10. Not updating database statistics

One common mistake is -

    WHERE LOWER(email) = 'arvind@example.com'

Even if there is an index of 'email', the database may not use it because the query applies a function. In such cases, we may need a functional index depending on the database.

Add an index when -

    1. A query is slow
    2. The table is large
    3. The query runs frequently
    4. The query filters by specific columns
    5. The query joins on specific columns
    6. The query sorts large result sets
    7. The index filters out many rows

Avoid adding an index when -

    1. The table is tiny
    2. The column is rarely queried
    3. The column changes very frequently
    4. The query returns most rows anyway
    5. The index duplicates an existing index
    6. Write performance is more important

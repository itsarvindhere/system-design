In the last section, we learnt about Normalization which is used to avoid duplicate data and ensure data consistency.

There is also a term called Denormalization which means intentionally storing duplicate or derived data to make reads faster.

For example, in a normalized design -

    orders table:
        order_id
        user_id

To show the order details, we join the orders tables with the users table.

On the other hand, in the denormalized design -

    orders table:
        order_id
        user_id
        user_name
        user_email

Now, the order page can show user information without joining the users table.

This improves read performance but now, user data exists in multiple places.

# WHY DO WE NEED DENORMALIZATION?

It is useful when reads are frequent or expensive. For example -

    1. Feed pages
    2. Order details pages
    3. Dashboards
    4. Reports
    5. Search pages
    6. Product listing pages
    7. High-scale read APIs

If a query runs millions of times and joins many tables, denormalization can reduce the database load. The main idea is -

    Do extra work during writes, so reads become faster.

# EXAMPLE

Common denormalization examples are -

    1. Store user_name in orders
    2. Store product_name and price in order_items
    3. Store comment_count on posts
    4. Store like_count on posts
    5. Store average_rating on products
    6. Store daily revenue summary
    7. Store read models for dashboards

For example -

    posts table:

        post_id
        title
        like_count
        comment_count

Instead of counting likes and comments every time, we simply store the counts.

# PRE-COMPUTED DATA

Denormalization often stores precomputed data.

For example -

    Product average rating

Without denormalization we have to do -

    SELECT AVG(rating) FROM reviews
    WHERE product_id = 123;

With denormalization -

    products table:
        product_id
        average_rating
        review_count

Now, product pages load faster. But, whenever a new review is added, we need to update these values.

# DENORMALIZATION AND HISTORICAL DATA

Sometimes, denormalization is not just for performance but is needed for historical accuracy. For example, when a user places an order, we may store -

    product_name
    product_price
    seller_name

inside an order item.

Why?

Because product details can change later. If the product price changes tomorrow, the old order should still show the price at the purchase time.

So, this is correct -

    order_items:
        product_id
        product_name_at_purchase
        price_at_purchase

This is duplicate data, but it preserves history.

# BENEFITS OF DENORMALIZATION

Denormalization has several benefits including -

    1. Faster reads
    2. Fewer joins
    3. Lower query complexity
    4. Better performance for high-traffic read paths
    5. Useful for dashboards and reports
    6. Helpful in distributed and sharded systems
    7. Can preserve historical snapshots

# DOWNSIDES OF DENORMALIZATION

Here are some downsides -

    1. Duplicate data
    2. More storage usage
    3. More complex writes
    4. Risk of stale data
    5. Risk of inconsistent data
    6. More update logic
    7. Harder debugging

For example, if user name is stored in both 'users' and 'orders' table, they can become inconsistent.

For example -

    users.name = Arvind Rana
    orders.user_name = Arvind

So, we need to now decide whether that is acceptable.

# SOURCE OF TRUTH

When denormalizing, we must know which data is the source of truth. For example -

    users table:
        source of truth for current user email

    orders table:
        stores email snapshot at order time

if the order email is historical, it does not need to update when the user email changes. But if it is supposed to show the current email, we need a sync strategy.

So, always ask -

    Is this duplicate data a snapshot?
                Or
    should it stay synchronized?

# EVENTUAL CONSISTENCY

Denormalized data may be updated asynchronously. For example -

    User updates name
    users table updates immediately
    background job updates posts.author_name later

For a short time, posts may show the old name but after some time, they will update to show the latest name. This is eventual consistency. This is acceptable for some systems but not acceptable for others. For example -

    Bank account balance:
        stale value may be dangerous

# WHEN TO DENORMALIZE

Denormalize when -

    1. Read performance is a bottleneck
    2. Joins are too expensive
    3. The same data is read together often
    4. Aggregations are expensive
    5. Slightly stale data is acceptable
    6. You need historical snapshots
    7. You are designing read models
    8. You are avoiding cross-shard joins

Do not denormalize just because it sounds scalable. There needs to be a clear reason.

Some practical uses include -

    Social media feed:
        Store author name and profile image with feed item

    E-commerce order:
        Store product name and price in order item

    Video platform:
        Store view_count and like_count on video

    Dashboard:
        Store daily summary table

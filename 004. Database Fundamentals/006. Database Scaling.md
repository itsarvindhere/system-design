Simply put, Database Scaling means increasing the capacity of a database so it can handle more data, more traffic, and more queries without becoming slow or unavailable.

A database can become a bottleneck because of limits in -

    1. CPU
    2. Memory
    3. Disk I/O
    4. Storage
    5. Network bandwidth
    6. Number of connections
    7. Lock contention
    8. Write throughput

So database scaling is about understanding the bottleneck and applying the right solution.

# OPTIMIZE BEFORE SCALING

Before adding more machines, we should first optimize the existing database. Some common first steps are -

    1. Fix slow queries
    2. Add proper indexes
    3. Use pagination
    4. Avoid N+1 queries
    5. Fetch only required columns
    6. Use connection pooling
    7. Archive old data
    8. Remove unused indexes

A missing index or bad query can make even a powerful database slow.

# VERTICAL SCALING

It means making the existing database server more powerful by adding more CPU, RAM, using faster SSD, increasing disk capacity etc.

This is the simplest scaling approach because the application usually does not need major changes.

But, vertical scaling has limits.

Eventually, one machine may not be enough or may become too expensive.

# HORIZONTAL SCALING

It means adding more database servers. Some common ways are -

    1. Read replicas
    2. Sharding
    3. Partitioning
    4. Distributed databases

Horizontal scaling can handle larger workloads, but it adds complexity. For example -

    1. Replication can introduce stale reads.
    2. Sharding can make joins and transactions harder.
    3. Partitioning requires good partition keys.]

# READ SCALING

If the database is read-heavy, we can scale reads using -

    1. Caching
    2. Read replicas
    3. Materialized views
    4. Denormalized read models

For example -

    Product details -> Cache
    Product browsing -> Read replica
    Dashboard numbers -> Materialized view

Read scaling is usually easier than write scaling.

# WRITE SCALING

If writes become the bottleneck, scaling is harder. Some common options are -

    1. Batch writes
    2. Queue-based async writes
    3. Partitioning
    4. Sharding
    5. Reducing unnecessary writes
    6. Using a database designed for high write throughput

For example, event logs may be written to a system like Cassandra, DynamoDB, Kafka, or a time-series database depending on the use case.

# MATERIALIZED VIEWS

A normal view stores only the query definition.

A materialized view stores the actual query result. For example, let's say we frequently need total sales per product. Instead of calculating it each time -

    SELECT product_id, SUM(amount)
    FROM orders
    GROUP BY product_id;

We can store the result. This stored result is called a materialized view.

It is useful because some queries are expensive. For example -

    1. Aggregating millions of rows
    2. Joining many large tables
    3. Calculating dashboard metrics
    4. Computing reports
    5. Preparing denormalized read models

If the same expensive query runs again and again, it wastes database resources.

A materialized view avoids recomputing the result every time. Instead of running expensive query every request, we store the query result once and read the stored result quickly.

So, it is like a cache inside a database. We can say it is -

    Precomputed query result stored as a table-like object.

The issue in this is that the data may become stale. Suppose the materialized view says -

    Product 101 total sales = 6000

Then a new order is placed -

    Product 101 total sales = 6000

The real total should now be 7000. But the materialized view may still show 6000 until it is refreshed.

So materialized views improve read speed, but the result may not always be up to date.

# DENORMALIZATION

It means intentionally storing duplicate or precomputed data to make reads faster. In a fully normalized database, we try to reduce duplication like -

    users table:
        id
        name
        email

    orders table:
        id
        user_id
        total

Here, the orders table stores only the user_id, not the user's name and email. If we need order details with user information, we join tables -

    orders + users

In denormalization, we may store some user details directly inside orders -

    orders table:
        id
        user_id
        user_name
        user_email
        total

Now reading order details is faster because we may not need to join with users.

But, we introduce duplicate data. So, the tradeoff is -

    Faster reads,
    but more complex writes and updates.

# CONNECTION POOLING

Connection pooling helps control database connections.

Instead of opening a new connection for every request, the application reuses existing connections.

This prevents the database from being overloaded by too many connections.

For PostgreSQL, tools like PgBouncer are commonly used.

# CHOOSE BASED ON BOTTLENECK

    Slow queries:
        Indexing, query optimization

    Too many repeated reads:
        Caching

    Too many reads:
        Read replicas

    Too many writes:
        Sharding, batching, async processing

    Table too large:
        Partitioning, archiving

    Too many connections:
        Connection pooling

    Global read latency:
        Multi-region replicas

Common scaling path should be -

    1. Start with one good relational database
    2. Add indexes and optimize queries
    3. Add caching for hot data
    4. Add read replicas for read-heavy traffic
    5. Partition or archive large tables
    6. Use queues for asynchronous processing
    7. Denormalize hot read paths if needed
    8. Shard only when one database cannot handle size or write load

Sharding is powerful, but it should usually come later because it adds a lot of complexity.

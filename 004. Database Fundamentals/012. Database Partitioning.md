We have already gone through the terms 'Sharding' and 'Replication'. So, what is 'Partitioning'? Is is any different?

Well, Database Partitioning means splitting a large table into smaller pieces called partitions sho the database can manage and query the data more efficiently.

For example, instead of keeping all the orders in one huge table, we may split them by month -

    orders_2026_01
    orders_2026_02
    orders_2026_03

Or, the database may manage those partitions internally as one logical table.

# WHY DO WE NEED PARTITIONING?

As the tables grow, queries can become slower and maintenance becomes harder. For example -

    1. Millions of orders
    2. Billions of logs
    3. Huge event tables
    4. Large time-series data
    5. Massive audit tables

If all the data lives in one giant table, then -

    1. Queries may scan too much data
    2. Indexes may become large
    3. Deletes and archival become expensive
    4. Maintenance takes longer
    5. Backup and restore become harder

Partitioning helps reduce the amount of data the database needs to touch for a query.

# EXAMPLE

Let's say we have an 'orders' table with 500 million rows. Most queries are about recent orders like -

    SELECT * FROM orders
    WHERE created_at >= '2026-06-01'
    AND created_at < '2026-07-01';

If the table is partitioned by month, the database may only scan the 'June' partition.

    orders_jan
    orders_feb
    orders_mar
    ...
    orders_jun

This is faster than scanning the entire table.

# PARTITIONING VS SHARDING

The two are related, but they are not the same.

Partitioning means splitting the data into smaller pieces, often inside the same database system.

Sharding means splitting the data across multiple database servers.

For example -

    Partitioning:
        One database
        Table split into partitions

    Sharding:
        Multiple databases/servers
        Data split across machines

So, 'Partitioning' is usually about logical organization while 'Sharding' is about horizontal distribution across servers.

In many real systems, each shard may have partitions inside it.

# HOW PARTITIONING WORKS

A partitioned table is still treated like one logical table by the application. For example -

    SELECT * FROM orders
    WHERE created_at >= '2026-06-01'
    AND created_at < '2026-07-01';

The application queries 'orders.'

The database decides which partition contains the matching rows.

This is important because the application does not need to know about each and every partition.

# PARTITION KEY

The partition key is the column used to decide where a row goes. For example -

    created_at
    user_id
    region
    status

If we partition by 'created_at,' then the database places the rows into partitions based on the date or time.

If we partition by 'region,' then rows go into partitions based on the region.

The partition key is really important because it determines -

    1. How data is distributed
    2. Which queries are fast
    3. Which queries may still be slow

# TYPES OF PARTITIONING

## 1. RANGE PARTITIONING

Range partitioning splits the data by a range of values.

For example -

    orders_2026_01 -> January 2026
    orders_2026_02 -> February 2026
    orders_2026_03 -> March 2026

Or by ID range -

    1 - 1,000,000
    1,000,001 - 2,000,000
    2,000,001 - 3,000,000

Range partitioning is useful for Time-based data, Logs, Events, and Historical Records. It is very common because many queries are time-based. For example -

    Show orders from the last 30 days

This can scan only the recent partition. But, range partitioning can create hot partitions if new writes always go to the latest range.

## 2. LIST PARTITIONING

It splits data by a specific set of values. For example -

    India partition
    USA partition
    Europe partition

OR 
    
    pending partition
    paid partition
    cancelled partition

This is useful when the number of categories is limited and known in advance. For example -

    SELECT * FROM orders
    WHERE region = 'India';

The database can go directly to the India partition. It is good when queries often filter by a known category.

## 3. HASH PARTITIONING

It uses a hash function on the partition key. For example -

    partition = hash(user_id) % 4

This distributes the rows more evenly.

For example -

    user_id 101 -> Partition 1
    user_id 102 -> Partition 3
    user_id 103 -> Partition 0

It is useful when we want even distribution and it avoids hot partitions. But, range queries are less efficient because data is spread out. For example if we do -

    Get all users created between two dates

This may touch many partitions if the partition key is not time-based.

## 4. KEY PARTITIONING

This is similar to hash partitioning. The database uses a built-in mechanism to distribute rows based on one or more columns. This is database specific, but the idea is the same -

    distribute rows across partitions using a key

It helps balance the data and the load.

## 5. COMPOSITE PARTITIONING

It means using more than one partitioning strategy. For example -

    First partition by range on date
    Then sub-partition by hash on user_id

This can be useful in very large tables. For example -

    orders
        partitioned by month
        sub-partitioned by hash(user_id)

This helps combine benefits like -

    1. Time-based pruning
    2. Better load distribution
    3. More balanced partition sizes

But, it also adds complexity.

# PARTITION PRUNING

It means the database skips partitions that cannot possibly contain matching rows. For example -

    SELECT * FROM orders
    WHERE created_at >= '2026-06-01'
    AND created_at < '2026-07-01';

If the table is partitioned by month, the database can ignore all the other months. This reduces the work significantly.

Without pruning, the database may still need to inspect every partition.

So, partition pruning is one of the main performance benefits of partitioning.

Suppose, we have these partitions -

    orders_jan
    orders_feb
    orders_mar
    orders_apr
    orders_may
    orders_jun

If we query June order only, the database may skip all the other partitions and scan only the june partition. This is partition pruning.

# PARTITIONING AND INDEXES

Partitions can have their own indexes. For example, if each partition is smaller, indexes on that partitions may also be smaller. This can help performance. For example -

    orders_jun index on created_at
    orders_jul index on created_at

But remember that Partitioning does not automatically replace indexing. They solve different problems -

    Index:
        Helps find rows inside a table or partition

    Partition:
        Helps reduce how much data must be scanned

Often, both are used together.

# PARTITIONING AND LARGE TABLE MAINTENANCE

Partitioning makes maintenance easier.

Let's say we want to delete old logs older than 1 year. Without partitioning -

    DELETE FROM logs WHERE created_at < '2025-01-01';

This may scan and delete millions of rows. With partitioning -

    DROP PARTITION logs_2024

That is much faster. This is a big reason why partitioning is so popular for time-series, logs, and audit tables.

# BENEFITS AND DOWNSIDES OF PARTITIONING

Benefits include -

    1. Faster queries for partition-aware workloads
    2. Smaller indexes per partition
    3. Easier archival of old data
    4. Easier deletion of old data
    5. Better maintenance
    6. Better manageability for huge tables
    7. Can improve backup and restore workflows
    8. Can improve pruning of irrelevant data

Some downsides are -

    1. More design complexity
    2. Bad partition key can hurt performance
    3. Some queries still touch many partitions
    4. Cross-partition queries can be expensive
    5. Too many partitions can create overhead
    6. Partition maintenance adds operational work
    7. Not every database feature works equally well with partitions

So partitioning helps, but only if the data access pattern matches the partitioning strategy.

# WHEN TO PARTITION AND WHEN NOT TO PARTITION?

Partitioning is useful when -

    1. Tables are very large
    2. Queries often filter by one dimension
    3. Data is naturally time-based
    4. Old data is rarely queried
    5. Deleting or archiving data is expensive
    6. The table is growing too large for comfortable maintenance

Some good use cases include Logs, Events, Orders by time, Audit records, Metrics, IoT readings, Activity streams etc.

Avoid partitioning when -

    1. The table is small
    2. Queries are random and not partition-friendly
    3. You do not know the access pattern yet
    4. Maintenance complexity is not worth it
    5. The partition key is unclear
    6. Partitioning would create hot partitions

Remember, Partitioning is not the first optimization step for most systems.

Usually, we first look at Indexing, Query tuning, Caching, and Replication. Then partitioning if the table is still too large.

Choosing a wrong partition key is usually the biggest mistake. For example, partitioning orders by 'status' may seem simple -

    paid
    pending
    cancelled

But, if 95% orders are 'paid', one partition becomes huge and hot.

This is a bad partition key.

Another bad example if partitioning by a very low-cardinality column like 'gender' or boolean flag or status.

These do not usually make good partition keys on their own.

A good partition key usually has -

    1. Enough distinct values
    2. Even distribution
    3. Strong query alignment

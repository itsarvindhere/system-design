When it comes to databases, there are two broad categories -

    1. SQL
    2. NoSQL

SQL databases are usually relational databases and they store the data in tables with rows and columns.

On the other hand, NoSQL databases are non-relational databases. They can store the data in different formats like documents, key-value pairs, wide columns, graphs etc.

For example, MongoDB stores documents, Redis stores key-value data, Cassandra stores wide-column data, and Neo4j stores graph data.

The main difference is -

    SQL:
        Structured data, tables, relationships, strong consistency

    NoSQL:
        Flexible data models, horizontal scalability, high availability

# WHY DO WE NEED DIFFERENT TYPES OF DATABASES?

The thing is, not all applications store and access data in the same way.

For example, a banking system needs strong consistency -

    Transfer money from Account A to Account B

This needs correctness.

A social media feed, however, needs high scale and fast reads -

    Show posts from people I follow

This needs performance and availability.

A product catalog may have flexible fields -

    Shoes have size
    Phones have storage
    Books have author

This needs a flexible schema.

A recommendation system may need relationship traversal -

    Users similar to this user
    Products frequently bought together

This may need graph-like access patterns.

So, different databases exist because different systems have different needs.

# SQL DATABASE

SQL Databases are relational databases and they store data in tables. Tables can be related using keys -

    Orders.user_id -> Users.id

This means, each order belongs to a user.

SQL databases use SQL (Structured Query Language).

Some common examples include PostgreSQL, MySQL, Oracle. SQL Server, MariaDB, etc.

We call them 'Relational' because they organize data into tables. Each table represents an entity like Users, Products, Orders, Payments, etc.

Each row of a table represents one record and each column represents one attribute. For example -

    users table:
        id
        name
        email
        created_at

Relationships are represented using foreign keys. For example -

    orders.user_id references users.id

This allows the database to enforce rules like -

    Do not create an order for a user that does not exist.

This is called a 'Referential Integrity.'

## SCHEMA IN SQL

SQL databases usually have a pre-defined schema. A schema defines the structure of teh data. For example -

    users table:
        id          integer
        name        varchar
        email       varchar
        created_at  timestamp

The database enforces this structure. If the 'email' is required, the database can reject rows that do not have an email. If 'user_id' must reference an existing user, the database can enforce that.

This gives strong data integrity.

But, schema changes can require migrations. For example, if we want to add a new column -

    ALTER TABLE users ADD COLUMN phone varchar;

This is manageable, but in large systems, schema changes needs planning.

## JOINS IN SQL

One major strength of SQL databases in Joins. A join combines data from multiple tables.

For example, let's say we want all the orders with user details -

    SELECT orders.id, users.name, orders.total
    FROM orders
    JOIN users ON orders.user_id = users.id;

This is useful because data can be normalized. Instead of storing user information inside every order, we store the user once and reference it.

This reduces duplication

## NORMALIZATION

Normalization means organizing data to reduce duplication and maintain consistency. For example, instead of storing user email in every order -

    order_id    user_email
    1           arvind@example.com
    2           arvind@example.com
    3           arvind@example.com

We store the user once -

    users table:
        id    email

    orders table:
        id    user_id

This avoids duplication. If the user changes the email, we update it only in one place.

# NOSQL DATABASES

NoSQL stands for Not-Only-SQL. They are non-relational databases and they do not follow the traditional table-based relational model.

They are designed for different data models and scaling needs.

Some examples are MongoDB, Redis, Cassandra, DynamoDB, etc.

NoSQL databases are often used when we need flexible schema, high scalability, or special access patterns.

### DOCUMENT DATABASE

Document databases store data as documents, often JSON-like. For example - MongoDB. A user document may look like -

    {
        "_id": "user_123",
        "name": "Arvind",
        "email": "arvind@example.com",
        "addresses": [
            {
                "city": "Delhi",
                "type": "home"
            }
        ]
    }

This document can contain nested data. This is useful when related data is often read together. For example, a product document may contain product details, images, attributes, and metadata.

Document databases are useful in Product catalogs, content management, user profiles, flexible data models, event data, etc.

### KEY-VALUE STORES

Key-value stores store the data as simple key-value pairs. For example - Redis.

For example -

    user:123 -> {"name": "Arvind", "email": "arvind@example.com"}

Or

    session:abc123 -> user_123

Key-value stores are very fast for simple lookups and they are commonly used for caching, session storage, rate limiting, feature flags, counters, and temporary data.

Redis is a popular key-value store that stores data in memory, hence it is very fast.

### WIDE-COLUMN DATABASES

They store data in rows and columns, but they are different from traditional relational databases. For example - Cassandra.

They are designed for very high write throughput and horizontal scalability. They are useful when data is accessed by known query parameters. For example -

    user_activity_by_user_id

Each row may store activity events for a user.

They are commonly used for Time-series data, logs, IOT data, high-volume writes, event tracking, and large-scale distributed systems.

Cassandra tables might look similar to relational database tables but there is no concept of foreign keys or true joins. Instead, we design tables around queries. So, if we need -

    Get all orders for a user

We create a table specifically for that query -

    CREATE TABLE user_orders (
        user_id UUID,
        order_id UUID,
        amount DECIMAL,
        PRIMARY KEY (user_id, order_id)
    );

If we also need -

    Get order by order_id

We create another table -

    CREATE TABLE orders_by_id (
        order_id UUID,
        user_id UUID,
        amount DECIMAL,
        PRIMARY KEY (order_id)
    );

So, the same data exists in multiple tables.

Since Cassandra lacks fundamental relational concepts like True Joins, foreign keys, referential integrity, etc, they are not called relational databases.

A relational database says -

    "Store data once. I'll figure out relationships at query time."

Cassandra says -

    "Tell me your queries upfront. I'll store the data in a way that makes those queries fast."

### GRAPH DATABASES

They store data as nodes and edges. For example - Neo4J. They are useful when relationships are the main focus. Some common use cases are Social networks, recommendation systems, fraud detection, knowledge graphs, etc.

## SCHEMA FLEXIBILITY

One benefit of NoSQL databases is the schema flexibility. In a document database, different documents can have different fields -

    Product 1:
    {
        "name": "Phone",
        "storage": "128GB",
        "color": "Black"
    }

    Product 2:
    {
        "name": "Shoes",
        "size": 9,
        "material": "Leather"
    }

This is useful when different records have different attributes.

In SQL, we may need many nullable columns, separate attribute tables, or JSON columns.

# SCALING

SQL databases can scale vertically and horizontally but traditionally, they are easier to scale vertically. It means, adding more CPU, RAM, or disk to the same machine.

For horizontal scaling, SQL databases use techniques like Read replicas, Partitioning, Sharding, etc.

Many NoSQL databases are designed for horizontal scaling from the very beginning. For example, Cassandra and DynamoDB can distribute data across a lot of nodes. This allows large-scale reads and writes.

NoSQL databases often make tradeoffs.

They may sacrifice some relational features, joins, or strong consistency to gain scalability and availability.

For example, instead of joining tables, data may be denormalized. Denormalization means storing duplicate data to make reads faster. But, it creates a challenge. We may have to make changes in multiple places when some attribute changes.

# CONSISTENCY

SQL databases often provide strong consistency which means after a write is committed, future reads should see the latest data. For example -

    Update account balance
    Read account balance
    Get latest value

NoSQL databases vary.

Some provide strong consistency, some provide eventual consistency. Eventual Consistency means all replicas will become consistent eventually, but a read immediately after a write may return old data. 

This may be acceptable for cases like social media likes, view counts, recommendations, etc.

# SQL vs NoSQL EXAMPLE

Let's say we are building an e-commerce application. In SQL, we may design tables like -

    users
    products
    orders
    order_items
    payments

An order may be split across multiple tables -

    orders table:
        id
        user_id
        status
        total

    order_items table:
        order_id
        product_id
        quantity
        price

To get order details, we join tables -

    orders + order_items + products

In NoSQL, we may store an order as one document -

    {
        "orderId": "order_123",
        "userId": "user_456",
        "status": "paid",
        "items": [
            {
                "productId": "product_1",
                "name": "Phone",
                "quantity": 1,
                "price": 79999
            }
        ],
        "total": 79999
    }

This is easy to read in one query but if product name changes, old order documents may still have old product name. Sometimes that is okay because orders should preserve historical product details. This shows an important point -

    Denormalization is not always bad.
    Sometimes duplicated data is intentional.

# WHEN TO USE WHICH

Use SQL when data is structured and relationships matter. For example -

    1. Banking systems
    2. Payment systems
    3. Order management
    4. Inventory systems
    5. Booking systems
    6. Accounting systems
    7. CRM systems
    8. Admin dashboards
    9. Reporting systems

SQL is a good choice when we need strong consistency, ACID transactions, Complex queries, Joins, clear schema, and data integrity constraints.

Use NoSQL when the data model, scale or access pattern fits better. For example -

    1. Caching
    2. Session storage
    3. Product catalogs with flexible attributes
    4. Large-scale event logging
    5. Real-time analytics ingestion
    6. User activity feeds
    7. IoT data
    8. Recommendation graphs
    9. High-volume key-value lookups

NoSQL is a good choice when we need Flexible schema, high horizontal scalability, very high write throughput, low-latency key-value access, special data models like graphs or documents, etc.

But NoSQL should not be chosen only because it sounds more scalable. The database should match the problem.

# POLYGLOT PERSISTENCE

In real systems, companies often use multiple database for different needs.

This is called Polyglot Persistence.

For example, an e-commerce system may use -

    PostgreSQL:
        Orders, payments, users

    Redis:
        Cache, sessions, rate limits

    Elasticsearch:
        Product search

    MongoDB:
        Product catalog metadata

    Cassandra:
        Clickstream analytics

So, each database solves a different problem. The goal is not to pick one database for everything. The goal is to use the right database for right workload.

# COMMON MISUNDERSTANDINGS

## 1. NOSQL DOES NOT MEAN NO STRUCTURE

Even if the database does not enforce a strict schema, the application still expects some structure. For example, a mongoDB collection may still expect documents with userId, name, and email. The schema may be flexible, but the application still needs discipline.

## 2. SQL CAN ALSO SCALE

SQL databases can scale very far. Many large systems use PostgreSQL, MySQL or distributed SQL databases. Some scaling techniques include Indexing, Read replicas, Partitioning, Sharding, caching etc.

So, instead of assuming SQL cannot scale, a better statement is -

    Traditional SQL write scaling can be harder than some NoSQL systems.

## 3. NOSQL IS NOT ALWAYS FASTER

It all depends on the query. A key-value lookup in redis can be extremely fast. But a complex query that needs filtering across many fields may be difficult in some NoSQL databases.

So performance depends on workload and data model.

## 4. NOSQL DOES NOT ALWAYS MEAN EVENTUAL CONSISTENCY

Some NoSQL databases provides strong consistency for certain operations. For example, MongoDB and DynamoDB.

So, the exact behavior depends on the database and the configuration.

# COMPARISON

    Data model:
        SQL -> Tables
        NoSQL -> Documents, key-value, wide-column, graph

    Schema:
        SQL -> Fixed schema
        NoSQL -> Flexible schema

    Query language:
        SQL -> SQL
        NoSQL -> Depends on database

    Relationships:
        SQL -> Strong support with joins
        NoSQL -> Usually handled by embedding or denormalization

    Transactions:
        SQL -> Strong ACID support
        NoSQL -> Varies by database

    Scaling:
        SQL -> Traditionally vertical, plus replicas/sharding
        NoSQL -> Often designed for horizontal scaling

    Consistency:
        SQL -> Usually strong consistency
        NoSQL -> Strong or eventual depending on system

    Best for:
        SQL -> Structured relational data
        NoSQL -> Flexible, large-scale, specialized workloads

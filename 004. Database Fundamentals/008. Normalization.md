Normalization in a Database means organizing the data to reduce duplication and improve consistency.

But, why do we need Normalization?

In databases, the data can appear in many places. For example, if we store order in a order table like this -

    orders table:

    order_id    user_name    user_email          product_name    price
    1           Arvind       arvind@example.com  Phone           50000
    2           Arvind       arvind@example.com  Headphones      3000
    3           Rahul        rahul@example.com   Laptop          90000

Here, the user information is repeated in every order. If 'Arvind' changes his email, we need to update a lot of rows. If we miss one row, the database is not consistent.

Normalization solves this by separating data into related tables.

So, to fix the above issues, what we can do is first create a separate table for the users -

    user_id    name      email
    1          Arvind    arvind@example.com
    2          Rahul     rahul@example.com

Now, we create a separate table for the products -

    product_id    name         price
    101           Phone        50000
    102           Headphones   3000
    103           Laptop       90000

And finally, we update the orders table like this -

    order_id    user_id    product_id
    1           1          101
    2           1          102
    3           2          103

So, now the user email is stored only once in the 'users' table. The orders table just stores the 'user_id'. This reduces duplication and improves consistency.

# BENEFITS OF NORMALIZATION

Normalization has various benefits. Some of them are -

    1. Reduces duplicate data
    2. Improves data consistency
    3. Saves storage
    4. Makes updates safer
    5. Reduces data anomalies
    6. Makes relationships clear
    7. Works well with foreign keys and constraints

For example, if a user updates the email, we just have to update one row in one table. All the orders still point to the same user through the 'user_id'.

# DATA ANOMALIES

Normalization helps prevent data anomalies. Some common anomalies are -

    1. Update anomaly
    2. Insert anomaly
    3. Delete anomaly

## 1. UPDATE ANOMALY

An update anomaly happens when the same data is stored in many places and not all the copies are updated. For example -

    Arvind's email appears in 100 orders.

If we update only 99 rows, then one row still has the old email. Now, the database has conflicting data.

## 2. INSERT ANOMALY

An insert anomaly happens when we cannot insert one type of data without unrelated data.

For example, let's say product data is stored only inside orders.

So, we cannot add a new product until someone places an order for it. This is a bad design. Products should exist independently.

## 3. DELETE ANOMALY

A delete anomaly happens when deleting one record accidentally removes important data.

For example, if product details are stored only in orders and we delete the last order for a product, we may lose the product information.

Normalization avoids this by storing products separately.

# NORMAL FORMS

Normalization is often explained using normal forms. 

## 1. FIRST NORMAL FORM

It means each column should contain atomic values. 'Atomic' here means one value, not a list.

For example, this is bad -

    user_id    phone_numbers
    1          99999, 88888

A better approach is -

    user_id    phone_number
    1          99999
    1          88888

This makes data easier to query and update.

Also, each column in a table must contain value sof the same data type and each row should be unique.

## 2. SECOND NORMAL FORM

It means non-key columns should depend on the full primary key. This mainly matters when a table has a composite primary key.

For example, if we have -

    order_items table:

        order_id
        product_id
        product_name
        quantity

If the primary key is -

    (order_id, product_id)

Then. the 'quantity' depends on both order and product. But, product_name depends only on the 'product_id'. So, in this case, the 'product_name' should belong in a separate 'products' table.

So, the rule is that the table must already be in First Normal Form and every non-key column must depend entirely on the entire primary key, not just a part of it.

## 3. THIRD NORMAL FORM

It means non-key columns should not depend on other non-key columns. For example -

    users table:

        user_id
        city_id
        city_name

Here, the 'city_name' depends on the 'city_id', not on the 'user_id'. A better option is -

    users table:
        user_id
        city_id

    cities table:
        city_id
        city_name

This avoid duplicate city names across users.

So, the rule is the table must already be in second Normal Form and non-key columns must depend directly on the primary key and not on other non-key columns.

# EXAMPLE

Let's say we have a table -

    order_id    user_name    user_email          product_name    product_price
    1           Arvind       arvind@example.com  Phone           50000
    2           Arvind       arvind@example.com  Laptop          90000

This table mixes users, orders and products.

A normalized design would be -

    users:
        user_id
        name
        email

    products:
        product_id
        name
        price

    orders:
        order_id
        user_id

    order_items:
        order_id
        product_id
        quantity
        price_at_purchase

Now, each table has one responsibility and this makes the database cleaner.

# DOWNSIDES OF NORMALIZATION

Normalization is good for correctness but it can make reads for expensive. For example, if we want to show order details, we may need to join a lot of tables -

    orders
    users
    order_items
    products
    payments
    addresses

It is fine for small systems but for high-traffic systems, repeated joins can become expensive.

# WHEN TO NORMALIZE

You should normalize when -

    1. Data consistency is important
    2. Data changes frequently
    3. You want clean schema design
    4. Relationships matter
    5. You want database constraints
    6. The system is still evolving
    7. Writes should be simple and safe

For most relational databases, normalization is a good starting point.

Consistency Models define what kind of data correctness guarantee a system gives when data is copied, replicated, caches, or distributed across multiple places.

They answer questions like -

    1. After I write data, will I immediately read the latest value?
    2. Can different users see different values at the same time?
    3. Can replicas return old data?
    4. Will all replicas eventually agree?
    5. What ordering of updates will clients observe?

# WHY DO WE NEED CONSISTENCY MODELS?

In a single database server, consistency is easier.

There is one main copy of the data.

But, in distributed systems, data may exist in many places. For example -

    Primary Database
        -> Replica 1
        -> Replica 2

OR -

    Region India
    Region USA
    Region Europe

OR

    Cache + Database

Now, suppose a user updates their profile -

    name = Arvind

The write happens only on the primary database. But, replicas or caches may still have the old value for a short time.

So, if someone reads immediately, what should they see?

Different systems make different promises. Those promises are what we call 'Consistency Models.'

# SIMPLE EXAMPLES

Let's say the current value is -

    username = arvind_old

User updates it to -

    username = arvind_new

Now, immediately after the write, different systems may behave differently.

Strong consistency may guarantee -

    Any read after the write returns arvind_name

Eventual consistency may allow -

    Some reads return arvind_old for a while
    Later all reads return arvind_new

Read-after-write consistency may guarantee -

    The user who updated it sees arvind_new
    Other users may temporarily see arvind_old

So, consistency models define what kind of behavior is allowed.

# STRONG CONSISTENCY

Strong Consistency means once a write is successful, all the future reads return the latest value.

For example -

    Write:
        balance = Rs. 5000

After the write succeeds -

    Read from any replica:
        balance = Rs. 5000

So, the system behaves like there is only one copy of the data. This is easiest for developers to reason about.

Strong consistency is important in -

    1. Banking
    2. Payments
    3. Inventory booking
    4. Ticket booking
    5. Account balances
    6. Critical configuration

The tradeoff is that strong consistency can reduce availability or increase latency in distributed systems. The system may need co-ordination between the replicas before confirming reads or writes.

# EVENTUAL CONSISTENCY

It means replicas may temporarily disagree, but if no new updates happen, all the replicas will eventually converge to the same value.

For example -

    Write:
        profile_photo = new_photo.jpg

Immediately after write -

    Replica 1:
        new_photo.jpg

    Replica 2:
        old_photo.jpg

After some time -

    Replica 1:
        new_photo.jpg

    Replica 2:
        new_photo.jpg

So, stale reads are possible but only for a while.

Eventual consistency is useful when high availability and low latency matter more than immediate correctness.

Some good use cases include -

    1. Social media likes
    2. View counts
    3. User activity feeds
    4. Product recommendations
    5. Analytics counters
    6. Search indexes

For example, if a like count is slightly behind for a few seconds, that is usually acceptable.

# READ-AFTER-WRITE CONSISTENCY

It means after a user writes data, that same user can immediately read their own update.

For example -

    User updates profile name to Arvind

Then the same user refreshes their profile and sees the name as 'Arvind,' not the old name.

This is important for user experience.

Without read-after-write consistency, users may think their update failed.

To implement read-after-write, these things can be done -

    1. Reading from primary after writes
    2. Sticky sessions
    3. Routing user reads to an up-to-date replica
    4. Waiting until replica catches up
    5. Updating cache immediately after write

This model is weaker than strong consistency because other users may still see old data temporarily.

But it gives a good user experience to the user who made the change.

# MONOTONIC READS

Monotonic reads mean once a user sees a new value, they should not later see an older value. For example -

    First read:
        profile name = Arvind New

The second read should not return -

    profile name = Arvind Old

That would feel like time went backward.

This can happen if requests go to a different replica.

For example -

    Request 1 -> Replica A, which is up to date
    Request 2 -> Replica B, which is behind

Monotonic reads prevent this. They can be implemented by routing user's reads consistently to replicas that are at least as fresh as what the user has already seen.

# MONOTONIC WRITES

Monotonic Writes mean the writes from the same client are applied in the order they were issued. For example, a user updated the profile twice -

    Write 1:
        name = Arvind

    Write 2:
        name = Arvind Rana

The system should not apply Write 2 and then Write 1 later. Monotonic writes are important when the order matters. For example - Profile Updates, Document edits, Status updates, Sequential user actions, etc.

# CASUAL CONSISTENCY

Casual Consistency means operations that are casually relation must be seen in the correct order. If one event depends on the other, everyone should observe them in that order.

For example -

    User A posts:
        "I got a new job"

    User B comments:
        "Congratulations"

The comment depends on the post. So, the other users should not see the comment before seeing the post.

Causal consistency is stronger than eventual consistency but weaker than strong consistency.

It is useful in social networks, messaging systems, collaborative apps, and comment systems.

# SEQUENTIAL CONSISTENCY

It means all the operations appear in some single global order, and each client's operations appear in the order they were issued. It does not necessarily mean the order matches real time.

For example, if Client A writes -

    A1 then A2

everyone must see A1 before A2.

If a client B writes -

    B1 then B2

everyone must see B1 before B2.

The system may choose a global order like -

    A1, B1, A2, B2

OR

    A1, A2, B1, B2

OR

    B1, A1, B2, A2

as long as each client's order is preserved.

Sequential consistency is easier than linearizability but still gives a single agreed order.

# LINEARIZABILITY

It is one of the strongest consistency models.

It means every operation appears to happen instantly at some point between start and finish, and the order respects real time.

In simple words -

    Once a write completes,
    every later read must see that write.

This is what people often mean when they say 'Strong Consistency'.

For example -

    At 10:00:00, write balance = Rs. 5000 completes.

Any read after 10:00:00 must return -

    Rs. 5000

It is very useful for correctness-critical systems. Some good use cases include -

    1. Locks
    2. Leader election
    3. Account balances
    4. Inventory counts
    5. Distributed coordination
    6. Unique username creation

The tradeoff is co-ordination cost. In distributed systems, it can increase latency and reduce availability during network partitions.

# BOUNDED STALENESS

It means reads may be stable, but only up to a known limit.

For example -

    Reads may be at most 5 seconds old.

This is between strong and eventual consistency.

It gives a predictable freshness guarantee.

For example -

    Analytics dashboard can show data up to 1 minute old

This may be acceptable.

Bounded Staleness is useful when exact real-time data is not required, but unlimited staleness is also not acceptable.

# CONSISTENCY IN REPLICATION

Consistency models are closely related to replication.

Suppose, we have -

    Primary DB
        -> Replica 1
        -> Replica 2

If replication is synchronous, consistency is stronger. The primary waits for replicas before confirming the writes.

If replication is asynchronous, consistency is weaker. The primary confirms the writes before replicas catch up.

So, replication strategy affects consistency.

# CONSISTENCY IN CACHING

Caching also creates consistency problems. Let's say a database has -

    Product price = Rs. 1000

Cache also stores -

    Product price = Rs. 1000

Now, price changes in the database -

    Product Price = Rs. 1200

But cache still has the old value.

    Produce Price = Rs. 1000

This is stale data. Some cache consistency strategies include -

    1. TTL expiration
    2. Cache invalidation on write
    3. Write-through cache
    4. Write-behind cache
    5. Cache-aside pattern

Caching often trades consistency for performance.

# CONSISTENCY VS AVAILABILITY

In distributed systems, stronger consistency usually requires more co-ordination. And more coordination can reduce availability or increase latency.

For example, if replicas must agree before a write succeeds, then a network issue can block writes. So, there is a tradeoff.

    Stronger consistency:
        More correctness
        More coordination
        Higher latency
        Lower availability during failures

    Weaker consistency:
        Better availability
        Lower latency
        Possible stale or out-of-order data

This connects to the CAP theorem. During a network partition, a distributed system often has to choose between consistency and availability.

# CHOOSING THE RIGHT CONSISTENCY MODEL

Not all the data needs the same consistency. For example -

    Bank balance:
        Strong consistency

    Inventory for last item:
        Strong consistency

    Profile update for same user:
        Read-after-write consistency

    Social media like count:
        Eventual consistency

    Analytics dashboard:
        Bounded staleness

    Chat messages:
        Consistent ordering / causal consistency

    Product search index:
        Eventual consistency

The right consistency model depends on the business requirement.

Do not use strong consistency everywhere unless needed, because it can be expensive.

Also do not use eventual consistency for critical data where stale values can cause serious bugs.

# EXAMPLE - AN E-COMMERCE APP

Let's say we are building an e-commerce app. Different parts may use different consistency models.

## 1. PAYMENT STATUS

Payment status should be strongly consistent. We do not want confusion about whether the payment succeeded.

## 2. INVENTORY

Inventory for the stock should be strongly consistent since the stock is limited. If only one item is left, two users should not both buy the same.

## 3. PRODUCT REVIEWS

Reviews can be eventually consistent. If a new review appears after a few seconds, that is usually okay.

## 4. SEARCH INDEX

Product search can be eventually consistent. If a newly added product appears in search results after a short delay, it is usually acceptable.

## 5. USER PROFILE

The user should get read-after-write consistency. If they update their address, they should immediately see the updates address. Other systems may catch up shortly after.

Connection Pooling means reusing a fixed set of database connections instead of creating a new connection for every request.

Opening a database connection is expensive.

So, instead of doing this each time -

    Open new DB connection
    Run query
    Close DB connection

We create a pool of reusable connections -

    Connection Pool:
        Connection 1
        Connection 2
        Connection 3
        ...
        Connection N

Now, when a request needs the database, it borrows a connection from the pool. After the query finishes, it returns the connection back to the pool.

# WHY DO WE NEED CONNECTION POOLING?

Databases cannot handle unlimited connections. Each database connection uses resources like Memory, CPU, Network Socket, Authentication/Session state, Transaction state, etc.

If every application request opens a new database connection, the database can get overloaded. For example suppose we have 1000 incoming requests.

Without connection pooling -

    1000 requests -> 1000 database connections

This can be too much for the database.

With pooling -

    1000 requests -> reuse 50 database connections

The pool controls how many database connections are active. This protects the database.

Suppose, our application has a connection pool of 10 connections -

    Pool Size = 10

Now, 5 requests come in -

    Request 1 -> Connection 1
    Request 2 -> Connection 2
    Request 3 -> Connection 3
    Request 4 -> Connection 4
    Request 5 -> Connection 5

After each request finishes, the connection is returned to the pool.

Now, if 20 requests come in at the same time -

    First 10 requests:
        Get connections

    Next 10 requests:
        Wait until a connection is free

So, connection pooling also acts like a pressure control mechanism.

If prevents the application from overwhelming the database with too many concurrent queries.

So, the flow is -

    1. Application starts
    2. Pool opens some database connections
    3. Request needs database
    4. Request borrows a connection
    5. Request runs query
    6. Request returns connection
    7. Another request reuses same connection

# HOW CONNECTION POOLING WORKS?

A connection pool usually manages -

    1. Minimum pool size
    2. Maximum pool size
    3. Idle connections
    4. Connection timeout
    5. Max connection lifetime
    6. Health checks
    7. Waiting queue

For example -

    Minimum pool size:
        5 connections

    Maximum pool size:
        50 connections

When traffic is low, the pool may keep 5 connections open. When it increases, it may grow up to 50. If all 50 are busy, new requests wait of fail after a timeout.

# POOL SIZE

If the pool is too small, requests wait too much. This increases the latency.

    App has many requests
    Only 5 DB connections
    Requests queue up

But, if the pool is too large, the database ges overloaded -

    500 app connections
    Database CPU spikes
    Queries slow down
    Memory usage increases

So, the pool size must be chosen carefully. A bigger pool does not always mean better performance. After a point, too many concurrent queries make everything slower.

Suppose we have 10 servers. Each server has a pool size of 100. So, total possible database connections are 1000.

If the database can comfortably handle only 300 connections then this is a problem. Each though each app server has a reasonable-looking pool size, total connections are too high.

So, we should calculate pool size across all application instances.

In microservices, connection count can grow quickly. Suppose, we have -

    20 instances of Order Service
    20 instances of Payment Service
    20 instances of User Service

If each instance has a pool of 50, then there are 3000 possible DB connections. That may overload the database.

This is why microservices need careful connection management. Sometimes, teams use an external connection pooler. For example, PgBouncer for PostgreSQL.

# APPLICATION-LEVEL POOLING VS EXTERNAL POOLING

## 1. APPLICATION-LEVEL POOLING

Most frameworks and database clients have built-in pooling. For example -

    Java HikariCP
    Node.js pg pool
    Python SQLAlchemy Pool
    Go database/sql pool

So, the app directly manages the connections -

    App -> Database

## 2. EXTERNAL POOLING

An external pooler sits between the app and the database.

    App -> Pooler -> Database

For example -

    PgBouncer
    ProxySQL
    RDS Proxy

External pooling is useful when many app instances create too many database connections.

# CONNECTION LEAK

A connection leak happens when an application borrows a connection but does not return it to the pool. For example -

    Request borrows connection
    Query fails
    Code forgets to release connection

Now, that connection stays busy forever. Over time, more connections leak. Eventually, the pool runs out. So, the app becomes slow or fails.

Good frameworks usually help release connections automatically, but bugs can still happen.

Using 'finally' blocks or proper context managers helps prevent leaks.

# CONNECTION TIMEOUT

If no connections are available, the request fails. This prevents the request from waiting forever. A timeout is important because if the database is overloaded, we want failure to happen clearly instead of letting requests pile up endlessly.

# IDLE CONNECTIONS

Idle connections are open connections that are not currently being used. Keeping some idle connections are useful because they are ready for future requests. But, too many idle connections waste the database resources. A pool usually closes idle connections after some time.

For example -

    If connection is idle for 10 minutes:
        close it

This balances readiness and resource usage.

# MAX CONNECTION LIFETIME

It means how long a connection can stay open before it is replaced. For example -

    Replace connection after 30 minutes

This is useful because long-lived connections can sometimes become stale or problematic due to -

    1. Network changes
    2. Database failover
    3. Load balancer timeouts
    4. Server-side connection limits

By recycling connections, the pool avoids keeping bad connections forever.

# TRANSACTIONS AND CONNECTIONS

A transaction usually holds a connection until it completes. For example -

    Begin transaction
        Query 1
        Query 2
        Query 3
    Commit

During this whole time, the connection cannot be used by another request. So, long transactions can reduce the available pool capacity. This is why transactions should be kept short.

Avoid doing slow external calls inside transactions.

For example, this is bad -

    Begin transaction
        Update order
        Call payment provider API
        Send email
    Commit

The connection is held while waiting for the external services. This can exhaust the pool.

# CONNECTION POOLING AND DATABASE LIMITS

The database has a maximum connection limit. For example -

    PostgreSQL max_connections = 500

But, we should not use all 500 for application traffic since some connections are needed for things like Admin Access, Monitoring, Replication, Background jobs, and Maintenance.

So, application pool sizes should leave room. For example -

    Database max connections = 500
    Reserve 50 for admin/maintenance
    Available for app = 450

So, if we have 10 app instances -

    450 / 10 = 45 connections per instance

This is a rough calculation.

# CONNECTION POOLING AND BACKPRESSURE

Connection pooling provides backpressure. It means slowing down incoming work when a downstream system is overloaded.

If the database can only handle 50 concurrent queries, the pool prevents 500 queries from hitting it once. So, in this case 50 queries run and the extra requests either wait of fail fast.

Without backpressure, too many queries can make the database slower for everyone.

# READ REPLICAS AND POOLS

If we use read replicas, we may have separate pools. For example -

    Write pool -> Primary DB
    Read pool  -> Replica DB

This helps route the traffic properly.

But we must remember that replicas may be stale. For read-after-write consistency, some reads should go to primary.

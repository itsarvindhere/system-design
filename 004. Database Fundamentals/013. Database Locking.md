Locking is a mechanism databases use to control concurrent access to data so multiple transactions do not corrupt each other.

When many users or services read and write the same data at the same time, conflicts can happen. For example -

    User A tries to book the last seat
    User B tries to book the same last seat

If both update at the same time without control, the same seat may get booked twice.

Locking helps the database co-ordinate these operations safely.

# WHY DO WE NEED LOCKING?

Database are shared by many transactions at the same time. For example -

    1. Update wallet balance
    2. Book a ticket
    3. Reserve inventory
    4. Change order status
    5. Increment like count
    6. Transfer money

If two transactions try to modify the same row at once, the final result can become wrong. For example -

    Balance = Rs. 1000

Transaction 1 adds Rs. 100.

Transaction 2 adds Rs. 200.

If both read the old value and write back independently, one update may overwrite the other.

Locking helps prevent this.

# WHAT IS A LOCK

A lock is a temporary control places on a row, a page, a table, or resource so other transactions cannot interfere in an unsafe way.

For example -

    Transaction 1 locks Account A

While Transaction 1 is working, Transaction 2 may need to wait before touching Account A. This ensures operations happen in a controlled way.

Locks protect data from unsafe concurrent access. They are used to prevent problems like Lost Updates, Dirty Writes, Double booking, Race conditions etc.

# COMMON LOCK TYPES

## 1. SHARED LOCK

A shared lock is usually used for reading.

Multiple transactions can hold shared locks on the same data at the same time. For example -

    Transaction 1 reads Product A
    Transaction 2 also reads Product A

Both may read safely.

Shared locks are useful when multiple readers are okay. But, if someone needs to write, they may need to wait.

## 2. EXCLUSIVE LOCK

An exclusive lock is for writing.

If a transaction holds an exclusive lock, other transactions usually cannot read or write that data depending on the database and the isolation level.

They are used when data is being changed.

## 3. ROW-LEVEL LOCK

A row-level lock locks only a specific row. For example -

    Lock only order_123

Other rows can still be accessed normally. This gives better performance because unrelated rows are not blocked. Row-level locking is common and useful in high-traffic systems.

## 4. TABLE-LEVEL LOCK

A table-level lock locks the whole table. For example -

    Lock all rows in orders table

This is simpler but much more restrictive. If one transaction holds the table lock, other may have to wait even if they want different rows.

Table-level locks reduce concurrency, so they are usually less preferred for busy systems.

# LOCKING AND TRANSACTIONS

Locking is often tied to transactions. For example -

    Begin transaction
        Read seat availability
        Lock seat row
        Update seat to booked
    Commit

While the transaction is open, the lock may be held. After commit or rollback, the lock is released. That is why long transactions can be dangerous. They can hold locks for too long and block other users.

# LOCK CONTENTION

Lock contention happens when many transactions want the same lock. For example -

    100 users trying to update the same counter

Only one of few can proceed, and the rest wait. This slows down the system. High lock contention often happens in Popular product inventory, Ticket booking, Wallet updates, Global counters, Hot rows in social apps etc.

If contention is high, the database may become slow even if CPU is fine.

# DEADLOCK

A deadlock happens when two transactions wait for each other forever. For example -

    Transaction 1 locks Row A
    Transaction 2 locks Row B

Then -

    Transaction 1 waits for Row B
    Transaction 2 waits for Row A

Now, both are stuck.

The database usually detects this and aborts one transaction.

That transaction can be retried.

Deadlocks are more likely when transactions lock multiple rows in different orders.

Some common ways to reduce deadlocks include -

    1. Lock rows in a consistent order
    2. Keep transactions short
    3. Avoid doing slow work inside transactions
    4. Lock only what is necessary
    5. Retry failed transactions safely

# OPTIMISTIC AND PESSIMISTIC LOCKING

## 1. OPTIMISTIC LOCKING

Optimistic locking assumes conflicts are rare. It does not lock the data immediately when reading.

Instead, it checks whether the data changed before saving.

The idea is -

    Read freely
    Update only if nobody else changed it

This works well when the conflicts are uncommon.

Suppose, we have a product row -

    product_id = 101
    stock = 10
    version = 5

User A reads the row -

    stock = 10
    version = 5

User B also reads the same row -

    stock = 10
    version = 5

Now, User A updates stock to 9.

The database checks the version -

    Update only if version = 5

Now, the row becomes -

    stock = 9
    version = 6

User B now tries to update based on the old version 5. The database checks -

    version = 5

But the current version is already 6. So, the update fails. User B must re-read the row and try again.

So, Optimistic Locking prevents lost updates without locking the row for the entire read period. It is useful when users read data but only a few actually update it.

For example - User Profile editing, Product metadata, Draft documents, Low-conflict business data, Content Management System, etc.

A common method if using a 'version' column. For example -

    users
        id
        name
        email
        version

Update query may look like this conceptually -

    UPDATE users
    SET name = 'Arvind Rana',
        version = version + 1
    WHERE id = 123
    AND version = 5;

If another transaction already updated the row, the version will no longer be 5 an the update will affect zero rows. This tells us there was a conflict.

Some systems also use timestamps of ETags instead of version numbers.

The benefits of Optimistic Locking is -

    1. No long-held locks
    2. Better concurrency for read-heavy workloads
    3. Lower blocking
    4. Good when conflicts are rare
    5. Simpler for users reading data first and updating later

The downsides are -

    1. Conflicts are detected only at write time
    2. Updates may fail and need retry
    3. Not ideal when conflicts are frequent
    4. User may need to re-read data and resubmit

If the same row is updated constantly, optimistic locking can cause many retries.

## 2. PESSIMISTIC LOCKING

It assumes conflicts are likely. It locks the data early, usually when it is read for update. The idea is -

    If I need to change this row, lock it now,
    so nobody else can change it while I work

This is useful when concurrent updates are common and correctness is critical.

Suppose, we are booking the last seat. Transaction 1 reads the seat and locks it -

    Seat A10 locked by Transaction 1

Transaction 2 tries to read the same seat for update. It must wait.

    Transaction 2 waits

Transaction 1 finishes booking and commits. Then Transaction 2 can continue and see the updated state. This prevents double booking.

A database may lock rows with a query conceptually like -

    SELECT * FROM seats
    WHERE seat_id = 'A10'
    FOR UPDATE;

This means the selected row is now locked for updating. Other transactions trying to update the same row may block until the lock is released.

So, the benefits are -

    1. Prevents conflicts up front
    2. Good for high-contention data
    3. Simpler correctness for critical flows
    4. Reduces retries in hot write scenarios
    5. Useful for booking and reservation systems

There are also some downsides -

    1. Lower concurrency
    2. Requests may block and wait
    3. Long locks can slow the system
    4. Deadlock risk
    5. Can hurt throughput under high load

So, an easy way to remember is -

    Optimistic:
        “Let’s proceed and verify later.”

    Pessimistic:
        “Let’s lock first and avoid surprises.”

Use Optimistic locking when -

    1. Conflicts are rare
    2. Reads are much more common than writes
    3. You want better concurrency
    4. Re-reading and retrying is acceptable
    5. Data is not highly contended

For example - User profile updates, document editing, product metadata, configuration management, low-contention business records.

Use Pessimistic locking when -

    1. Conflicts are likely
    2. Correctness is critical
    3. Double booking must be prevented
    4. The row is highly contested
    5. Retry overhead would be expensive or confusing

For example - Ticket booking, Inventory reservation, Wallet transfers, Seat selection, 5. Locking a shared resource before update etc.

# LOCK GRANULARITY

It means the size of the object being locked. For example - Row, Page, Table, or database.

Smaller locks give better concurrency. Larger locks are simpler but block more work.

Usually, row-level locking is preferred when possible.

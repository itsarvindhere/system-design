Data replication means keeping copies of the same data on multiple database servers. Instead of having only one database server, we maintain replicas.

Replication helps improve availability, reliability, and read scalability.

# WHY DO WE NEED REPLICATION?

A single database server can fail due to reasons like disk failure, network issue, data center outage, machine crash, maintenance downtime etc.

If we only have one database and it goes down, the whole application may stop working.

Replication helps because another database copy can take over.

Replication also helps with read scaling.

Suppose one database can handle 10,000 reads per second. If our application needs 50,000 reads per second, we can add replicas and distribute read traffic -

    Writes -> Primary
    Reads  -> Replicas

Suppose, we have an e-commerce app. The main database stores Users, Products, Orders, and Payments.

If the main database fails, the users may not be able to place orders or view data. So, we create replicas -

    Primary DB:
        Handles writes

    Replica DB 1:
        Handles reads

    Replica DB 2:
        Standby for failover

Now, if the primary database fails, one replica can be promoted to become the new primary. This improves availability.

# PRIMARY-REPLICA REPLICATION

This is the most common replication model. It is also called a leader-follower or a master-slave replication.

In this model -

    1. Primary handles writes
    2. Replicas copy data from primary
    3. Reads can be served from replicas

The flow looks like this -

    Client -> Primary:
        Write order

    Primary -> Replicas:
        Replicate order data

    Client -> Replica:
        Read order details

This is simple and common.

But there is one important points -

    Replicas may lag behind the primary.

# REPLICATION LAG

It means replicas are behind the primary. For example -

    Primary:
        order_123 status = Paid

    Replica:
        order_123 status = Pending

This can happen because replication takes time. Usually the delay is small, maybe milliseconds or seconds. But, under heavy load, it can become larger.

Replication lag can cause confusing behavior. For example -

    1. User places order
    2. App writes order to primary
    3. App reads from replica immediately
    4. Replica has not received the update yet
    5. User does not see the new order

This is called a Stale Read.

# STALE READ 

A stale read happens when a client reads old data from a replica. For example -

    User updates profile name to Arvind

Write goes to primary -

    Primary:
        name = Arvind

But, read goes to replica before replication completes -

    Replica:
        name = Old Name

So, the user may see the old data for a short time. Stale reads are common in asynchronous replication systems.

Whether this is acceptable depends on the use case.

    Product reviews:
        Slight delay may be okay

    Bank balance:
        Stale read may be unacceptable

# SYNCHRONOUS REPLICATION

In this, the primary waits for the replicas to confirm the write before telling the client it succeeded. So, the flow is -

    1. Client sends write to primary
    2. Primary writes data
    3. Primary sends data to replica
    4. Replica confirms write
    5. Primary returns success to client

This gives stronger consistency. If the client receives success, at least one replica also has the data.

But, it is slower. The write latency increases because the primary must wait for replicas. If the replica is slow or unavailable, writes may become slow or fail. So -

    Synchronous replication:
        Stronger consistency
        Higher write latency
        Lower availability if replicas are unhealthy

# ASYNCHRONOUS REPLICATION

In this, the primary returns success immediately after writing locally. Then, it sends data to the replicas in the background.

So, the flow is -

    1. Client sends write to primary
    2. Primary writes data
    3. Primary returns success to client
    4. Primary later replicates data to replicas

This is faster for writes but replicas may lag. Also, if the primary crashes before replication completes, some committed data may be lost from replicas. So -

    Asynchronous replication:
        Faster writes
        Better availability
        Possible replication lag
        Possible data loss during failover

This is very common in real systems.

# SEMI-SYNCHRONOUS REPLICATION

This is a middle-ground.

The primary waits for at least one replica to acknowledge receiving the data, but not necessarily all the replicas.

This improves durability compared to fully asynchronous replication.

But it is usually faster than waiting for all replicas.

# READ REPLICAS

They are replicas used to handle read traffic. For example -

    Primary:
        Handles writes

    Replica 1:
        Handles product reads

    Replica 2:
        Handles order history reads

    Replica 3:
        Handles reporting reads

This improves the read scalability. If reads are much more common than writes, then read replicas are very useful.

# READ-AFTER-WRITE CONSISTENCY

Read-after-write consistency means after a user writes data, they should immediately see their own update. For example -

    User updates profile photo
    User refreshes profile page
    User should see new photo

Replication lag can break this if the read goes to a replica. So, solutions are -

    1. Read from primary after write
    2. Use sticky reads for the same user
    3. Track replication lag
    4. Use synchronous replication for critical data
    5. Wait until replica catches up

A common simple solution is -

    After a user writes something, read their next few requests from primary.

This avoids confusing user experience.

# FAILOVER

Failover means switching to another database server when the primary fails.

For example -

    Primary DB fails
    Replica 1 is promoted to primary
    Application starts writing to Replica 1
    Other replicas are reconfigured to follow the new primary

It improves availability but failover must be handled carefully. If replication is asynchronous, the promoted replica may not have the latest writes.

# AUTOMATIC VS MANUAL FAILOVER

Automatic failover means the system automatically detects failure and promotes a replica. This reduces downtime but it must be carefully designed to avoid mistakes.

For example, if the network is temporarily partitioned, the system may incorrectly think the primary is dead.

Manual failover means humans decide when to promote a replica. This is safer in complex situations but slower. Manual failover may be acceptable for less critical systems. For high availability systems, automatic failover is usually preferred.

# SPLIT-BRAIN PROBLEM

Split-brain happens when two database nodes both think they are the primary. For example -

    Primary A is still running
    Replica B is promoted to primary

Now, both access writes.

This can create conflicting data.

    User writes order to A
    Another write goes to B

Later merging this data becomes difficult.

Split-brain is a serious problem in replicated systems. To prevent it, systems use mechanisms like -

    1. Leader election
    2. Quorum
    3. Fencing
    4. Consensus protocols
    5. Proper failover coordination

# MULTI-PRIMARY REPLICATION

Multiple database nodes can accept writes. For example -

    Primary 1 accepts writes
    Primary 2 accepts writes
    Primary 3 accepts writes

Each primary replicates changes to the others. This can improve write availability and reduce latency in different regions.

For example -

    Indian users write to India primary
    US users write to US primary

But multi-primary replication is complex.

The biggest challenge is conflict resolution.

Suppose two primaries update the same user at the same time -

    Primary India:
        user name = Arvind

    Primary US:
        user name = Rahul

Now, both changes replicate. Which value should win? Some conflict resolving strategies are -

    1. Last write wins
    2. Version numbers
    3. Timestamps
    4. Application-level merge logic
    5. Reject conflicting writes

# REPLICATION TOPOLOGIES

There are different ways to arrange replication.

## 1. PRIMARY-REPLICA

One primary accepts writes and replicas copy from the primary. This is simple and common.

## 2. MULTI-PRIMARY

Multiple nodes accept writes. This improves write availability but creates conflict problems.

## 3. CHAIN REPLICATION

Data is replicated in a chain -
    
    Node A -> Node B -> Node C

This can reduce load on the primary, but increases replication delay across the chain.

## 4. LEADERLESS REPLICATION

There is no single leader. Clients can write to multiple replicas. Systems like Cassandra and Dynamo-style databases use leaderless ideas. They often use quorum reads and writes.

# QUORUM REPLICATION

In Quorum-based systems, reads and writes need agreement from a certain number of replicas.

Let's say we have 3 replicas. We may configure -

    Write quorum W = 2
    Read quorum R = 2

This means -

    Write must be accepted by 2 replicas
    Read must check 2 replicas

So, if -

    R + W > N

    where N = number of replicas

Then reads are more likely to see the latest writes. For example -

    R = 2
    W = 2
    N = 3

    R + W = 4 > 3

Quorum systems balance consistency, availability, and latency.

# REPLICATION VS SHARDING

They are different. Replication copes the same data to multiple servers while Sharding splits different data across multiple servers.

Replication helps with availability and read scaling.

Sharding helps with data size and write scaling.

In production, they are often used together.

    Shard 1:
        Primary + replicas

    Shard 2:
        Primary + replicas

    Shard 3:
        Primary + replicas

# REPLICATION AND BACKUPS

They are not the same.

Replication copies changes quickly. If someone accidentally deletes data, that deletion may also replicate. So replication does not protect against all human errors or data corruption.

Backups are separate historical copies that can be restored.

# REPLICATION LOG

Many database use a replication log. The primary records changes in a log and the replicas read from this log to apply the same changes.

In MySQL, this is related to the binary log.

In PostgreSQL, this is related to WAL, which means Write-Ahead Log.

The exact mechanism depends on the database.

# BENEFITS AND DOWNSIDES OF REPLICATION

Data replication has several benefits -

    1. High availability
    2. Failover support
    3. Read scaling
    4. Disaster recovery
    5. Lower read latency in multiple regions
    6. Maintenance with less downtime
    7. Better durability

There are also some downsides -

    1. Replication lag
    2. Stale reads
    3. Failover complexity
    4. Split-brain risk
    5. More infrastructure cost
    6. More monitoring complexity
    7. Possible data loss with async replication
    8. Conflict resolution in multi-primary setups

Sharding means splitting a large database into smaller parts and storing those parts across multiple database servers.

Each smaller part is called a 'Shard.'

Instead of storing all the data on one database server, we divide the data.

For example -

    Users with ID 1 - 1,000,000       -> Shard 1
    Users with ID 1,000,001 - 2,000,000 -> Shard 2
    Users with ID 2,000,001 - 3,000,000 -> Shard 3

So, each shard stores only a portion of the total data.

The main idea is that one big database becomes multiple smaller databases.

# WHY DO WE NEED SHARDING?

A single database server has limits. It has limited CPU, memory, disk space, IOPS, Network bandwidth, and write capacity.

As the application grows, one database may become too large or too busy. For example -

    1 billion users
    10 billion orders
    100 million writes per day

A single database server may not handle this well.

Vertical scaling can help for some time -

    Add more RAM
    Add more CPU
    Use faster storage

But, vertical scaling has an upper limit and becomes expensive.

Sharding helps to scale horizontally -

    Add more database servers
    Distribute data across them

Suppose, we have a 'users' table with 300 million users. Storing all users on one database server may become slow. So, we split users into 3 shards -

    Shard 1:
        users 1 to 100 million

    Shard 2:
        users 100 million to 200 million

    Shard 3:
        users 200 million to 300 million

Now, each shard only stores around 100 Million users. If we need user 150 million, we know it belongs to Shard 2.

This reduces the load on each database server.

# SHARDING vs PARTITIONING

Sharding and partitioning are related.

Partitioning means splitting data into smaller parts.

Sharding usually means those parts are distributed across different database servers.

For example -

    Partitioning:
        One database server
        Table split into partitions

    Sharding:
        Multiple database servers
        Data split across shards

So, Partitioning can happen inside one database and Sharding spreads data across multiple databases or machines.

In practice, people sometimes use these terms loosely.

But in system design, sharding usually means horizontal data distribution across servers.

The primary goal in partitioning is query optimization and easier maintenance. For example, splitting a massive 'orders' table into 12 monthly partitions on the same disk.

Sharding's primary goal is horizontal scaling and handling massive read/write traffic. For example, users from Asia are routed to Database Node A, while users from Europe are routed to Database Node B.

# SHARD KEY

A Shard Key is the field used to decide which shard a record belongs to.

For example -  

    user_id
    customer_id
    tenant_id
    order_id
    region

If we shard by the 'user_id', then the 'user_id' is the shard key.

For example -

    shard = user_id % number_of_shards

If we have 4 shards -

    user_id 101 -> 101 % 4 = 1 -> Shard 1
    user_id 102 -> 102 % 4 = 2 -> Shard 2
    user_id 103 -> 103 % 4 = 3 -> Shard 3
    user_id 104 -> 104 % 4 = 0 -> Shard 0

The shard key is one of the most important decisions in sharding. A bad shard key can create serious problems.

A good shard key should -

    1. Distribute data evenly
    2. Distribute traffic evenly
    3. Match common query patterns
    4. Avoid hot shards
    5. Be stable
    6. Have high cardinality

For example, 'user_id' is often a good shard key because there are many users and data can be spread across them.

A bad shard key means put too much data or traffic on one shard. For example, sharding by 'country' may be bad in some cases when most of the users are from one country. In this case, one shard becomes overloaded which is called a 'Hot Shard.'

# SHARDING STRATEGIES

There are several common sharding strategies -

    1. Range-based Sharding
    2. Hash-based Sharding
    3. Directory-based Sharding
    4. Geo-based Sharding

## 1. RANGE-BASED SHARDING

It splits the data based on the ranges of the shard key. For example -

    user_id 1 - 1,000,000       -> Shard 1
    user_id 1,000,001 - 2,000,000 -> Shard 2
    user_id 2,000,001 - 3,000,000 -> Shard 3

This is simple and easy to understand and good for range queries. For example -

    Get users with ID between 1,000,000 and 1,500,000

The system knows this range belongs to Shard 2. But, range-based sharding can create hot shards. For example, if new users always get increasing IDs, all the new writes may go to the latest shard.

So, Shard 3 becomes hot. So, range sharding is simple, but it can cause uneven load.

## 2. HASH-BASED SHARDING

It applied a hash function to the shard key. For example -

    shard = hash(shard_key) % number_of_shards

This distributes data more evenly.

For example -

    user_id 101 -> Shard 2
    user_id 102 -> Shard 0
    user_id 103 -> Shard 3

The benefit here is better distribution but the downside is that range queries become harder. For example -

    Get users with ID between 1000 and 2000

Those users may be spread across many shards. So, hash-sharding is good for even distribution, but not ideal for range queries.

Also, if number of shards are changed, then many keys may have to be moved. In this case, virtual shards or consistent hashing is used to reduce the movement.

## 3. DIRECTORY-BASED SHARDING

It uses a lookup table or mapping service to decide where data lives. For example -

    user_123 -> Shard 1
    user_456 -> Shard 3
    user_789 -> Shard 2

The application checks the directory -

    Where is user_456?
    Directory says Shard 3

Then, it queries Shard 3. This is flexible because we can move users between shards by simply updating the directory. But, it adds complexity.

The directory becomes an important component. If directory is slow or unavailable, routing requests becomes difficult.

## 4. GEO-BASED SHARDING

It stores the data based on geographic region. For example -

    Indian users -> India shard
    European users -> Europe shard
    US users -> US shard

This can reduce latency and help with data residency laws. For example, European user data may need to stay in Europe. But, geo-based sharding can cause imbalance.

If one region has more traffic than others, that shard can become overloaded. It can also make global queries harder.

# SHARDING ARCHITECTURE

A sharded system usually has three parts -

    1. Application
    2. Shard routing logic
    3. Database shards

The flow is -

    1. Request comes to application
    2. Application extracts shard key
    3. Routing logic decides which shard to query
    4. Application sends query to that shard
    5. Shard returns result

For example -

    GET /users/123

Application checks -

    user_id = 123
    shard = hash(123) % 4
    target = shard 2

Then -

    Application -> Shard 2:
        SELECT * FROM users WHERE user_id = 123;

# SHARD ROUTING

Shard routing means deciding which shard should handle a request. Routing can happen in different places -

    1. Application-level routing
    2. Proxy-level routing
    3. Database-managed routing

## 1. APPLICATION-LEVEL ROUTING

The application knows the sharding logic. For example -

    shard = user_id % 4

The application directly connects to the right shard. This is simple and fast. But, every service that accesses the database needs to understand sharding logic.

## 2. PROXY-LEVEL ROUTING

A database proxy sits between the application and the shards.

    Application -> DB Proxy -> Shards

The proxy decides where the queries should go.

This hides sharding complexity from the application but the proxy becomes an extra component that must be scaled and made highly available.

## 3. DATABASE-MANAGED ROUTING

Some databases manage sharding internally. For example -

    MongoDB sharded clusters
    Cassandra
    DynamoDB
    CockroachDB
    Vitess

The database system handles routing and distribution. This reduces application complexity but we still need to understand the shard keys and access patterns.

# CROSS-SHARD QUERIES

They are queries that need data from multiple shards. For example, let's say we sharded the orders by the 'user_id'.

This query is easy -

    Get orders for user_123

because all the orders for user_123 are on one shard.

But what about this query -

    Get all orders created today

This one is hard because today's orders may exist on every shard. The system may need to query all the shards and merge the results. This is more expensive and complex. So, sharding works the best when most queries include the shard key.

# CROSS-SHARD JOINS

Joins become harder after sharding. Suppose users and orders are on different shards -

    users table sharded by user_id
    orders table sharded by order_id

Now, joining users and orders may require data from multiple shards -

    SELECT * FROM users
    JOIN orders ON users.id = orders.user_id;

This may become expensive. To avoid this, we often design data so related records are stored together. For example -

    users and orders both sharded by user_id

Then, orders for a user are likely on the same shard as the user. This is called 'Co-Location.'

# CO-LOCATION

Co-location means storing related data on the same shard. For example -

    User 123
    Orders of User 123
    Cart of User 123
    Addresses of User 123

All stored on the same shard using 'user_id.'

This makes user-specific queries efficient. For example -

    Get user profile and orders

The application only needs one shard. Co-location is very useful in sharded systems. But, it requires choosing a shard key that matches the main access pattern.

# HOT SHARDS

A hot shard is a shard that receives much more traffic than others. For example -

    Shard 1 -> 10% traffic
    Shard 2 -> 10% traffic
    Shard 3 -> 80% traffic

Shard 3 becomes the bottleneck. This can happen due to -

    1. Bad shard key
    2. Celebrity users
    3. Popular tenants
    4. Time-based writes
    5. Uneven geography
    6. Sequential IDs with range sharding

For example, if we shard by created_at, all new writes may go to the latest time range shard. This creates a hot shard.

# RESHARDING

Resharding means changing the sharding layout. For example if we change the number of shards from 4 to 8 or change the shard key.

If we use simple modulo sharding -

    shard = user_id % 4

And later increase the count to 8 -

    shard = user_id % 8

Many users move to different shards. This can require massive data movement. To reduce this problem, systems may use consistent hashing or virtual shards.

In consistent hashing, instead of using simple modulo, keys and shards are placed on a hash ring. When a shard is added, only some keys move to the new shard. This is useful for distributed systems.

Virtual shards are logical shards that can be mapped to physical machines -

    1024 virtual shards
    4 physical database servers

Each physical server owns many virtual shards. Later, if we add more servers, we can move some virtual shards -

    Server 1 owns virtual shards 1-256
    Server 2 owns 257-512
    Server 3 owns 513-768
    Server 4 owns 769-1024

If we add Server 5, we move some virtual shards to Server 5. This makes rebalancing easier than moving arbitrary rows one by one.

# TRANSACTIONS IN SHARDED DATABASES

Transactions become harder with sharding. If a transaction only touches one shard, it is manageable like -

    Transfer money between two accounts on same shard

But, if a transaction touches multiple shards, it becomes more complex -

    Transfer money from user on Shard 1 to user on Shard 3

Now we need a distributed transaction or some other consistency strategy.

Some options are -

    1. Two-phase commit
    2. Saga pattern
    3. Eventual consistency
    4. Carefully chosen shard key to avoid cross-shard transactions

Cross-shard transactions are expensive and should be avoided when possible.

# GLOBAL UNIQUE IDS

In a sharded systems, generating unique IDs needs care. If every shard uses auto-increment IDs starting from 1, we may get duplicates like -

    Shard 1 creates order_id = 1
    Shard 2 creates order_id = 1

This is a problem if order IDs need to be globally unique. Some solutions are -

    1. UUIDs
    2. Snowflake IDs
    3. Database sequences with shard prefix
    4. ID generation service

For example -

    shard_id + local_sequence

Shard 3 order 100 may become -

    3-100

Large systems often use distributed ID generators.

# SHARDING AND SECONDARY INDEXES

Secondary indexes are harder in sharded databases. Let's say users are sharded by 'user_id.'

This query is easy -

    Get user by user_id

But, this query is harder -

    Get user by email

If email is not the shard key, we do not know which shard contains that email. So, options include -

    1. Query all shards
    2. Maintain a global secondary index
    3. Store email -> user_id mapping separately
    4. Choose email as shard key if that is the main access pattern

For example -

    email_lookup table:
        email -> user_id -> shard

# WHEN TO USE SHARDING?

Use sharding when one database server cannot handle the data size or traffic. Good reasons include -

    1. Database is too large for one machine
    2. Write throughput is too high
    3. Storage limit is reached
    4. Read/write load cannot be handled by replicas alone
    5. Multi-tenant data needs isolation
    6. Geographic data placement is needed

But sharding should not be the first solution for most systems.

Before sharding consider -

    1. Indexing
    2. Query optimization
    3. Caching
    4. Read replicas
    5. Vertical scaling
    6. Partitioning
    7. Archiving old data

# DOWNSIDES OF SHARDING

Sharding has many downsides like -

    1. More operational complexity
    2. Harder queries
    3. Harder joins
    4. Harder transactions
    5. Rebalancing complexity
    6. Hot shard risk
    7. More complex backups
    8. More complex migrations
    9. More complex monitoring
    10. More complex application logic

Sharding solves scale problems, but it creates complexity problems.

This is why we usually avoid sharding until we really need it.

# EXAMPLE SYSTEM DESIGN

Let's say we are designing a large messaging app. Each user has many messages. A common query is -

    Get messages for conversation_id

So, we shard messages by 'conversation_id'.

For example -

    shard = hash(conversation_id) % number_of_shards

This means all messages for one conversation are likely stored on the same shard.

This is good because chat history queries are usually conversation-based.

But, if we need global analytics like -

    Count all messages sent today

That becomes cross-shard.

For analytics, we can stream message events to a data warehouse. So, production read/write path uses shared databases while analytics path uses a separate system.

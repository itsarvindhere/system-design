ACID is a set of properties that make database transactions reliable. ACID stands for -

    1. Atomicity
    2. Consistency
    3. Isolation
    4. Durability

A transaction is a group of database operations that should be treated as one unit.

For example, if we are sending Rs. 1000 from Account A to Account B, the database needs to do two things -

    1. Deduct Rs. 1000 from Account A
    2. Add Rs. 1000 to Account B

These two operations must happen together. If the money is deducted from Account A but not added to Account B, then the system is broken.

So, we wrap both operation inside a transaction -

    Begin transaction
        Deduct Rs. 1000 from Account A
        Add Rs. 1000 to Account B
    Commit transaction

ACID properties make sure this transaction is handled safely.

Relational databases like MySQL, Oracle, SQL Server, PostgreSQL are known for strong ACID transaction support. They are commonly used when correctness matters. For example -

    1. Banking systems
    2. Payment systems
    3. Order management
    4. Inventory systems
    5. Booking systems
    6. Accounting systems

In these systems, data correctness is extremely important.

You do not want money, inventory, or tickets to be duplicated or lost.

# WHY TRANSACTIONS ARE REQUIRED?

Databases often need to perform multiple related operations. For example -

    1. Transfer money
    2. Place an order
    3. Book a ticket
    4. Reserve inventory
    5. Create a user and profile
    6. Update wallet balance
    7. Process payment status

In these cases, partial success can be dangerous. If some steps succeed and some fail, the database may become inconsistent. 

Transactions help us avoid this. They allow us to say -

    Either all important database changes happen, or none of them happen.

Taking the money transfer example, the transaction may look like -

    Begin transaction

        UPDATE accounts
        SET balance = balance - 1000
        WHERE account_id = 'A';

        UPDATE accounts
        SET balance = balance + 1000
        WHERE account_id = 'B';

    Commit transaction

If both updates succeed, we commit.

If something fails, we rollback.

Rollback means undo the changes made inside the transaction.

# ATOMICITY

Atomicity means a transaction is 'all-or-nothing.'

Either all operations inside the transaction happen, or none of them happen. For example -

    Deduct money from Account A
    Add money to Account B

Both should happen together. If the second step fails, the first step should also be undone.

Without Atomicity -

    Account A loses money
    Account B does not receive money

With atomicity -

    Either transfer completes fully
    Or transfer does not happen at all

So atomicity protects us from partial updates.

# CONSISTENCY

Consistency means a transaction should move the database from one valid state to another valid state. The database should obey all rules before and after the transaction.

Rules can be -

    1. Primary key constraints
    2. Foreign key constraints
    3. Unique constraints
    4. Not null constraints
    5. Check constraints
    6. Business rules

For example, if we have a rule -

    Account balance cannot be negative

If Account A has Rs 500, we should not allow a transfer for Rs. 1000. The transaction should fail.

Consistency means the database should not end up in an invalid state.

Another example is -

    Order must belong to an existing user.

If the user does not exist, the database should reject the order because it violates a foreign key rule.

# ISOLATION

Isolation means concurrent transactions should not interfere with each other in an unsafe way.

Databases are used by many users at once. For example -

    User A is booking the last ticket
    User B is also booking the same ticket

Both requests may happen at almost the same time. Without isolation, both users may block the same last ticket.

This is bad.

Ideally this should happen -

    Transaction B waits for A to finish, or Transaction B sees the updated value after A commits.

Isolation makes sure that transactions behave as if they are properly separated from each other.

## ISOLATION PROBLEMS

Isolation is one of the trickiest parts of ACID. When many transactions run at the same time, some problems can happen if isolation is weak such as -

    1. Dirty Read
    2. Non-repeatable Read
    3. Phantom Read
    4. Lost Update

### 1. DIRTY READ

Dirty Read means one transaction reads data written by another transaction that has not yet committed. For example -

    Transaction 1 updates balance to Rs. 4000
    Transaction 2 reads Rs. 4000
    Transaction 1 rolls back

Now, Transaction 2 read data that was never actually committed. This is a dirty read.

### 2. NON-REPEATABLE READ

It means a transaction reads the same row twice and gets different values because another transaction updated it. For example -

    Transaction 1 reads balance = Rs. 5000
    Transaction 2 updates balance = Rs. 4000 and commits
    Transaction 1 reads balance again = Rs. 4000

The same query returned different results inside one transaction.

### 3. PHANTOM READ

It means a transaction runs the same query twice and sees new rows added by another transaction. For example -

    Transaction 1:
        SELECT orders WHERE amount > 1000
        Result = 5 rows

    Transaction 2:
        INSERT new order amount = 2000
        Commit

    Transaction 1:
        SELECT orders WHERE amount > 1000
        Result = 6 rows

This new row is called a 'Phantom Row.'

### 4. LOST UPDATE

It happens when two transactions update the same data and one update overwrites the other. For example -

    Balance = Rs. 1000

    Transaction 1 reads balance = Rs. 1000
    Transaction 2 reads balance = Rs. 1000

    Transaction 1 adds Rs. 100
    Transaction 2 adds Rs. 200

    Transaction 1 writes Rs. 1100
    Transaction 2 writes Rs. 1200

Final balance = Rs 1200. But, the correct balance should be -

    Rs. 1300

So, it means Transaction 1's update was lost.

## ISOLATION LEVELS

Isolation levels define how much one transaction can see the changes made by another concurrent transaction.

Databases provide different isolation levels. Higher isolation gives stronger correctness but may reduce concurrency. Some common isolation levels are -

### 1. READ UNCOMMITTED

It is the weakest isolation level. A transaction may read uncommitted changes from another transaction. This can allow dirty reads. It gives high concurrency but weak correctness.

### 2. READ COMMITTED

A transaction only reads committed data. This prevents dirty reads. But, non-repeatable reads can still happen. This is the common default in many databases.

### 3. REPEATABLE READ

If a transaction reads the same row multiple times, it sees the same value. This prevents dirty reads and non-repeatable reads. Depending on the database, Phantom reads may still be possible.

In databases like PostgreSQL, MVCC or Multi-Version Concurrency Control is used that creates safe, distinct snapshots or versions of data for each transaction. This allows users to read historical versions of data while others are updating it.

Repeatable Read sacrifices "always seeing the latest committed value" in order to guarantee "the same query gives the same result throughout the transaction."

Only when a new transaction starts, the updated value is returned.

### 4. SERIALIZABLE

This is the strongest isolation level. Transactions behave as if they ran one after another. This prevents dirty reads, non-repeatable reads, phantom reads, and many other concurrency anomalies.

But, it can reduce performance because the database needs stricter locking or conflict detection.

## LOCKING

Databases often use locks to maintain isolation. A lock prevents conflicting operations from happening at the same time. For example, if one transaction is updating a row, another may need to wait -

    Transaction 1 locks account A
    Transaction 2 wants to update account A
    Transaction 2 waits

Common locks are -

    1. Shared locks
    2. Exclusive locks
    3. Row-level locks
    4. Table-level locks

### 1. SHARED LOCK

A shared lock is usually used for reading. Multiple transactions can hold shared locks at the same time.

### 2. EXCLUSIVE LOCK

An Exclusive lock is used for writing. If a transaction has an exclusive lock, other transactions usually cannot read or write that data depending on the isolation rules.

### 3. ROW-LEVEL LOCK

A row-level lock locks only the specific rows. This allow better concurrency. For example -

    Lock only account A

Other transactions can still update account B.

### 4. TABLE-LEVEL LOCK

A table-level lock locks the whole table. This is simpler but reduces concurrency.

## DEADLOCK

A deadlock happens when two transactions are waiting for each other forever. For example -

    Transaction 1 locks Row A
    Transaction 2 locks Row B

Then -

    Transaction 1 waits for Row B
    Transaction 2 waits for Row A

Now, both are stuck. Databases usually detect deadlocks and abort one transaction which can then be retried.

# DURABILITY

Durability means once a transaction is committed, the result should not be lost. For example, after the money transfer happened -

    Account A = Rs. 4000
    Account B = Rs. 3000

Even if the database crashes immediately after the commit, the committed data should survive. We achieve durability using -

    1. Write-ahead logs
    2. Disk storage
    3. Replication
    4. Backups
    5. Crash recovery mechanisms

The main idea here is -

    Once the database says "commit successful", the data should be safely stored.

# COMMIT AND ROLLBACK

Two important transaction operations are 'Commit' and 'Rollback.'

## 1. COMMIT

It means permanently save the changes made in the transaction. For example -

    Begin transaction
        Create order
        Reduce inventory
    Commit

After the commit, other transactions can see the final changes depending on the isolation level.

## 2. ROLLBACK

Rollback means undo the changes made in the transaction. For example -

    Begin transaction
        Create order
        Reduce inventory
        Payment fails
    Rollback

After rollback -

    Order is not created
    Inventory is not reduced

It helps recover safely from errors.

# EXAMPLE - SEAT BOOKING

Suppose only one seat is left for a movie.

    Seat A10 - Available

Two users try to book it at the same time. Without proper transaction handling -

    User 1 sees A10 available
    User 2 sees A10 available
    User 1 books A10
    User 2 books A10

Now, the same seat is double-booked. With transaction and isolation, the database can prevent this. For example -

    Transaction 1 locks Seat A10
    Transaction 2 waits
    Transaction 1 books seat and commits
    Transaction 2 checks again and sees seat is no longer available

So only one user gets the seat.

# ACID vs BASE

ACID is often compared with BASE.

BASE stands for -

    1. Basically Available
    2. Soft state
    3. Eventually consistent

It is commonly associated with some distributed NoSQL systems.

ACID focuses on strong correctness while BASE focuses more on availability and eventual consistency.

    ACID:
        Strong consistency
        Transactions
        Safer for critical data

    BASE:
        High availability
        Eventual consistency
        Often used in large distributed systems

For example -

    Bank transfer:
        ACID is important

    Social media like count:
        Eventual consistency may be acceptable

Not every system needs strict ACID for everything.

The choice depends on the business requirements.

# ACID IN DISTRIBUTED SYSTEMS

ACID transactions are easier inside one database. But, across multiple services or databases, they become harder.

For example -

    Order Service database
    Payment Service database
    Inventory Service database

Suppose placing an order requires changes in all three databases. A single local database transaction cannot easily cover all of them.

This is where distributed transaction problems appear.

Some options are -

    1. Two-phase commit
    2. Saga pattern
    3. Event-driven consistency
    4. Outbox pattern

In microservices, we usually avoid large distributed transactions and use eventual consistency with compensating actions. For example -

    Create order
    Reserve inventory
    Payment fails
    Cancel order and release inventory

This is not the same as one ACID transaction, but it is often more practical in distributed systems.

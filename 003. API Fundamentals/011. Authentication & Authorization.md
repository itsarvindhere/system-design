Authentication and Authorization are used to control access to an API.

Authentication checks who the user is.

Authorization checks what the user is allowed to do.

So, even though they sound similar, they are different.

Let's say a user logs into an e-commerce app. First, the system checks their identity -

    Email + Password -> Is this really Arvind?

This is authentication.

After login, the user tries to access an admin page -

    Can Arvind access admin dashboard?

This is authorization.

So authentication happens first, and authorization happens after that.

# WHY DO WE NEED AUTHENTICATION AND AUTHORIZATION?

APIs expose data and actions. Some data is public like -

    GET /products

Anyone may be allowed to see products.

But, some data is private like -

    GET /users/me
    GET /orders/123
    POST /payments
    DELETE /users/123

We do not want anyone to call these API endpoints freely.

Authentication and Authorization help protect APIs from unauthorized access. They help answer questions like -

    1. Is this user logged in?
    2. Is this API key valid?
    3. Is this token expired?
    4. Can this user view this order?
    5. Can this user delete this product?
    6. Can this service call this internal API?
    7. Is this client allowed to perform this action?

Without proper authentication and authorization, APIs can leak private data or allow dangerous actions.

# AUTHENTICATION

It is the process of verifying identity. It answers -

    Who is making this request?

Some common authentication methods are -

    1. Username and password
    2. Session cookies
    3. API keys
    4. JWT tokens
    5. OAuth 2.0
    6. Mutual TLS
    7. Service tokens

The right method depends on the type of client. FOr example -

    Browser app:
        Session cookie or JWT

    Mobile app:
        Access token + refresh token

    Public developer API:
        API key or OAuth

    Internal service-to-service API:
        mTLS or service tokens

# AUTHORIZATION

Authorization is the process of checking permissions. It answers -

    What is this user or client allowed to do?

For example -

    Can this user view this order?
    Can this user update this product?
    Can this API key access this endpoint?
    Can this service call the payment service?

Authorization usually depends on -

    1. User role
    2. Permissions
    3. Resource ownership
    4. Organization/team membership
    5. API scopes
    6. Business rules

# 401 vs 403

This is a common confusion.

'401 Unauthorized' means that the user is not authenticated. It basically means -

    We do not know who you are.

For example, mission token, invalid token, expired session, etc.

'403 Forbidden' means that the user is authenticated but some action is not allowed. It basically means -

    We know who you are, but you cannot do this.

For example -

    Normal user tries to access admin API
    Customer tries to view another customer's order

So -

    401:
        Please login or provide valid credentials

    403:
        You are logged in, but you do not have permission

# SESSION-BASED AUTHENTICATION

Session-based authentication is commonly used in traditional web apps. The flow is basically this -

    1. User logs in with email and password
    2. Server verifies credentials
    3. Server creates a session
    4. Server stores session data
    5. Server sends session ID to browser as a cookie
    6. Browser sends cookie on future requests
    7. Server uses session ID to identify the user

For example -

    Client -> Server:
        POST /login

    Server:
        Verify password
        Create session session_123

    Server -> Client:
        Set-Cookie: session_id=session_123

After login -

    Client -> Server:
        GET /orders
        Cookie: session_id=session_123

In session-based authentication, server usually stores session data. This can be stored in memory, redis, database etc.

For production systems, Redis is commonly used. If we store sessions in the server memory, horizontal scaling becomes harder. For example -

    User logs in on Server 1
    Session stored on Server 1 memory

Next request goes to Server 2 but Server 2 does not know the session_id. This is a problem.

So, the solutions are -

    1. Store sessions in shared Redis
    2. Use sticky sessions
    3. Use stateless token-based auth

# COOKIE SECURITY

When using cookies for authentication cookie security is important. Some common flags used in cookies are -

    1. HttpOnly
    2. Secure
    3. SameSite

## 1. HTTPONLY

It means JavaScript cannot read this cookie. This helps protect against token theft through XSS attacks. For example -

    Set-Cookie: session_id=abc123; HttpOnly

## 2. SECURE

It means the cookie is only sent over HTTPS. This prevents the cookie from being sent over unencrypted HTTP. For example -

    Set-Cookie: session_id=abc123; Secure

## 3. SAMESITE

This controls whether cookies are sent with cross-site requests. It prevents the CSRF attacks. Some common values are -

    SameSite=Strict
    SameSite=Lax
    SameSite=None

If SameSite is set to None, then cookie must also use 'Secure.'

# TOKEN-BASED AUTHENTICATION

Token-based authentication is common for APIs, mobile apps, SPAs, and microservices. So, the basic flow is this -

    1. User logs in
    2. Server verifies credentials
    3. Server returns an access token
    4. Client stores the token
    5. Client sends token with future requests
    6. Server verifies token

Usually, the token is sent in the 'Authorization' header like this -

    Authorization: Bearer token123

If the token is invalid, server returns '401 Unauthorized.'

# JWT

JWT stands for JSON Web Token. A JWT is a token format that can carry claims about the user. It usually has three parts -

    header.payload.signature

For example -

    xxxx.yyyy.zzzz

The payload may contain claims like -

    {
        "userId": "123",
        "role": "admin",
        "exp": 1710000000
    }

The signature is used to verify that the token was issues by a trusted server and was not modified. So, when the server receives a JWT, it can verify the signature and read the claims.

## STATELESS AUTHENTICATION WITH JWT

Often, JWTs are used for stateless authentication. This means, the server does not need to store session data for every user. The token itself is enough and contains enough information like -

    userId
    role
    expiry time

The server verifies the token signature and if valid, it trusts the claims. So, the horizontal scaling is easier.

For example -

    Client -> Any App Server:
        Authorization: Bearer JWT

Any server can verify the JWT without checking a central session store.

Now, the JWTs have downsides. The first downside is revocation. If a JWT is valid for only 1 hour, it may remain valid until expiry even if the user logs out or access is removed. With session, the server deletes the session.

With JWT, revocation is harder unless we maintain a denylist or user short expiry times.

The second downside is the token size. JWTs can be larger than simple session IDs. If the token is sent with every request, this adds an overhead

The third downside is stale permissions. If a user's role changes, the old JWT may contain the old role until it expires.

Finally, we have the storage risk. If an attacker steals the token, they can use it until the token expires. So, JWTs should be short-lived and stored carefully.

# ACCESS TOKENS AND REFRESH TOKENS

Many systems use two tokens - Access Token and Refresh Token.

## 1. ACCESS TOKEN

This is used to call APIs. It is usually short-lived. For example -

    Expires in 5 minutes
    Expires in 15 minutes
    Expires in 1 hour

The client sends it with the requests -

    Authorization: Bearer access_token

Short expiry reduces damage if the token is stolen.

## 2. REFRESH TOKEN

This is used to get a new access token and it is usually long-lived. For example -

    Expires in 7 days
    Expires in 30 days

When the access token expires, the client calls -

    POST /auth/refresh

The server then verifies the refresh token and returns a new access token. Refresh tokens should be stored more securely because they live longer.

# API KEYS

They are commonly used for developer APIs and service integrations. For example -

    X-API-Key: api_key_123

API keys identify the calling application or developer. They are useful for -

    1. Public APIs
    2. Third-party integrations
    3. Usage tracking
    4. Rate Limiting
    5. Billing

They are simple but they are not always enough for user-level authorization. An API key usually tells us which application is calling. It may not tell us which user is using that application.

# OAUTH 2.0

OAuth 2.0 is a framework for delegated authorization. It is commonly used when one application wants to access user's data from another application.

For example -

    A calendar app wants to access user's Google Calendar.

The user does not give their Google password to the calendar app. Instead, the user approves access through Google.

The app receives a token with limited permissions. For example -

    Scope: read_calendar

Now, the app can read the calendar data, but it cannot access everything in the user's account. This is what delegated access means.

For example, let's say a photo printing app wants to access a user's Google Photos. The flow can be -

    1. User clicks "Connect Google Photos"
    2. User is redirected to Google
    3. Google asks user for permission
    4. User approves selected permissions
    5. Google redirects back with an authorization code
    6. App exchanges code for access token
    7. App uses access token to call Google Photos API

So, the app never sees user's Google password. It only gets a token with limited access.

## SCOPES

A scope defined what a token is allowed to do. For example -

    read:orders
    write:orders
    read:profile
    admin:users

If a token only has "read:orders" then it should not be allowed to create orders. Scopes are common un OAuth and public APIs.

# RBAC

RBAC stands for Role-Based Access Control. In RBAC, the permissions are based on roles. For example -

    Admin
    Seller
    Customer
    Support Agent

Each role has permissions -

    Admin:
        Manage users
        Delete products
        View all orders

    Seller:
        Manage own products
        View own sales

    Customer:
        View own orders
        Place orders

When a user performs an action, the system checks their role. RBAC is simple and common but can be limiting if permissions become too complex.

# ABAC

ABAC stands for Attribute-Based Access Control. Unlike RBAC, the authorization decisions in ABAC are made based on 'attributes.'

Attributes can include -

    1. User attributes
    2. Resource attributes
    3. Environment attributes
    4. Request attributes

For example -

    User department = Finance
    Document department = Finance
    Request time = office hours

So, a rule can be -

    User can access document if user.department == document.department

Another example is -

    User can approve expense only if amount < approval_limit

It is more flexible than RBAC but also more complex.

# RESOURCE-LEVEL AUTHORIZATION

It means checking whether a user can access a specific resource. For example -

    GET /orders/123

It is not enough to check -

    Is the user logged in?

We must also check -

    Does order 123 belong to this user?

Without it, one user may access another user's data by changing the ID.

    GET /orders/123
    GET /orders/124

This type of vulnerability is very common and so, every sensitive resource access should check ownership or permission.

# AUTHENTICATION IN MICROSERVICES

In microservices, authentication and authorization can happen at multiple levels. A common flow is -

    Client -> API Gateway -> Service A -> Service B

The API Gateway may authenticate the user -

    Is the token valid?

Then it forwards the user identity to the backend service. For example -

    userId: 123
    roles: customer

But, the backend services should still check for authorization. For service-to-service calls, services may use -

    1. mTLS
    2. Service tokens
    3. JWTs issued for internal services
    4. API gateway/service mesh identity

For example, if Order Service makes a call to Payment Service, the payment service should know -

    Is this really Order Service?
    Is Order Service allowed to create payment?

API Gateways often handle common authentication tasks like -

    1. Validate token
    2. Reject unauthenticated requests
    3. Apply rate limits
    4. Check basic scopes
    5. Forward identity to services

But, not all authorization should be done at the gateway. Deep business authorization should usually stay inside the service. For example -

    Can user_123 cancel order_456?

The Order Service is the best place to know this because it owns order data.

# EXAMPLE SYSTEM DESIGN

Let's say we are designing authentication and authorization for an e-commerce API. The flow can be -

    Client -> API Gateway -> Backend Services

User Login -

    1. User sends email and password
    2. Auth Service verifies password
    3. Auth Service issues access token and refresh token
    4. Client stores tokens securely
    5. Client sends access token with API requests

So, the request flow is -

    Client -> API Gateway:
        GET /orders/123
        Authorization: Bearer access_token

The API gateway -

    1. Verifies token signature
    2. Checks token expiry
    3. Extracts userId and role
    4. Forwards request to Order Service

The Order service -

    1. Fetches order 123
    2. Checks whether order.userId == currentUserId
    3. If yes, returns order
    4. If no, returns 403 Forbidden

So, remember -

    Authentication proves identity.
    Authorization controls access.

Simply put, an API is a way for two software systems to talk to each other.

API stands for Application Programming Interface.

It defines how one application can request data or functionality from another application, and what kind of response it can expect.

So, if one system wants to use something from another system, it does not directly go inside that system's code or database. Instead, it communicates through an API.

For example, suppose we have a food delivery app.

When the app wants to show nearby restaurants, it may call an API like -

    GET /restaurants?location=delhi

The server then responds with data like -

    {
        "restaurants": [
            {
                "name": "Burger House",
                "rating": 4.5
            },
            {
                "name": "Pizza Corner",
                "rating": 4.2
            }
        ]
    }

The mobile app does not need to know how the backend stores restaurant data, how the database is designed, or how the search logic works.

All it needs to know is -

    1. What API endpoint to call?
    2. What data to send?
    3. What response it will get back?

So, the core idea is -

    API is a contract between two software systems

One system exposes some functionality, and another system uses it through a defined interface.

# TYPES OF APIs

When we talk about the types of APIs, they can mean two different things -

    1. Types based on "who can access the API."
    2. Types based on "how the API is designed."

## 1. TYPES OF APIs BASED ON ACCESS

### 1. PUBLIC API

A Public API is an API that is exposed to external developers or users. This means people outside the company can use it, usually after creating an account or getting an API key.

For example -

    - Google Maps API
    - Stripe API
    - X API
    - OpenWeather API

If you are building an app and you want to show maps, you can use Google Maps API instead of building your own map system. So, Public APIs allow outside developers to use some functionality of a system.

### 2. PRIVATE API

A Private API is used only inside an organization. It is not exposed to external users. For example, suppose Amazon has internal services like -

    1. User Service
    2. Order Service
    3. Payment Service
    4. Inventory Service

These services may communicate with each other using private APIs. The outside world does not directly access these APIs. So, Private APIs are used for internal communication between systems or services.

### 3. PARTNER API

A Partner API is exposed only to selected business partners. It is not fully public, but it is also not completely internal.

For example, a travel company may expose APIs to selected hotel partners or airline partners. A payment company may expose special APIs to merchants or banking partners.

So, Partner APIs are shared with trusted external parties under some agreement.

## 2. TYPES OF APIs BASED ON DESIGN STYLE

### 1. REST

REST stands for Representational State Transfer.

REST API is one of the most common types of API. It is usually built on top of HTTP. In REST, we think in terms of resources. A resource can be anything like -

    user
    product
    order
    payment
    comment

Each resource usually has a URL. For example -

    GET /users/10
    GET /products/55
    POST /orders
    DELETE /comments/99

The HTTP method tells what action we want to perform.

    GET     fetch data
    POST    create something
    PUT     replace something
    PATCH   partially update something
    DELETE  remove something
 
REST APIs are popular because they are simple, easy to understand, and widely supported.

### 2. GRAPHQL

GraphQL is another API style where the client can ask for exactly the data it needs. In REST, if you call an endpoint, the server decides the response structure. For example -

    GET /users/10

The server may return name, email, address, profile picture, created date, and many other fields. But maybe the client only needs name and email. In GraphQL, the client can ask only for those fields.

For example -

    user {
        name
        email
    }

This can reduce over-fetching, where the client receives more data than it needs.

GraphQL is useful when the frontend needs flexible data, especially in apps where different screens need different combinations of data.


### 3. SOAP

SOAP stands for Simple Object Access Protocol. It uses XML for request and responses. SOAP APIs are more strict and formal compared to REST.

They are still used in some enterprise systems, banking systems, government systems, and legacy applications. A SOAP request may look more complex because it uses XML structure.

SOAP is not as common in modern web applications as REST, but it is still important to know because many older systems use it.

### 4. gRPC

RPC stands for Remote Procedure Call.

gRPC is a modern API style created by Google. It is commonly used for communication between backend services. Instead of using JSON like REST, gRPC usually uses Protocol Buffers, also called 'Protobuf'.

Protobuf is a compact binary format, so it is faster and smaller than JSON.

gRPC is useful when services need fast communication with each other.

For example in a microservices system -

    Order Service -> Payment Service
    Payment Service -> Fraud Detection Service
    Inventory Service -> Product Service

gRPC can be a good choice for internal service-to-service communication.

### 5. WEBSOCKET

A WebSocket API is used when we need real-time two-way communication. In normal HTTP APIs, the client sends a request and the server sends a response. But with WebSockets, once a connection is opened, both client and server can send messages to each other anytime.

This is useful for -

    1. Chat applications
    2. Live notifications
    3. Multiplayer games
    4. Live stock prices
    5. Collaborative editing
    6. Real-time dashboards
 
For example, in a chat app, the server should be able to push a new message to the user immediately.

Using normal REST APIs, the client would have to keep asking the server again and again -

    Any new message?
    Any new message?
    Any new message?

This is called polling.

WebSockets avoid this by keeping an open connection.

### 6. SSE

SSE stands for Server-Sent Events. SSE is used when the server needs to continuously send updates to the client over a single long-lived HTTP connection. The important thing is that SSE is one-way.

    Server -> Client

The client opens a connection, and after that the server can keep sending events to the client.

But the client cannot send messages back on the same SSE connection.

It can be useful for -

    1. Live notifications
    2. Live scores
    3. News feed updates
    4. Monitoring dashboards
    5. Stock price updates
    6. AI response streaming

Let's say we have a live cricket scores app. The client can open an SSE connection -

    GET /scores/live

Now, whenever the score changes, the server can push an update like -

    event: score_update
    data: India 152/3 after 18 overs

The client does not need to keep asking again and again for updates.

So, SSE avoids polling.

But unlike WebSockets, SSE is only server-to-client.

This makes it simpler than WebSockets when we only need live updates from the server.

For example, if we are building a chat app, SSE may not be enough by itself because the client also needs to send messages to the server.

But for a live dashboard, notification stream, or AI text streaming response, SSE can be a very good choice.

### 7. WEBHOOKS

A Webhook is used when one system wants to notify another system when something happens.

In normal APIs, the client calls the server whenever it wants data.

But in Webhooks, the direction is reversed.

The server calls a URL provided by the client when an event happens.

For example, suppose we are using Stripe for payments.

Our application can give Stripe a webhook URL like -

    POST /webhooks/stripe

Now when a payment succeeds, Stripe sends a request to our webhook endpoint.

    {
        "event": "payment_succeeded",
        "payment_id": "pay_123",
        "amount": 500
    }

So instead of our system repeatedly asking Stripe -

    Did the payment succeed?
    Did the payment succeed?
    Did the payment succeed?

Stripe itself tells our system when the payment succeeds.

This is very useful for event-based communication between systems.

Some examples are -

    1. Stripe sends payment status updates
    2. GitHub sends push or pull request events
    3. Shopify sends order creation events
    4. Slack sends event notifications
    5. Razorpay sends payment updates

One important thing about Webhooks is reliability. What if our server is down when Stripe sends the webhook?

In that case, good webhook providers usually retry the webhook multiple times.

Also, our system should verify that the webhook really came from the provider.

For example, Stripe signs webhook requests. Our server should verify the signature before trusting the event.

Another important point is that webhooks should usually be idempotent.

This means if the same webhook event is received multiple times, processing it multiple times should not create duplicate side effects.

For example, if we receive the same payment_succeeded event twice, we should not mark the order as paid twice or send two confirmation emails.

So, Webhooks are powerful, but we need to handle retries, duplicates, and security carefully.

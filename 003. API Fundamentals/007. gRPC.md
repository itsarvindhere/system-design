RPC stands for Remote Procedure Call.

A procedure is basically a function or a method.

So, RPC means calling a function that exists on another machine or another service.

For example, in normal code, we may call -

    createPayment(orderId, amount)

But, in distributed system, the Payment service may be on a different server. So, the order service may need to call it over the network -

    Order Service -> Payment Service:
        createPayment(orderId, amount)

This is the idea behind RPC.

It makes communication between services look like function calls.

But, we should remember that -

    A remote function call is still a network call.

It means, it can fail, timeout, be slow, or return errors.

# WHAT IS gRPC?

gRPC is a modern RPC framework originally created by Google.

It is built on top of HTTP/2 and commonly uses protocol buffers for data serialization.

So, the main pieces are -

    1. RPC style communication
    2. HTTP/2 as the transport
    3. Protocol Buffers as the data format
    4. Strongly typed service contracts
    5. Code generation for clients and servers

In simple terms -

    gRPC = RPC + HTTP/2 + Protobuf + Code Generation

It is designed to be fast, efficient, and language independent. For example, the User Service may be written in Go, the Order Service in Java, and the Payment Service in Python.

They can still communicate using gRPC as long as they share the same '.proto' contract.

# PROTOCOL BUFFERS

Protocol Buffers are also known as 'Protobuf'. They are the default data format used by gRPC.

Protobuf is a binary serialization format. This means, the data is converted into compact binary form before it is sent over the network.

In REST APIs, we often use JSON.

For example -

    {
        "userId": "123",
        "name": "Arvind",
        "email": "arvind@example.com"
    }

It is human-readable, but it is larger and slower to parse.

In gRPC, the same data is usually sent as Protobuf binary. It is not easily readable by humans, but is smaller and faster.

In gRPC, services and messages are defined in a '.proto' file. This file acts like a contract between client and the server. For example -

    service PaymentService {
        rpc CreatePayment (CreatePaymentRequest) returns (CreatePaymentResponse);
    }

    message CreatePaymentRequest {
        string order_id = 1;
        int32 amount = 2;
        string currency = 3;
    }

    message CreatePaymentResponse {
        string payment_id = 1;
        string status = 2;
    }

This tells us -

    1. There is a service called PaymentService
    2. It has a method called CreatePayment
    3. CreatePayment accepts CreatePaymentRequest
    4. It returns CreatePaymentResponse
    5. The request and response have clearly defined fields

From this '.proto' file, gRPC tools can generate client and server code.

For example -

    Java client code
    Go server code
    Python client code
    TypeScript types

This gives strong typing between services.

# HOW gPRC WORKS

A typical gRPC flow looks like this -

    1. Define service and messages in a .proto file
    2. Generate client and server code
    3. Server implements the service methods
    4. Client calls the generated method
    5. gRPC serializes the request using Protobuf
    6. Request is sent over HTTP/2
    7. Server receives and deserializes it
    8. Server executes the method
    9. Response is serialized and sent back
    10. Client receives the response

For example -

    Order Service calls Payment Service

    Order Service:
        paymentClient.CreatePayment(request)

    gRPC sends request over network

    Payment Service:
        CreatePayment(request)

    Payment Service returns response

    Order Service receives response

To the developer, it feels like calling a function. But, under the hood, gRPC handles network communication, serialization, and response handling.

# WHY HTTP/2?

gRPC uses HTTP/2 instead of HTTP/1.1.

HTTP/2 gives gRPC several advantages like Multiplexing, Header compression, long-lived connections, streaming support, and better performance.

Multiplexing means multiple requests and responses can share the same TCP connection at the same time.

    One TCP connection:
        Stream 1 -> CreatePayment
        Stream 2 -> GetUser
        Stream 3 -> UpdateInventory

The reduces the overhead of opening many connections.

HTTP/2 compresses headers. This reduces repeated metadata overhead. This is useful because service-to-service calls often include repeated headers like authentication tokens, trace IDs, content types etc.

HTTP/2 also supports streams naturally. gRPC uses this to support different communication patterns like server streaming, client streaming, and bi-directional streaming.

# TYPES OF gRPC COMMUNICATION

gRPC supports four main types of communication.

## 1. UNARY RPC

This is the simplest type. The client sends one request and the server sends one response.

This is similar to a REST API call.

    GetUser(userId) -> User
    CreatePayment(request) -> PaymentResponse
    GetProduct(productId) -> Product

Most gRPC APIs start with unary calls.

## 2. SERVER STREAMING RPC

In server streaming, the client sends one request and the server sends many responses -

    Client -> Server: Request
    Server -> Client: Response 1
    Server -> Client: Response 2
    Server -> Client: Response 3

For example, let's say a client asks for live order updates 

    GetOrderUpdates(orderId)

The server can keep sending updates -

    Order accepted.
    Food preparing.
    Out for delivery.
    Delivered.

This is useful when the server has multiple pieces of data to send over time.

## 3. CLIENT STREAMING RPC

In Client streaming, the client sends many requests and the server sends one response.

For example, suppose a client uploads a large file in chunks -

    UploadChunk 1
    UploadChunk 2
    UploadChunk 3

After all the chunks are sent, the server then returns -

    Upload complete

This is useful when the client needs to send a sequence of data.

## 4. BI-DIRECTIONAL STREAMING RPC

As the name suggests, in this one, both client and server can send multiple messages over the same connection.

    Client -> Server: Message 1
    Server -> Client: Message A
    Client -> Server: Message 2
    Server -> Client: Message B

Both sides can communicate independently.

This is useful in real-time systems. For example -

    1. Chat between services
    2. Live collaboration
    3. Real-time game state sync
    4. Streaming telemetry
    5. Voice/video metadata signaling

This looks similar to WebSockets, but it is usually used more for service-to-service communication.

# gRPC BENEFITS AND DOWNSIDES

Some benefits include -

    1. High performance
    2. Smaller payloads using Protobuf
    3. Strongly typed contracts
    4. Code generation
    5. Supports multiple languages
    6. Built-in streaming support
    7. Works well for microservices
    8. Uses HTTP/2 features
    9. Good for internal APIs

The biggest benefit is that the service contracts are very clear The '.proto' file defines exactly what methods exist and what messages they accept and return. This reduces confusion between teams and services.

But, gRPC also has downsides. The first downside is that it is harder to debug manually. With REST, we can easily use a browser, curl, or Postman and read the JSON response.

With gRP, the data is binary Protobuf, so we need special tools like grpcurl, BloomRPC, Postman gRPC support etc.

Second downside is browser support. Browsers do not yet fully support raw gRPC directly in the same way backend services do. For browser clients, we often need gRPC-Web or a gateway.

The third downside is complexity. Teams need to manage '.proto' files, generated code, versioning, and compatibility.

gRPC is not ideal for public APIs. Public developers often prefer REST because it is easier to inspect, test, and integrate.

The fifth downside is infrastructure support. Some order proxies, load balancers, or API gateways may not fully support HTTP/2 or gRPC features.

# DEADLINES AND TIMEOUTS

gRPC supports deadlines. A deadline means the client says -

    I am willing to wait only until this time.

For example -

    CreatePayment deadline: 2 seconds

It means, if the server does not respond within this time, the call fails. This is important in microservices. Without deadlines, one slow service may cause many other services to wait.

For example, if we have -

    API Gateway -> Order Service -> Payment Service -> Bank Service

If Bank Service is slow and no timeouts exist, the whole request chain will get stuck.

# ERROR HANDLING

gRPC has some standard status codes like -

    OK
    INVALID_ARGUMENT
    NOT_FOUND
    ALREADY_EXISTS
    PERMISSION_DENIED
    UNAUTHENTICATED
    RESOURCE_EXHAUSTED
    FAILED_PRECONDITION
    UNAVAILABLE
    DEADLINE_EXCEEDED
    INTERNAL

For example, GetUser(unknown_user) may return -

    NOT_FOUND

# VERSIONING

Versioning is important in gRPC because many services may depend on the same '.proto' contract. The key rule is -

    Do not break existing clients.

In Protobuf, each field has a number.

    message User {
        string id = 1;
        string name = 2;
        string email = 3;
    }

These field numbers are important. They are used in the binary format so we should not reuse field numbers for different meanings.

    For example, if we remove email = 3, we should not later use 3 for phone.

A safer approach is -

    message User {
        string id = 1;
        string name = 2;
        reserved 3;
        string phone = 4;
    }

This prevents accidental reuse.

# gRPC USE CASES

gRPC is commonly used in backend systems where performance and strong contracts matter. Good use cases are -

    1. Microservice-to-microservice communication
    2. Low-latency internal APIs
    3. High-throughput systems
    4. Streaming telemetry
    5. ML model serving
    6. Internal platform APIs
    7. Communication between polyglot services

For example -

    Order Service -> Payment Service
    Payment Service -> Fraud Detection Service
    Inventory Service -> Warehouse Service
    API Gateway -> User Service

gRPC is especially useful when the services are owned by the same organization and can share '.proto' contracts.

Avoid gRPC when -

    1. The API is mainly public-facing and should be easy for external developers
    2. Browser clients need to call the API directly without extra tooling
    3. Human-readable requests and responses are important
    4. The team does not want to manage `.proto` files and generated code
    5. The system is simple and REST is enough
    6. Infrastructure does not support HTTP/2 well

# gRPC vs WEBSOCKETS

gRPC and WebSockets can both support streaming, but they are not the same.

WebSocket is a protocol used for long-lived bi-directional communication.

It is commonly used between browser clients and servers. For example -

    Browser <-> Chat Server

gRPC is an RPC framework for calling methods on remote services. It is commonly used between backend services.

So, WebSocket is good for real-time client-server communication while gRPC is good for structured service-to-service communication.

Also, gRPC gives us service contracts, generated code, deadlines, status codes, and typed messages.

# EXAMPLE SYSTEM DESIGN

Let's say we are designing an e-commerce backend.

The frontend talks to an API Gateway using REST.

    Browser -> API Gateway:
        POST /orders

The API Gateway then talks to internal services using gRPC.

    API Gateway -> Order Service:
        CreateOrder()

The Order Service calls other services.

    Order Service -> Inventory Service:
        ReserveItems()

    Order Service -> Payment Service:
        CreatePayment()

    Order Service -> Notification Service:
        SendOrderConfirmation()

The internal flow may look like this -

    Browser
        -> REST
        -> API Gateway
        -> gRPC
        -> Order Service
        -> gRPC
        -> Payment Service

This is a common pattern.

REST is used externally because it is simple and browser-friendly. gRPC is used internally because it is fast and strongly typed.

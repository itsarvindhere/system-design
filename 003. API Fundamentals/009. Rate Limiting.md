Rate Limiting means controlling how many requests a user, client or a system can make in a given amount of time.

For example, a user can make 100 requests per minute. So, if a user sends requests within the limit, we allow them. Otherwise, we reject or slow down the extra requests.

# WHY DO WE NEED RATE LIMITING?

Rate Limiting is used to protect systems from overload, abuse, spam and unfair usage.

Let's say we have an API.

Normally, users may send a few requests per minute. But, one user or bot starts sending thousands of requests per second.

If we do not control it, this abusive user can overload the system. And this will affect all the users.

    1 bad client -> server overloaded -> everyone suffers

Rate limiting helps prevent this. It makes sure one user or client cannot consume too many resources.

Rate Limiting helps in -

    1. Preventing abuse
    2. Protecting backend services
    3. Avoiding accidental overload
    4. Reducing spam
    5. Controlling API usage
    6. Improving fairness
    7. Protecting against brute-force attacks
    8. Managing cost in cloud systems

Let's take an example where we have a login API.

    POST /login

If there is no rate limiting, an attacker can try millions of passwords -

    Try password 1
    Try password 2
    Try password 3
    ...
    Try password 1,000,000

This is dangerous. So, we should add a rate limit -

    Maximum 5 login attempts per minute per IP

Now, if someone tries too many passwords quickly, we simply block or slow them down.

This helps prevent the user accounts from brute-force attacks.

# WHERE RATE LIMITING IS USED?

Rate Limiting can be used in many places like -

    1. API Gateway
    2. Load Balancer
    3. Reverse Proxy
    4. Application Server
    5. Service Mesh
    6. Database Access Layer
    7. Message consumers
    8. External API Clients

For public APIs, it is done at the API Gateway. For example -

    Client -> API Gateway -> Backend Service

So here, the API Gateway will check the rate limits before forwarding the requests -

    If within limit:
        Forward request

    If over limit:
        Return 429 Too Many Requests

# RATE LIMITING RESPONSE

When a client exceeds the limit, the server commonly returns -

    429 Too Many Requests

The response may include headers like -

    Retry-After: 60

This tells the client when to retry. Some APIs also return rate limit headers. For example -

    X-RateLimit-Limit: 100
    X-RateLimit-Remaining: 20
    X-RateLimit-Reset: 1710000000

These headers help clients understand their current limit.

# RATE LIMITING KEY

To apply rate limiting, we need to decide what we are limiting by. This is called a Rate Limiting Key.

Some common keys are -

    1. IP Address
    2. User ID
    3. API Key
    4. Account ID
    5. Device ID
    6. Session ID
    7. Endpoint
    8. Region

For example -

    100 requests per minute per user

Here, the key is 'user_id'.

OR -

    1000 requests per minute per API key

Here, the key is 'api_key'.

For login protection, we may use -

    5 login attempts per minute per IP
    10 login attempts per minute per username

This prevents both IP-based and account-based attacks.

# RATE LIMITING BY IP

IP-based rate limiting is simple. For example -

    100 requests per minute per IP address

This is useful for unauthorized users because we may not know who they are yet. But, IP-based rate limiting has problems.

Many users share the same IP address. For example -

    1. Office network
    2. College Wi-Fi
    3. Mobile carrier network
    4. Public Wi-Fi

If we rate limit too aggressively by IP, we may block many real users. Also, attackers can use many IP addresses.

So IP-based rate limiting is useful, but it should not be the only strategy for important systems.

# RATE LIMITING BY USER OR API KEY

For authenticated APIs, rate limiting by user ids or api key is usually better. For example -

    Free user:
        100 requests/minute

    Premium user:
        1000 requests/minute

    Enterprise customer:
        custom limit

This allows fair usage based on account type.

API-key based limiting is common for developer APIs.

For example -

    API key abc123:
        10,000 requests/day

This lets the platform enforce usage plans and billing limits.

# RATE LIMITING ALGORITHMS

## 1. FIXED WINDOW COUNTER

Fixed Window Counter divides time into fixed windows. For example -

    Limit: 100 requests per minute.

The server counts requests from -

    10:00:00 to 10:00:59

Then, resets the counter at -

    10:01:00

So, the flow is -

    User sends request
    Increment counter for current minute
    If counter <= 100:
        allow
    Else:
        reject

This is simple and fast.

But, it has a problem at window boundaries. Suppose, the limit is 100 requests per minute.

A user sends -

    100 requests at 10:00:59
    100 requests at 10:01:00

Technically, both are allowed because they fall into different windows. But, the user sent 200 requests in 2 seconds. This burst can overload the system.

So, Fixed Window Counter is simple, but not very smooth.

## 2. SLIDING WINDOW LOG

Sliding Window Log stores timestamps of each request.

For example, the limit is -

    100 requests per minute.

When a new request comes in, we check all requests from the last 60 seconds -

    Current time: 10:01:20
    Check requests after: 10:00:20

If there are fewer than 100 requests in that rolling period, allow the request. Otherwise, reject it.

So, the flow is -

    1. Remove timestamps older than 60 seconds
    2. Count remaining timestamps
    3. If count < limit, allow and store current timestamp
    4. Else reject

This is accurate and it prevents the fixed-window boundary problem.

But, it can use a lot of memory because we store every request timestamp. If millions of users send many requests, this becomes expensive.

## 3. SLIDING WINDOW COUNTER

Sliding Window Counter is an approximation of Sliding Window Log. It uses counters for the current window and the previous window.

So, instead of storing every request timestamp, it estimates the rolling count.

For example -

    Limit: 100 requests per minute

Suppose -

    Previous window counter = 80
    Current window count = 30

    Current window is 25% complete

We calculate a weighted count from the previous window -

    Estimated count = previous_count * remaining_fraction + current_count

If the estimate is under the limit, we allow.

This is less accurate than Sliding Window Log, but much more memory efficient.

It is a good balance between accuracy and performance.

## 4. TOKEN BUCKET

Token Bucket is one of the most popular rate limiting algorithms.

Imagine every user has a bucket. Tokens are added to the bucket at a fixed rate. For example -

    Add 10 tokens per second
    Bucket capacity = 100 tokens

Each request needs one token.

    If token available:
        consume token and allow request

    If no token:
        reject or delay request

This allows controlled bursts. For example, if the user has not sent requests for some time, tokens accumulate and then the users can send a burst of requests.

But, the burst cannot exceed the bucket capacity.

For example -

    Refill rate: 10 requests/second
    Bucket capacity: 100

This means -

    Long-term rate = 10 requests/second
    Maximum burst = 100 requests

Token Bucket is useful because real traffic is often bursty. It allows short bursts while still controlling long-term usage.

## 5. LEAKY BUCKET

This is another common algorithm. Imagine requests entering a bucket. The bucket leaks at a constant rate.

For example -

    Process 10 requests per second

If requests arrive faster than that, they wait in that bucket. If the bucket becomes full, extra requests are rejected.

So, the flow is -

    Request arrives
    If queue has space:
        add to queue
    Else:
        reject request

Requests are processed at a steady rate. This smoothens the traffic.

Token Bucket allows burst while Leaky Bucket smooths the bursts into a constant output rate.

Token Buckets and Leaky Buckets are often confused. The simple difference is -

    Token Bucket:
        Controls how many requests can be accepted
        Allows bursts if tokens are available

    Leaky Bucket:
        Controls how fast requests are processed
        Smooths traffic into a steady flow

So -

    Token Bucket:
        User can suddenly send 50 requests if 50 tokens are available

    Leaky Bucket:
        User may send 50 requests, but they are processed gradually

Token Bucket is commonly used for API rate limiting.

Leaky Bucket is useful when we want a steady processing rate.

# DISTRIBUTED RATE LIMITING

On a single server, Rate Limiting is easy. But in real systems, we often have multiple application servers.

Let's say we have three servers -

    Client -> Load Balancer -> App Server 1
    Client -> Load Balancer -> App Server 2
    Client -> Load Balancer -> App Server 3

Now, if the limit is -

    100 requests per minute per user

If each server keeps its own counter, the user may get more than 100 requests. This is wrong. So, for distributed rate limiting, we need a shared state.

Some common options are -

    1. Redis
    2. Memcached
    3. Central rate limiting service
    4. API Gateway
    5. Service mesh
    6. Local counters with approximate limits

Redis is often used for Distributed Rate Limiting. For example, for a fixed window counter -

    Key:
        rate_limit:user_123:2026-06-20T10:01

    Value:
        request_count

Each request increments the counter -

    INCR rate_limit:user_123:minute

If this is the first request, set the expiration -

    EXPIRE key 60 seconds

If the count is under limit, allow. If over limit, reject.

Redis operations can be atomic, which helps avoid race conditions.

# LOCAL VS GLOBAL RATE LIMITING

## 1. LOCAL RATE LIMITING

Local rate limiting happens inside one server. Each server tracks its own limits.

This is simple and fast because no external call is required. But, it is less accurate in distributed systems.

For example, if we have 5 servers and each allow 100 requests/minute, the global limit may become 500 requests/minute.

Local limits are useful when approximate protection is enough.

## 2. GLOBAL RATE LIMITING

Global rate limiting uses shared state or a central service. So, all the servers check the same limit.

For example -

    App Server -> Redis:
        Is user_123 under limit?

This is more accurate.

But, it adds latency and dependency on the rate limiting store. If Redis goes down, we need a fallback strategy.

# FAIL OPEN VS FAIL CLOSED

When the rate limiting system fails, we need to decide what to do.

For example, what if Redis goes down? The app cannot check whether the user is over the limit or not.

So, we have two options.

## 1. FAIL OPEN

Fail Open means allow requests if the rate limiter is unavailable. This protects availability but it may allow abuse during the failure.

This should be used when user availability is more important.

For example -

    Normal product browsing
    Low-risk APIs

## 2. FAIL CLOSED

Fail closed means reject requests if the rate limiter is unavailable.

This protects the system.

But it may block legitimate users.

Use this for sensitive or high-risk endpoints.

For example -

    Login attempts
    Payment attempts
    Password reset
    Expensive AI generation APIs

# HARD LIMITING VS SOFT LIMITING

Rate Limiting does not always mean immediately rejecting requests. There are different behaviors.

## 1. HARD LIMITING

Hard Limiting rejects requests after the limit is reached. For example -

    429 Too Many Requests

This one is simple and common.

## 2. SOFT LIMITING

Soft Limiting warns, delays, or throttles the requests before fully rejecting them. For example -

    1. Add artificial delay
    2. Lower priority of requests
    3. Return warning headers
    4. Queue requests for later
    5. Degrade response quality

Soft limiting can give a smoother user experience.

For example, instead of rejecting immediately, we may slow down repeated login attempts.

# RATE LIMITING VS THROTTLING

Rate Limiting and Throttling are related, but slightly different.

Rate Limiting usually means enforcing a maximum number of requests in a time period. For example -

    100 requests per minute.

Throttling usually means slowing down the requests or controlling processing speed.

For example -

    Process only 10 requests per second.

In practice, people often use these terms interchangeably. But the idea is -

    Rate limiting:
        Reject or block requests above a limit

    Throttling:
        Slow down or smooth traffic

# RATE LIMITING VS QUOTAS

Rate Limits usually apply over short time windows like -

    100 requests per minute

Quotas usually apply over longer periods like -

    10,000 requests per day
    1 Million requests per month

Rate Limiting protect system stability while Quotas control usage plans, billing and fairness over time.

For example -

    Free plan:
        100 requests/minute
        10,000 requests/month

    Paid plan:
        1000 requests/minute
        1,000,000 requests/month

Both can be used together.

# ENDPOINT SPECIFIC RATE LIMITS

Not all endpoints should have the same limit. For example -

    GET /products:
        1000 requests/minute

    POST /login:
        5 requests/minute

    POST /payments:
        10 requests/minute

    POST /ai/generate:
        20 requests/day

Some endpoints are cheap and safe while some are expensive and sensitive. So, Rate Limits should reflect risk and cost.

# RATE LIMITING FOR LOGIN

Login endpoints need special care. A good login rate limit may combine multiple keys -

    5 attempts per minute per IP
    10 attempts per hour per username
    100 attempts per hour per subnet

This helps prevent attacks from one IP and distributed attacks against one account. We may also add -

    1. Captcha after repeated failures
    2. Temporary account lock
    3. Alert for suspicious activity
    4. Progressive delay

The goal is to slow attackers without hurting real users too much.

# RATE LIMITING AND CACHING

Rate Limiting and caching can work together. Suppose one popular API is expensive -

    GET /products/trending

If many users request it, we can cache the response. Caching reduces backend load.

Rate Limiting controls abusive usage.

So, 

    Caching:
        Reduces repeated work

    Rate limiting:
        Controls how much traffic is allowed

Both are useful for protecting systems.

# COMMON RATE LIMITING MISTAKES

    1. Only rate limiting by IP
    2. Using the same limit for every endpoint
    3. Forgetting distributed systems
    4. Not using atomic counters
    5. Blocking real users behind shared IPs
    6. Not returning useful 429 responses
    7. Not setting Retry-After headers
    8. Not monitoring rate limit events
    9. Failing closed for low-risk endpoints
    10. Failing open for highly sensitive endpoints

A good rate limiter should protect the system without creating too many false positives.

# EXAMPLE SYSTEM DESIGN

Let's say we are designing rate limiting for a public API. The requirement is -

    100 requests per minute per API key

A simple architecture may be -

    Client
        -> API Gateway
        -> Rate Limiter
        -> Backend Service

The flow is -

    1. Client sends request with API key
    2. API Gateway extracts API key
    3. Rate Limiter checks usage for that API key
    4. If within limit, request goes to backend
    5. If over limit, return 429 Too Many Requests

With Redis, the flow may be -

    API Gateway -> Redis:
        Increment counter for api_key_123

Redis returns the current count -

    If count <= 100:
        allow request

    If count > 100:
        reject request

For Token Bucket -

    1. Each API key has a bucket
    2. Tokens refill at a fixed rate
    3. Each request consumes one token
    4. If tokens are available, allow
    5. If no tokens are available, reject

So, the simple way to remember Rate Limiting is -

    It is a traffic control system for your APIs.
    
    It makes sure one client cannot take more than their fair share.

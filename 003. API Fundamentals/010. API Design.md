An API is not just a URL. It is a contract. It tells the clients -

    1. What operations are available
    2. What data they need to send
    3. What response they will get
    4. What errors can happen
    5. How authentication works
    6. What limits and rules apply

API design means deciding how different clients or services will communicate with our system.

A well-designed API is easy to understand, easy to use, consistent, secure, and hard to misuse.

# WHY API DESIGN MATTERS?

Often, APIs are used by a lot of clients. For example Web apps, mobile apps, admin dashboards, third-party developers, and internal services.

If poorly designed, every client suffers.

For example, if our API endpoints look like this -

    GET /getUser
    POST /fetch-products
    GET /order_details
    POST /deleteCartItem

This becomes confusing.

A better API is predictable. For example -

    GET /users/{userId}
    GET /products
    GET /orders/{orderId}
    DELETE /cart/items/{itemId}

So, a good api design reduces confusion between frontend, backend, mobile, and external developers.

An API should be treated like a contract between a provider and a consumer. If the backend exposes this endpoint -

    GET /users/123

The client expects a specific kind of response. For example -

    {
        "id": "123",
        "name": "Arvind",
        "email": "arvind@example.com"
    }

If the response is suddenly changes to this -

    {
        "user_id": "123",
        "full_name": "Arvind"
    }

This may break existing clients. So, once an API is released, we should be careful when changing it. API design is not just about making things work today. It is also about making sure they can evolve safely later.

# RESOURCE-BASED DESIGN

In REST APIs, the endpoints are usually designed around resources.

A resource is simply a thing in the system. For example, if we have an e-commerce app, the resources can be Users, Products, Orders, Card Items, Payments, Reviews, etc.

So, URLs should usually represent resources.

For example -

    /users
    /users/{userId}
    /products
    /products/{productId}
    /orders
    /orders/{orderId}

Action-heavy URLs should be avoided. For example -

    /getUser
    /createOrder
    /deleteProduct
    /updateUserEmail

Since in REST, the HTTP method already tells the action, the URL should usually be a noun, not a verb.

# HTTP METHODS

## 1. GET

GET is used to read data. For example -

    GET /products
    GET /products/123
    GET /orders/456

GET should not change the state of the server. It should be safe and idempotent.

## 2. POST

POST is usually used to create a new resource or trigger a non-idempotent operation. For example -

    POST /orders
    POST /payments
    POST /users

Every POST /orders may create a new order. So, POST is usually not idempotent by default.

For important operations like payments, idempotency keys should be used.

## 3. PUT

PUT is used to replace a resource or set it to a specific state. For example -

    PUT /users/123

This may replace the full user object. PUT is generally idempotent. This means, sending the same PUT request again should result in the same final state.

## 4. PATCH

PATCH is used for partial updates. For example -

    PATCH /users/123
    {
        "email": "new@example.com"
    }

This updates only the user email. This is useful when the client does not want to send the full response.

## 5. DELETE

DELETE is used to remove a resource. For example -

    DELETE /products/123

It is usually idempotent, meaning calling it multiple times should leave the system in the same final state.

# URL NAMING

Good URL naming makes the APIs easier to understand.

For collections, we should use plural nouns. For example -

    /users
    /products
    /orders

Use IDs for specific resources -

    /users/123
    /products/55
    /orders/789

Nesting is usually avoided. For example, instead of -

    GET /users/123/orders

which should mean -

    Get orders for the user with id 123

We should instead use query parameters like -

    GET /orders?user=123

# QUERY PARAMETERS

Query parameters are key-value pairs appended to the end of a URL after a question mark (?) to filter, sort, or paginate data returned by an API. 

They allow clients to pass optional arguments to the server to customize the response without changing the resource's core identity or endpoint structure.

For example -

    GET /products?category=shoes
    GET /orders?status=paid
    GET /users?role=admin

For sorting -

    GET /products?sort=price
    GET /products?sort=-createdAt

Here, '-createdAt' may mean descending order.

For search -

    GET /products?search=iphone

For pagination -

    GET /products?page=1&limit=20

Note that here, the connector (&) separates multiple parameters when chaining them together.

The Query parameters should be used for optional modifiers.

The path should identify the resource. The query parameters should modify how to fetch the resource.

# DESIGNING REQUEST BODY

For POST, PUT, and PATCH, the clients usually send data in the request body. For example -

    POST /products
    {
        "name": "iPhone 15",
        "price": 79999,
        "category": "phones"
    }

A good request body should be clear and predictable. We should avoid unclear names like -

    {
        "n": "iPhone 15",
        "p": 79999
    }

Also, avoid allowing clients to send fields they should not control. For example, when creating an order, they should not send -

    {
        "status": "paid"
    }

The server should be the one that decides the order status.

# DESIGNING RESPONSE BODY

API responses should be consistent. For a single resource -

    {
        "id": "123",
        "name": "Arvind",
        "email": "arvind@example.com"
    }

If we have to send a list of resources -

    {
        "data": [
            {
                "id": "1",
                "name": "Product A"
            },
            {
                "id": "2",
                "name": "Product B"
            }
        ],
        "pagination": {
            "limit": 20,
            "nextCursor": "abc123"
        }
    }

So, consistency is important. If one endpoint returns -

    {
        "data": [...]
    }

And the other endpoint returns -

    {
        "items": [...]
    }

And another just returns raw arrays -

    [...]

The clients become harder to write. We should pick a pattern and then use that everywhere.

# STATUS CODES

HTTP status codes tell the client what happens. We can group them in five classes -

    1. Informational responses (100 – 199)
    2. Successful responses (200 – 299)
    3. Redirection messages (300 – 399)
    4. Client error responses (400 – 499)
    5. Server error responses (500 – 599)

Here are some common ones -

    - 200 OK - Request succeeded and we are returning data
    - 201 Created - A new resource is created
    - 204 No Content - Request succeeded but there is no response body
    - 400 Bad Request - When request is malformed or invalid
    - 401 Unauthorized - When the user is not authenticated
    - 403 Forbidden - When the user is authenticated but now allowed
    - 404 Not Found - Resource does not exist
    - 409 Conflict - Request conflicts with the current state. For example, trying to create an account with an email that already exists
    - 422 Unprocessable Entity - The server understands your request and its structure, but cannot process it due to invalid data or validation errors within the content
    - 429 Too Many Requests - When the client exceeds the rate limits
    - 500 Internal Server Error - There is some unexpected condition encountered by the server and it cannot fulfil the request
    - 503 Service Unavailable - Web server is operational but temporarily unable to handle the request

# ERROR RESPONSE DESIGN

Error responses should also be consistent. Here is an example of a bad error response -

    Something went wrong

A better error response is -

    {
        "error": {
            "code": "ORDER_NOT_FOUND",
            "message": "Order not found",
            "requestId": "req_123"
        }
    }

For validation errors -

    {
        "error": {
            "code": "VALIDATION_ERROR",
            "message": "Invalid request body",
            "fields": {
                "email": "Email is required",
                "password": "Password must be at least 8 characters"
            },
            "requestId": "req_123"
        }
    }

A good error response helps clients understand what went wrong and how to fix it.

For example in the example above, the 'requestId' is useful for debugging. The client can report the request ID, and backend logs can be searched using that ID.

# PAGINATION

Pagination is important when returning large lists. Never return unlimited data from a list endpoint. For example -

    GET /orders

returns 1 million orders.

This can overload the server and the client. We should rather do -

    GET /orders?limit=20&page=1

There are two common pagination styles.

## 1. OFFSET-BASED PAGINATION

In offset-based pagination, the client specifies two things -

    1. offset = How many records to skip
    2. limit = How many records to return

For example -

    GET /users?offset=20&limit=10

This means, skip the first 20 records and return the next 10 records. So, the database query can be -

    SELECT * FROM users
    ORDER BY id
    LIMIT 10 OFFSET 20;

So, it is simple and easy to understand but has problems for large datasets.

It is easy to jump to a page for example -

    ?page=50

Because page numbers can be directly converted into offsets -

    offset = (page - 1) × pageSize

So, if page number is 50 and page size is 20, then the offset will be -

    (50 - 1) * 20
    = 980

so, if the API call is -

    GET /users?page=50&pageSize=20

Then the backend converts it to this SQL -

    SELECT * FROM users
    ORDER BY id
    LIMIT 20 OFFSET 980;

The database skips the first 980 rows and returns rows 981–1000.

But, the offset-based pagination has downsides.

If new items are inserted while the user is paginating, results can shift. Also, large offsets can be slow in databases. For example, if we have -

    OFFSET 100000

This means, database has to scan many rows before returning the results.

## 2. CURSOR-BASED PAGINATION

Cursor pagination uses a pointer to the next page.

For example -

    GET /products?limit=20&cursor=abc123

The response may include -

    {
        "data": [...],
        "nextCursor": "def456"
    }

So, the client uses the 'nextCursor' for the next request. Cursor pagination is better for large datasets and real-time changing data. It is commonly used for feeds, timelines, notifications and large APIs.

So, instead of saying 'skip N rows', we say -

    Give me records after this specific item

So, usually the cursor is the last record's ID, timestamp etc. For example, if the first request is -

    GET /users?limit=10

The response can be -

    {
        "data": [...],
        "nextCursor": "100"
    }

And so, the next request will be -

    GET /users?cursor=100&limit=10

Based on this, the SQL can be -

    SELECT * FROM users
    WHERE id > 100
    ORDER BY id
    LIMIT 10;

The cursor-based pagination is very fast for large database as it uses indexed lookups like -

    WHERE id > 100

instead of offsets like -

    OFFSET 100000

Assuming the cursor column is stable, this also means no duplicates or missing records due to inserts/deletes.

For infinite scrolling, it is preferred.

Speaking of disadvantages, we cannot easily jump to a specific page like page 50. The implementation is also more complex and it requires a stable sort key like ID or timestamp.

## WHEN TO USE WHICH

For things like admin dashboards, reports, tables where the users want page numbers, or small to medium datasets, offset-based pagination can be used.

For social media feeds, chat messages, activity timelines, and large scale systems, we should use cursor-based pagination.

# FILTERING AND SORTING

APIs should provide clear options for filtering and sorting. For filtering -

    GET /orders?status=paid
    GET /orders?userId=123
    GET /products?category=phones

For sorting -

    GET /products?sort=price
    GET /products?sort=-price

For multiple filters -

    GET /orders?status=paid&from=2026-06-01&to=2026-06-20

But, we should avoid making the filter logic too complex unless needed. For a very complex search, it is better to create a search endpoint such as -

    POST /products/search

So now, we can send a request body like -

    {
        "query": "iphone",
        "filters": {
            "priceMin": 50000,
            "priceMax": 100000
        },
        "sort": "-rating"
    }

# API VERSIONING

APIs change over time. Versioning helps us evolve the APIs without breaking existing clients. Some common approaches are -

    1. URL versioning
    2. Header versioning
    3. Query parameter versioning

## 1. URL VERSIONING

URL Versioning is common and simple. For example -

    /v1/users
    /v2/users

Here, version is the part of the URL. It is simple, easy to understand, and enables easy routing at API gateway.

But, the downside here is that the URL changes when the version changes.

## 2. HEADER VERSIONING

In this, version is passed in a custom header -

    GET /users/123
    API-Version: 2

Header versioning may also use the 'Accept' header like this -

    Accept: application/vnd.myapp.v2+json

This means the URL stays clean and resource remains the same. The disadvantages is that it is less intuitive and more complex.

## 3. QUERY PARAMETER VERSIONING

Query Parameter Versioning may look like -

    /users?version=2

It is easy for the clients to implement but generally not preferred and easy for clients to forget.

For public APIs, the URL versioning is easy for developers to understand. The main rule is -

    Do not break existing clients without a migration path.

Breaking changes include -

    1. Removing a field
    2. Renaming a field
    3. Changing field type
    4. Changing error format
    5. Changing endpoint behavior
    6. Making an optional field required

Adding new optional fields is usually safe.

# BACKWARD COMPATIBILITY

It means old clients should continue working after the API changes. For example, let's say the old response was -

    {
        "id": "123",
        "name": "Arvind"
    }

Adding a new field is usually safe -

    {
        "id": "123",
        "name": "Arvind",
        "profileImage": "https://example.com/a.png"
    }

Old clients can ignore the new field.

But, if we rename a field, it may break old clients -

    {
        "id": "123",
        "fullName": "Arvind"
    }

So, breaking changes should be avoided unless we create a new API version.

# AUTHENTICATION AND AUTHORIZATION

APIs should clearly define authentication and authorization. Authentication means 'Who are you?' while Authorization means 'Are you allowed to do this?'.

Some common methods of authentication are -

    1. Session cookies
    2. JWT tokens
    3. API keys
    4. OAuth 2.0

For example -

    Authorization: Bearer token123

So, if a call is made -

    GET /orders/456

The server should check if the user is allowed to view the order 456. It is not enough to only check that the user is logged in. The user must also have permission for the specific resource.

# API SECURITY

API design should include security from the very beginning. Important API security practices include -

    1. Use HTTPS
    2. Authenticate sensitive endpoints
    3. Check authorization at resource level
    4. Validate input
    5. Avoid exposing internal fields
    6. Rate limit sensitive endpoints
    7. Avoid leaking detailed system errors
    8. Use request size limits
    9. Log important actions
    10. Protect against injection attacks

For example, we should not return raw database errors to clients like -

    SQL syntax error near users table

A good error will be -

        {
        "error": {
            "code": "INTERNAL_ERROR",
            "message": "Something went wrong"
        }
    }

# API DOCUMENTATION

A good API should be well documented. Documentation should include things like -

    1. Endpoint URL
    2. HTTP method
    3. Authentication requirements
    4. Request body
    5. Query parameters
    6. Response body
    7. Error responses
    8. Rate limits
    9. Examples

For Rest APIs, OpenAI or Swagger are commonly used. A documented API is easier to test, debug, integrate, and maintain.

Without documentation, clients have to guess how the API works. This usually leads to bugs.

# API DESIGN EXAMPLE

Let's say we are designing an API for an e-commerce application. Some good endpoints may be -

    Users:
        GET /users/me
        PATCH /users/me

    Products:
        GET /products
        GET /products/{productId}

    Cart:
        GET /cart
        POST /cart/items
        PATCH /cart/items/{itemId}
        DELETE /cart/items/{itemId}

    Orders:
        POST /orders
        GET /orders
        GET /orders/{orderId}
        POST /orders/{orderId}/cancel

    Payments:
        POST /payments
        GET /payments/{paymentId}


Some operations do not fit perfectly in basic CRUD. For example -

    Cancel Order

We can design it as -

    POST /orders/{orderId}/cancel

This is acceptable because cancellation is a domain action. Not every API needs to be forced into pure CRUD. The important thing is that API remains clear and consistent.

# COMMON API DESIGN MISTAKES

    1. Inconsistent naming
    2. Using verbs everywhere in URLs
    3. Returning different response formats from different endpoints
    4. Not using proper HTTP status codes
    5. Returning too much data
    6. Not supporting pagination
    7. Not validating input properly
    8. Exposing internal implementation details
    9. Breaking clients without versioning
    10. Not designing error responses properly
    11. Not supporting idempotency for critical operations
    12. Not documenting the API
    13. Confusing authentication and authorization

Good API design is mostly about consistency and clarity.

So, remember -

    An API is not just how the backend works.

    It is how other systems understand and safely use your backend.

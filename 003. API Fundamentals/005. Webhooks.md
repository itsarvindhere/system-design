In the last section, we learnt about WebSockets and how they are really useful in real-time applications.

There is another concept that sounds similar to WebSockets and it is Webhooks.

A Webhook is simply a way for one system to automatically notify another system when something happens.

Instead of our application repeatedly asking another service -

    Did something happen?
    Did something happen?
    Did something happen?

The other service directly calls your application when the event happens.

So, Webhooks are event-driven.

For example, let's say we use Stripe for payments. When a user completes a payment, how do we know when the payment succeeds since we do not own Stripe.

Well, Stripe can send a request to our backend -

    Stripe -> Our Backend: Payment successful

Then, our backend updates the order status from Pending to Paid.

So, in Webhooks, the basic idea is -

    Something happens in System A
    System A sends an HTTP request to System B
    System B handles that event

# WHY DO WE NEED WEBHOOKS?

First, let's see what is the alternative of Webhooks. 

Suppose we are building an e-commerce app and we use a payment provider like Stripe, Razorpay, or PayPal.

After the user starts payment, our backend needs to know whether the payment succeeded or failed.

One approach is polling where our backend keeps asking the payment provider whether the payment is successful or not. This works, but this is not efficient.

In a lot of the cases, we are just making unnecessary calls.

With Webhooks, the payment provider tells us when the payment status changes.

So now, our backend does not need to constantly ask. This is better because it reduces unnecessary API calls, gives near real-time updates, is more scalable, allows systems to communicate asynchronously, and works well for event-driven workflows.

# HOW WEBHOOKS WORK?

In our application, we expose an HTTP endpoint and then register that endpoint with another service.

That service sends an HTTP request to our endpoint when an event happens. Our application receives the request and processes the event.

For example, we may have a backend that exposes this endpoint -

    POST /webhooks/payment

Then, we configure this URL in the payment provider dashboard -

    https://api.myapp.com/webhooks/payment

Now, whenever a payment event happens, the payment provider will send an HTTP request to this URL. For example, the request body might look like -

    {
        "event": "payment.success",
        "paymentId": "pay_123",
        "orderId": "order_456",
        "amount": 999
    }

Our backend receives this event and updates the database.

# WEBHOOKS ARE JUST HTTP REQUESTS

Unlike WebSockets, Webhooks are not some special protocol.

A Webhook is simply an HTTP request and most commonly, it is a POST request.

The external system calls our endpoint. So from our server's point of view, a Webhook is like any other API request.

The difference is in who starts the request.

In a normal API call -

    Our App -> External Service

In a Webhook -

    External Service -> Our App

# PRODUCER AND CONSUMER

In Webhooks, there are usually two sides -

    1. Webhook Producer
    2. Webhook Consumer

The Webhook Producer is the system where the event happens. Some examples are Stripe, GitHub, Shopify, Slack, Razorpay, and more.

The Webhook Consumer is the system that receives and processes the event. For example, our backend application.

# COMMON USE CASES OF WEBHOOKS

## 1. PAYMENT UPDATES

Payment providers use Webhooks to notify our backend about payment events. For example, payment success, payment failed, payment created, subscription cancelled and so on.

## 2. GITHUB EVENTS

GitHub can send Webhooks when something happens in a repository. For example push, pull request, issue created, release published etc. CI/CD systems often use this. For example -

    Developer pushes code
    GitHub sends Webhook to CI server
    CI server starts build and tests

So, the flow here is -

    GitHub -> CI/CD Server -> Build Pipeline

## 3. E-COMMERCE EVENTS

Platforms like Shopify send Webhooks for store events like order created, order paid, order cancelled, product updated etc. Our system can listen to these events and update internal data.

## 4. SUBSCRIPTION EVENTS

If we have a SaaS app, subscription providers can send events like subscription created, subscription renewed and more.

Our system can use these events to enable or disable user access.

## 5. NOTIFICATIONS AND AUTOMATION

Webhooks are also used to connect tools. For example -

    Monitoring system detects error
    Sends Webhook to Slack
    Slack posts alert in channel

OR

    User submits form
    Form tool sends Webhook to CRM
    CRM creates new lead

# WEBHOOK VS WEBSOCKET

While Webhooks and WebSockets are both used for real-time or near real-time communication, they solve different problems,

A WebSocket creates a long-lived connection and both sides can send messages anytime.

A Webhook does not keep a connection open. It is just an HTTP request sent when an event happens.

For example, in a chat app, we may use WebSocket to send messages instantly to users.

But in a payment system, we may use Webhooks to receive payment status updates from Stripe.

# WEBHOOKS CAN FAIL!

A very important thing to remember is that Webhook delivery is not always perfect. Many things can go wrong.

For example -

    1. Our server may be down
    2. Network may fail
    3. Request may timeout
    4. Webhook may be sent twice
    5. Events may arrive out of order
    6. Our processing may fail halfway

So we should not assume that every Webhook will arrive exactly once and in perfect order.

In real systems, Webhooks should be designed carefully.

# WEBHOOK RETRIES

Most Webhook providers retry failed Webhooks. For example, if our server returns a 500 error, the provider may retry the same Webhook after some time.

    Attempt 1 -> Failed
    Wait 1 minute
    Attempt 2 -> Failed
    Wait 5 minutes
    Attempt 3 -> Success

This is useful because temporary failures can recover automatically.

But retries create another problem.

The same event may be received more than once.

So our Webhook handler must be idempotent. Idempotency means that processing the same event multiple times should not create incorrect results.

# WEBHOOK SECURITY

Webhook endpoints are public URLs. This means anyone can try to send fake requests to them. So, if our webhook endpoint is -

    POST /webhooks/payment  

An attacker may try to send a fake payment success event -

    {
        "event": "payment.success",
        "orderId": "order_456"
    }

If we blindly trust this request, we may mark an unpaid order as paid. So, security is very important. Here are some common techniques -

## 1. SIGNATURE VERIFICATION

Most providers sign Webhook requests using a secret key. The provider creates a signature from the request body and the secret. Our backend also uses the same secret to verify the signature.

If both signatures match, we know the request really came from the provider and was not modified -

    Provider:
        signature = hash(payload + secret)

    Our Backend:
        expected_signature = hash(payload + secret)

    If signature == expected_signature:
        accept request
    Else:
        reject request

## 2. TIMESTAMP VALIDATION

Some providers include a timestamp in the request send to our application. Our backend then checks the timestamp whether it is recent. For example -

    Webhook timestamp should be within last 5 minutes

This helps protect against replay attacks.

A replay attack means someone captures a valid Webhook request and sends it again later.

## 3. HTTPS

Webhook endpoints should use HTTPS.

This protects the request in transit and prevents attackers from easily reading or modifying the payload.

# WEBHOOK SYSTEM DESIGN

Suppose we are designing a payment Webhook system. A simple design may look like this -

    Payment Provider
        -> Webhook Endpoint
        -> Database

This may work for a small system. For a production system however, a better design is -

    Payment Provider
        -> Webhook Endpoint
        -> Event Store
        -> Queue
        -> Worker
        -> Database

So, the flow is -

    1. Payment provider sends Webhook
    2. Webhook endpoint verifies signature
    3. Webhook endpoint stores raw event in database
    4. Webhook endpoint checks idempotency
    5. Webhook endpoint pushes event to queue
    6. Webhook endpoint returns 200 OK
    7. Worker processes event asynchronously
    8. Worker updates order/payment/subscription state

This is safer because even if worker processing fails, we still have the original event stored.

Let's say a user placed an order -

    Order ID: order_456
    Status: Pending

Our backend creates a payment using the payment provider API.

    Our Backend -> Payment Provider:
        Create payment for order_456

The user completes the payment so the payment provider sends a Webhook -

    Payment Provider -> Our Backend:
        payment.success

The payload may contain -
    
    {
        "eventId": "evt_123",
        "event": "payment.success",
        "paymentId": "pay_789",
        "orderId": "order_456",
        "amount": 999
    }

Our backend should then -

    1. Verify signature
    2. Check if evt_123 was already processed
    3. Store the event
    4. Mark order_456 as Paid
    5. Store paymentId pay_789
    6. Send confirmation email
    7. Mark evt_123 as processed

If the same Webhook comes again, our backend should detect it.

    evt_123 already processed

So it should not mark the order again or send duplicate emails.

# OUT OF ORDER EVENTS

Some Webhook events may arrive out of order. For example, we may receive -

    payment.refunded

Before 

    payment.success

This can happen due to retry delays, network issues or provider behavior.

So our system should not blindly assume event order.

Instead, we should check the current state before applying changes.

For example -

    If order is already Refunded:
        Do not move it back to Paid

OR

    If event timestamp is older than last processed event:
        Ignore or handle carefully

A good system usually stores the current state and applies valid transitions. For example -

    Pending -> Paid -> Refunded

But it should avoid invalid transitions like -

    Refunded -> Paid

Unless there is a special reason.

# WEBHOOKS AND QUEUES

Queues are very useful with Webhooks.

Instead of processing everything immediately, the Webhook can put the event into a queue -

    Webhook Provider -> Webhook Endpoint -> Queue -> Worker

The benefits are -

    1. The Webhook endpoint responds quickly
    2. Heavy processing happens in background
    3. Failed jobs can be retried
    4. Traffic spikes can be absorbed
    5. Processing becomes more reliable

For example, during a sale, thousands or webhooks might arrive quickly. If we process everything directly in the request, our endpoint may become slow. But, if we push events to a queue, workers can process them at a controlled pace.

# WEBHOOKS AND RECONCILIATION

Even with Webhooks, we should sometimes run reconciliation jobs.

Reconciliation means periodically checking the external system and comparing it with our internal state. For example -

    Our database says:
        order_456 is Pending

    Payment provider says:
        payment for order_456 is Successful

This means, we missed or failed to process a Webhook.

A reconciliation job can fix this.

The flow may be -

    1. Find pending orders older than 10 minutes
    2. Call payment provider API
    3. Check actual payment status
    4. Update order if needed

This is important because Webhooks are useful, but we should not depend on them as the only source of truth for critical workflows.

For payments, reconciliation is especially important.

# WHEN TO USE WEBHOOKS AND WHEN NOT TO

Use Webhooks when another system needs to notify your system about events. Good use cases include -

    1. Payment status updates
    2. Subscription lifecycle events
    3. GitHub push events
    4. CI/CD triggers
    5. Order updates from e-commerce platforms
    6. CRM lead creation
    7. Monitoring alerts
    8. Third-party integration events

Do not use Webhooks when your client needs live UI updates.

For that, WebSockets or Server-Sent Events may be better.

For example -

    Chat message to browser:
        WebSocket

    Payment success from Stripe to backend:
        Webhook

One of the most important concepts to understand in APIs is Idempotency.

Idempotency means doing the same operation multiple times should have the same final result as doing it once.

For example, let's say we have a request -

    Cancel order 123

If we send it once, the order becomes cancelled.

If we accidentally send it again, the order should still remain cancelled.

    First request:
        Order 123 -> Cancelled

    Second request:
        Order 123 -> Still Cancelled

The second request should not create a new side-effect. This is Idempotency.

# WHY DO WE NEED IDEMPOTENCY

In distributed systems, things fail in confusing ways.

For example, suppose a client sends a payment request.

    Client -> Server:
        Create payment of Rs. 1000

The server receives the payment, creates the payment, but the response goes lost because of a network issue. From the client's POV, it looks like the request failed. So, the client retries.

    Client -> Server:
        Create payment of Rs. 1000 again

Now, the dangerous questions is -

    Did the first request actually fail?
    Or did only the response fail?

If the server processes the second request as a new request, the user will be charged twice. This is the exact kind of problem Idempotency solves.

With idempotency, the server can recognize that both requests are actually the same operation.

So it does not process the payment twice.

# IDEMPOTENT VS NON-IDEMPOTENT OPERATIONS

Some operations are naturally idempotent and some are not.

An operation is Idempotent if repeating it does not change the final result after the successful execution. For example -

    Set user email to arvind@example.com

If we do it once -

    email = arvind@example.com

If we do it again -

    email = arvind@example.com
    
So, the final state is the same.

Another example is -

    email = arvind@example.com

After one request -

    order status = Cancelled

After five requests -

    order status = Cancelled

So, this is Idempotent.

But, some operations are non-idempotent. In such cases, repeating that operation creates additional effects. For example -

    Add Rs. 100 to wallet

If we do it once -

    wallet balance == 100

If we do it again -

    wallet balance = 200

The final state changes because the same operation was repeated.

# IDEMPOTENCY IN HTTP METHODS

## 1. GET

GET is idempotent. If is used to read data. So, we do make one call -

    GET /users/123

We will get the details of the user with id 123. It does not matter how many times we make this call, the result will not change server state.

Of course the returned data may change if some other operation updated it, but the GET request itself should not cause the change.

## 2. PUT

PUT is generally idempotent. It usually replaces or sets a resource to a specific state. For example -

    PUT /users/123
    {
        "name": "Arvind"
    }

If we send it once, the name becomes 'Arvind'. If we send it again, the name is still 'Arvind'. So, the final state remains the same.

## 3. DELETE

DELETE is generally idempotent. For example -

    DELETE /orders/123

If we do it once, it deletes the order. If we do it again, the server may return -

    404 Not Found

    OR

    204 No Content

But, the final state is the same -

    order_123 does not exist anymore.

So, it is idempotent in terms of the final state.

## 4. POST

POST is usually not idempotent by default. For example -

    POST /orders

Every call may create a new order -

    First call -> order_1
    Second call -> order_2
    Third call -> order_3

That is why POST operations like payments, order creation, booking, and transfers often need explicit idempotency handling.

# IDEMPOTENCY KEY

An idempotency key is a unique value sent by the client to identify one logical operation. For example -

    Idempotency-Key: 7f9a1c2d

The client generates this key before making the request. If the same operation is retried, the client sends the same key again -

    First attempt:
        POST /payments
        Idempotency-Key: 7f9a1c2d

    Retry attempt:
        POST /payments
        Idempotency-Key: 7f9a1c2d

The server stores the key and the result of the operation. So, if the same key comes again, the server does not repeat the operation.

It just returns the previous result. So, the flow is -

    1. Client generates an idempotency key
    2. Client sends request with that key
    3. Server checks if the key already exists
    4. If key does not exist, server processes the request
    5. Server stores the key and response
    6. If the same key comes again, server returns stored response

# WHERE TO STORE IDEMPOTENCY KEYS?

The server needs to store the idempotency keys somewhere durable. Some common options are Database tables, Redis, Distributed cache, Dedicated idempotency store, etc.

For critical operations like payments, a database is usually safe because we do not want the keys to disappear too early.

A simple table may look like -

    idempotency_key
    request_hash
    status
    response_body
    created_at
    expires_at

For example -

    abc123 -> payment_789 -> success

Now if abc123 comes again, the server knows what happened.

One important detail is that the same Idempotency key should not be reused for a different request. For example, this is valid -

    Idempotency-Key: abc123
    Amount: 1000

Retry -

    Idempotency-Key: abc123
    Amount: 1000

But, this is suspicious -

    Idempotency-Key: abc123
    Amount: 1000

Retry -

    Idempotency-Key: abc123
    Amount: 5000

The same key is being used for a different operation. So, may systems store a request hash -

    request_hash = hash(method + path + body)

When the same idempotency keys is used again, the server compares the request hash -

    If key exists and request hash is same:
        return stored result

    If key exists but request hash is different:
        reject request

This prevents accidental or malicious key reuse.

# IDEMPOTENCY AND RETRIES

Retries and Idempotency are closely related. Retries are common in distributed systems because requests can fail due to -

    1. Network timeout
    2. Server overload
    3. Temporary database issue
    4. Load balancer timeout
    5. Client losing connection
    6. Response getting lost

Without idempotency, retries can be dangerous.

With idempotency, retries become safer.

For example -

    Create payment request times out
    Client retries with same idempotency key
    Server returns original payment result

So the client does not need to know whether the first attempt succeeded or failed. The server handles it safely.

# IDEMPOTENCY AND DATABASE TRANSACTIONS

Idempotency should often be combined with database transactions.

Suppose we are creating a payment and storing the idempotency key.

We do not want this situation -

    Payment created
    Server crashes before storing idempotency key

Now if the client retries, the server may not know the first payment happened.

So we should try to make the business operation and idempotency record part of the same transaction.

For example -

    Begin transaction
        Insert idempotency key
        Create payment record
        Store response
    Commit transaction

This way, the system does not end up half-aware of what happened.

# RACE CONDITIONS

Idempotency can have race conditions. Suppose, the same request is sent twice at almost the same time -

    Request 1 with key abc123
    Request 2 with key abc123

Both reach the server together. If both check the database before either inserts the key, both may think -

    Key does not exist

Then both may process the operation. To prevent this, the idempotency key should have a unique constraint like -

    idempotency_key UNIQUE

Now, only one request can insert the key. The other request will see that the key already exists. This is very important.

Without a uniqueness guarantee, idempotency can fail under concurrency.

# EXPIRATION OF IDEMPOTENCY KEYS

Idempotency keys do not always need to be stored forever. For example, a payment API may store idempotency keys for 24 hours. After that the key may expire.

This prevents the idempotency store from growing forever.

But expiration depends on the use case.

For payments and financial systems, we may want longer retention.

For less critical operations, shorter retention may be fine.

The important question is -

    How long can a retry realistically happen?

The key should live at least that long.

# IDEMPOTENCY VS EXACT-ONCE

Idempotency is used to make the systems behave safely even when messages are delivered more than once. But, this does not mean the system has true exactly-once delivery.

In distributed systems, exactly-once delivery is very hard. Many systems provide 'at-least-once' delivery which means -

    The message will be delivered one or more times.

So, duplicates can happen.

Idempotency helps us safely handle those duplicates.

For example -

    Queue delivers same message twice
    Worker processes first message
    Worker ignores second message because eventId already processed

So idempotency gives us effectively-once behavior at the business level.

# IDEMPOTENCY VS DEDUPLICATION

Deduplication means detecting and removing duplicate requests or messages. Idempotency means handling repeated operations safely.

They are not exactly the same. For example -

    Deduplication:
        I have seen this event ID before, so I will ignore it.

    Idempotency:
        Even if this operation runs again, the final state will remain correct.

Deduplication is one way to implement Idempotency. But, some operations are naturally idempotent without explicit deduplication like -

    Set order status to Cancelled

Running this multiple times still gives the same final result.

# COMMON MISTAKES

Some common mistakes with idempotency are -

    1. Not using idempotency for payment/order creation
    2. Generating a new idempotency key on every retry
    3. Not storing the response of the original request
    4. Not using a unique constraint on idempotency keys
    5. Not handling concurrent duplicate requests
    6. Reusing the same key for different request bodies
    7. Expiring keys too early
    8. Assuming retries only happen from clients
    9. Forgetting idempotency in webhook handlers and queue consumers

We need idempotency when repeating an operation could cause incorrect side-effects. For example -

    1. Payments
    2. Wallet credits/debits
    3. Bank transfers
    4. Order creation
    5. Ticket booking
    6. Inventory updates
    7. Webhook processing
    8. Queue message processing
    9. Sending notifications
    10. Subscription updates

We usually do not need heavy idempotency for simple read operations.a

# EXAMPLE SYSTEM DESIGN

Let's say we are designing an order creation API.

The client sends -

    POST /orders
    Idempotency-Key: key_123

The backend flow can be -

    1. Receive request
    2. Validate idempotency key
    3. Create request hash
    4. Check idempotency table
    5. If key exists and hash matches, return stored response
    6. If key exists and hash does not match, reject request
    7. If key does not exist, insert key with IN_PROGRESS
    8. Create order inside transaction
    9. Store response in idempotency table
    10. Mark key as COMPLETED
    11. Return response

So, the table will now look like -

    idempotency_key: key_123
    request_hash: hash_abc
    status: COMPLETED
    response_body: {"orderId": "order_789"}
    created_at: timestamp

Now, if the client retries -

    POST /orders
    Idempotency-Key: key_123

The server simply returns the saved response -

    {"orderId": "order_789"}

It will not create another order.

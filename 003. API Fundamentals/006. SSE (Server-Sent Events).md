SSE stands for Server-Sent Events, and as the name suggests, it allows a server to continuously send updates to a client over a single long-lived HTTP connection.

The important part here is -

    Server -> Client


SSE is mainly used when a server needs to push updates to the client, but the client does not need to send messages back over the same connection.

For example, some scenarios where SSE can be used include Live notifications, Live news feed, Stock price updates, Sports score updates, Monitoring dashboards, AI response streaming, Order status updates, Build/deployment logs and more.

# WHY DO WE NEED SSE?

Normally, HTTP works using requests and responses. So, Client sends a request to the Server and then Server sends a response back to the Client.

After the server sends the response, the connection usually ends. But, sometimes, the client needs live updates from the server.

For example, if we are building a live cricket score application, if we use normal HTTP, the client may need to ask repeatedly - 

    Client -> Server: Any score update?
    Server -> Client: No

    Client -> Server: Any score update?
    Server -> Client: No

    Client -> Server: Any score update?
    Server -> Client: Yes, score changed

This is basically what polling is. It works, but it creates unnecessary requests.

With SSE, the client opens one connection, and the server keeps sending updates whenever something changes.

    Client -> Server: Open event stream

    Server -> Client: Score updated
    Server -> Client: Wicket fallen
    Server -> Client: Over completed

The client does not need to keep asking again and again.

# HOW SSE WORKS?

SSE works over HTTP. The client sends a normal HTTP request to the server -

    Client -> Server:
        GET /events

The server responds with a special content type -

    Content-Type: text/event-stream

But, unlike a normal HTTP response, the server does not close the connection immediately. It keeps the connection open and sends events whenever updates are available.

So, the flow is -

    1. Client opens an SSE connection
    2. Server accepts the connection
    3. Server keeps the connection open
    4. Server sends events over time
    5. Client receives those events
    6. Connection closes when client/server disconnects

# SIMPLE SSE EXAMPLE

Let's say we are building a food delivery application. The customer wants to see order status updates. Without SSE, the client may poll the server every few seconds -

    Client -> Server: What is my order status?
    Server -> Client: Preparing

    Client -> Server: What is my order status?
    Server -> Client: Preparing

    Client -> Server: What is my order status?
    Server -> Client: Out for delivery

With SSE, the client opens one stream -

    Client -> Server:
        GET /orders/123/events

The server keeps the connection open. Whenever the order status changes, the server sends an event.

    Server -> Client:
        Order accepted

    Server -> Client:
        Food is being prepared

    Server -> Client:
        Out for delivery

    Server -> Client:
        Delivered

This is much more efficient than repeatedly asking for the same information.

# SSE IS ONE-WAY

The most important thing to remember about SSE is that it is one-way.

The server can send events to the client but the client cannot send messages to the server over the same SSE connection.

If the client wants to send data to the server, it uses a normal HTTP request. For example -

    Client -> Server:
        POST /orders/123/cancel

    Server -> Client:
        order_cancelled event through SSE

    So, SSE is commonly used with normal HTTP APIs.

# SSE MESSAGE FORMAT

SSE messages are sent as plain text. A simple event can look like this -

    data: Hello from server

Each message usually ends with a blank line. A more detailed event can be -

    event: notification
    data: {"message": "Your order has shipped"}

The client receives this event and handles it.

SSE supports a few fields.

## 1. DATA

'data' contains the actual message. For example -

    data: New notification received

If we want to send JSON, we can send it as a text -

    data: {"orderId": "123", "status": "shipped"}

The client can then parse this JSON after receiving it.

## 2. EVENT

'event' tells the client what type of event this is.

For example -

    event: order_status
    data: {"status": "out_for_delivery"}

This helps the client handle different types of updates differently.

## 3. ID

'id' gives an identifier to an event.

For example -

    id: 101
    event: notification
    data: {"message": "New message received"}

This is useful in case of reconnection. If the connection breaks, the client can tell the server the last event ID it received.

Then the server can resume from the next event if it supports that.

# 4. RETRY

'retry' tells the client how long to wait before reconnecting if the connection breaks. For example -

    retry: 5000

This means the client should retry after 5000 milliseconds.

# EVENTSOURCE API

Browsers provide a built-in API called 'EventSource' for SSE. For example -

    const events = new EventSource("/events");

    events.onmessage = function(event) {
        console.log(event.data);
    };

If the server sends named events, the client can listen like this -

    events.addEventListener("order_status", function(event) {
        console.log(event.data);
    });

The nice thing is that EventSource automatically handles reconnection.

If the connection drops, the browser tries to reconnect.

This makes SSE simpler than raw WebSockets for many use cases.

If the server sends event IDs, the browser can also send the last received event ID when reconnecting.

This is very useful for systems where missed events matter.

But the server must support storing or replaying events.

If the server does not store old events, it cannot replay missed updates.

# SSE vs WEBSOCKETS

SSE and WebSockets are often compared because both can be used for real-time updates. But, they are different.

SSE is one-way -

    Server -> Client

WebSocket is two-way -

    Server <-> Client

SSE works over regular HTTP and is simpler for server-to-client updates.

WebSocket uses a protocol upgrade and keeps a full-duplex connection open.

So, use SSE when the server mainly needs to push updates to the client like live notifications, live feed, logs, progress updates and so on.

Use WebSockets when both sides need to send messages frequently like chat applications, multiplayer games, collaborative editing, real-time bidding and so on.

# SSE vs WEBHOOKS

SSE and Webhooks also look similar as both involve pushing events.

But, the direction and use cases are different.

SSE is usually server to browser/client.

Webhook is usually backend to backend.

For example -

    Payment Provider -> Our Backend:
        Webhook

    Our Backend -> User Browser:
        SSE update

In a real payment system, both can be used together -

    1. Stripe sends Webhook to our backend
    2. Our backend updates payment status
    3. Our backend sends SSE event to user's browser

So, 

    Webhook:
        Another system tells our backend something happened

    SSE:
        Our backend tells the client something changed

# DOWNSIDES OF SSE

SSE is useful but it has some limitations.

The first downside is that it is one-way which means the client cannot send messages to the server over the same connection. If it has to, it must use a separate HTTP request.

The second downside is the browser connection limits. Browsers limit how many open connections can exist to the same domain. This can matter if ma ny tabs are open or the app creates multiple SSE connections.

The third downside is that SSE is mostly designed for text data. It is not ideal for binary data.

The fourth downside is proxy and timeout issues. Since SSE keeps an HTTP connection open, some proxies, load balancers, or servers may close idle connections if they are not configured properly.

The fifth downside is scaling. Just like Websockets, SSE connections are long-lived. If many users are connected, the server must maintain many open connections.

So SSE is simpler than WebSockets, but it still needs careful handling at scale.

# HEARTBEATS

Sometimes, the server sends a small comment or heartbeat message to keep the connection alive.

For example -

    : heartbeat

This is not treated as a normal event by the client. It is just used to prevent proxies or load balancers from thinking that the connection is idle.

The server may send this every few seconds of minutes.

For example -

    : keep-alive

This is useful because long-lived connections can be closed by the infrastructure if there is no activity.

# SCALING SSE

Scaling SSE has some similar problems to scaling WebSockets.

Suppose, we have three application servers -

    App Server 1
    App Server 2
    App Server 3

A client opens an SSE connection to App Server 1 -

    Client -> Load Balancer -> App Server 1

Now, let's say an event is created on App Server 3 but the client is connected to App Server 1.

App Server 1 needs to know about that event so it can send it to the client.

So we may need a shared messaging layer. Some common options are Redis Pub/Sub, Kafka, RabbitMQ, NATS, etc.

So, the flow may looks like this -

    Event created on App Server 3
        -> Message Broker
        -> App Server 1
        -> SSE Client

This allows events to reach clients no matter which server they are connected to.

SSE connections usually go through a load balancer. The load balancer must support long-lived HTTP connections. Important load balancer settings include -

    1. Idle timeout
    2. Keep-alive timeout
    3. Maximum connections
    4. Buffering behavior

Buffering is especially important. Some proxies buffer HTTP responses before sending them to the client. But, SSE needs events to be sent immediately. If buffering is enabled, updates may be delayed. So, for SSE, response buffering is often disabled.

# AUTHENTICATION

SSE endpoints should usually be authenticated. So, if the client connects to -

    GET /notifications/stream

The server should know which user is connecting.

Authentication can be done using 0

    1. Cookies
    2. Session authentication
    3. Authorization tokens
    4. Short-lived signed URLs

One important browser limitation is that the native EventSource API does not easily allow custom headers in all cases. So if we need to send authentication details, cookies are commonly used.

Another option is to use a signed URL. For example -

    GET /events?token=short_lived_token

But tokens in URLs should be handled carefully because URLs can appear in logs.

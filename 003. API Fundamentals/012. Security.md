API Security means protecting the APIs from unauthorized access, abuse, data leaks, and attacks.

An API exposes your system to the outside world. Hence, if the API is not secure, attackers can -

    1. Read private data
    2. Modify data they should not control
    3. Delete resources
    4. Abuse expensive endpoints
    5. Impersonate users
    6. Overload the system
    7. Exploit bugs in input handling

A secure API should make sure that only the right users and systems can perform the right actions on the right resources.

Modern applications are heavily API-driven. For example -

    Browser app -> API
    Mobile app -> API
    Partner system -> API
    Internal service -> API

The API often has access to important data like User profiles, Orders, Payments, Messages, Documents, Admin actions, and Business reports.

If the API becomes insecure, the frontend design will not save us. Attackers can call the API directly using tools like curl, Postman, Bots, scripts etc.

So, API security should be enforced on the server side. Never rely on the frontend to protect sensitive actions.

# BASIC API SECURITY PRINCIPLES

Good API security usually follows these principles -

    1. Use HTTPS everywhere
    2. Authenticate users and clients
    3. Authorize every sensitive action
    4. Validate all inputs
    5. Do not expose internal details
    6. Apply rate limits
    7. Log and monitor suspicious activity
    8. Use least privilege
    9. Protect secrets and tokens
    10. Keep dependencies and infrastructure updated

Security is not one feature.

It is many small protections working together.

# BROKEN OBJECT LEVEL AUTHORIZATION

Also known as BOLA, it happens when an API does not properly check access to individual objects. For example -

    GET /users/100
    GET /users/101

If a user 100 can access user 101's private data just by changing the ID, that is BOLA. Another example is -

    DELETE /documents/55

If any logged in user can delete any document by changing the document ID, authorization is broken. So, always check resource ownership or permissions. For example -

    Does current_user own this document?
    Is current_user part of this organization?
    Does current_user have admin permission?

# BROKEN FUNCTION LEVEL AUTHORIZATION

Also known as BFLA, it happens when a user can access functions they should not be allowed to use. For example -

    Normal user calls:
        DELETE /admin/users/123

If the API only hides the admin button in the frontend but does not protect the backend endpoint, the user may still call it directly.

Another example is -

    POST /products/55/discount

Only sellers and admins should set discounts. If any logged-in uer can call this API, the function-level authorization is broken.

To prevent this -

    1. Protect admin endpoints on the server
    2. Check roles and permissions
    3. Do not rely on frontend hiding
    4. Apply authorization consistently

# INPUT VALIDATION

APIs should never blindly trust inputs from the clients. Every request body, query parameter, path parameter, and header should be treated as untrusted.

For example -

    POST /users
    {
        "email": "abc@example.com",
        "age": -500
    }

The server should validate -

    Is email valid?
    Is age allowed?
    Is required data present?
    Is field type correct?
    Is value within allowed range?

Input validation helps prevent bugs, crashes, injection attacks, and bad data.

A good API should reject invalid requests early.

    400 Bad Request

    OR

    422 Unprocessable Entity

# ALLOWLIST VALIDATION

Whenever possible, use allowlists instead of blocklists. For example, if an API accepts a 'sort' field. The bad approach is -

    Reject dangerous fields only

A better approach is -

    Allow only:
        name
        price
        createdAt

So, if the client sends -

    sort=password

Reject it.

Allowlist validation is safer because it only permits known good values. This is especially useful for -

    1. Sorting fields
    2. Filter fields
    3. File types
    4. Redirect URLs
    5. Enum values
    6. User roles

# INJECTION ATTACKS

Injection attacks happen when untrusted input is treated as code or commands. For example -

    1. SQL injection
    2. NoSQL injection
    3. Command injection
    4. LDAP injection
    5. Template injection

For example, if login code creates an SQL like this -

    SELECT * FROM users WHERE email = '${email}'

An attacker may send -

    ' OR '1'='1

This can change the meaning of the query. So, to prevent injection attacks -

    1. Use parameterized queries
    2. Use ORM safely
    3. Avoid building queries with string concatenation
    4. Validate inputs
    5. Escape output where needed
    6. Limit database permissions

Parameterized queries are one of the most important protections against SQL injection.

# MASS ASSIGNMENT

It happens when an API blindly accepts client-provided fields and saved them directly.

For example, suppose a user update API accepts -

    PATCH /users/me
    {
        "name": "Arvind",
        "role": "admin"
    }

If the backend blindly updates all the fields, the user may become the admin. This is dangerous. So, to prevent this -

    1. Allow only expected fields
    2. Ignore or reject protected fields
    3. Use separate request DTOs
    4. Never trust client-provided roles, permissions, or ownership fields. For example, role, isAdmin, balance, ownerId, and status should usually be controlled by the server.

# RATE LIMITING

It protects APIs from abuse and overload. For example -

    100 requests per minute per user

If the user exceeds the limit, we can return -

    429 Too Many Requests

It does not replace authentication or authorization. It just adds an extra layer of protection.

# REQUEST SIZE LIMITS

APIs should limit the request size. Without limits, attackers can send huge payloads and consume memory, CPU, or disk.

For example -

    POST /upload
    Body size: 10 GB

If the server tries to process it, it may crash or become slow. So, set limits for -

    1. JSON body size
    2. File upload size
    3. Header size
    4. Query string length
    5. Number of items in arrays

# CORS

CORS stands for Cross Origin Resource Sharing. It controls which browser origins are allowed to call an API from the frontend javascript.

For example, if the frontend is hosted at -

    https://app.example.com

And the API is at -

    https://api.example.com

The API may allow -

    Access-Control-Allow-Origin: https://app.example.com

CORS is important for browser-based apps. But CORS is not a complete security mechanism. It only affects browsers. Non-browser clients like curl, Postman, scripts, and backend services can still call the API. So, do not use CORS as a replacement for authentication.

# CSRF

CSRF stands for Cross-Site Request Forgery.

It mainly affects cookie-based authentication. Let's say a user is logged in to bank.com. Their browser has a session cookie.

If they visit a malicious website, the website may try to trigger a request to bank.com -

    Malicious Site -> Browser -> bank.com/transfer

The browser may automatically include bank.com cookies.

To protect against CSRF -

    1. Use SameSite cookies
    2. Use CSRF tokens
    3. Check Origin and Referer headers
    4. Avoid unsafe state-changing GET requests

CSRF is less common with APIs that use Authorization headers because browsers do not automatically attach those headers.

But for cookie-based APIs, CSRF protection is a must.

# XSS

XSS stands for Cross-Site Scripting.

It is usually thought of as a frontend issue, but it can affect API security too.

It an attacker can run JavaScript in the user's browser, they may be able to call APIs as that user.

For example -

    Malicious script -> GET /users/me
    Malicious script -> POST /transfer

If tokens are stored in localStorage, XSS can also steal them. To reduce XSS impact -

    1. Use HttpOnly cookies for sensitive tokens when possible
    2. Sanitize user-generated content
    3. Escape output properly
    4. Use Content Security Policy
    5. Avoid storing long-lived tokens in browser-accessible storage

# SECRETS MANAGEMENTS

APIs often need secrets like Database passwords, JWT signing keys, API keys etc. These should not be hardcoded in the code. For example -

    const JWT_SECRET = "my-secret-key";

A better option is -

    Store secrets in:
        Environment variables
        Secret manager
        Vault
        Cloud KMS/Secrets Manager

Secrets should be rotated periodically and if a secret is leaked, it should be revoked and replaced.

# OWASP

OWASP maintains a list of common API security risks. Some important ones are -

    1. Broken Object Level Authorization
    2. Broken Authentication
    3. Broken Object Property Level Authorization
    4. Unrestricted Resource Consumption
    5. Broken Function Level Authorization
    6. Server-Side Request Forgery
    7. Security Misconfiguration
    8. Improper Inventory Management
    9. Unsafe Consumption of APIs

You do not need to memorize all of this but the main idea is that -

    Most API security problems come from weak authorization, weak authentication, excessive data exposure, poor validation, and uncontrolled resource usage.

# SSRF

SSRF stands for Server Side Request Forgery. It happens when an attacker can make the server send requests to locations chosen by the attacker. For example if an API accepts a URL -

    POST /fetch-url
    {
        "url": "https://example.com/image.png"
    }

If the server fetches any URL blindly, an attacker can send -

    http://localhost:8080/admin

Or cloud metadata URLs. The server may access internal resources that the attacker cannot directly reach.

To prevent SSRF -

    1. Validate and allowlist domains
    2. Block private IP ranges
    3. Block localhost/internal addresses
    4. Avoid fetching arbitrary URLs
    5. Use network restrictions
    6. Disable redirects or validate redirected URLs

# COMMON MISTAKES

Some common API security mistakes are -

    1. Relying on frontend checks
    2. Checking authentication but not authorization
    3. Missing resource ownership checks
    4. Trusting client-provided roles or prices
    5. Exposing sensitive fields in responses
    6. Returning stack traces in errors
    7. Not rate limiting login or expensive endpoints
    8. Not verifying Webhook signatures
    9. Storing secrets in code
    10. Allowing huge request bodies
    11. Using overly broad CORS rules
    12. Treating internal APIs as automatically trusted
    13. Not monitoring suspicious behavior

A simple rule is to never trust the client. The server should verify everything important.

# EXAMPLE SYSTEM DESIGN

Let's say we are designing a secure API for an e-commerce application.

The request flow can be -

    Client -> HTTPS -> API Gateway -> Backend Service -> Database

At the API Gateway -

    1. Enforce HTTPS
    2. Validate access token
    3. Apply rate limits
    4. Enforce request size limits
    5. Add request ID
    6. Log basic request metadata

At the the Backend -

    1. Validate request body
    2. Check resource-level authorization
    3. Apply business rules
    4. Use parameterized database queries
    5. Return safe response fields
    6. Return safe error messages

For example -

    GET /orders/123

The API Gateway checks if the token is valid. The Order Service checks if the order belongs to the current user. The Database access uses a parameterized query.

Finally, the response includes only safe fields -

    {
        "id": "123",
        "status": "paid",
        "items": [...]
    }

It does not include internal fraud notes, payment provider secrets, or database metadata.

So remember -

    Do not trust the client.

    Verify identity, verify permission, validate input, and expose only what is necessary.

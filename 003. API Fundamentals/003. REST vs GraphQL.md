In the basics section we learnt that REST and GraphQL are two of the different ways to design APIs.

Both are used by clients to request data from a server, but they work in different ways.

While REST is resource-oriented, GraphQL is query-oriented.

In REST, the server exposes multiple endpoints, and each endpoint usually represents a resource. For example -

    GET /users/10
    GET /users/10/orders
    GET /products/55
    POST /orders

In GraphQL, the server usually exposes one endpoint, and the client sends a query describing exactly what data it wants. For example -

    POST /graphql

The query may look like this -

    {
        user(id: 10) {
            name
            email
            orders {
                id
                totalAmount
            }
        }
    }

So, the core idea is that - 

    REST gives fixed endpoints
    GraphQL gives flexible queries

# REST API

REST stands for Representational State Transfer. In REST we think in terms of 'resources.'

A resource can be anything like -

    1. User
    2. Product
    3. Order
    4. Payment
    5. Comment

Each resource is usually exposed through a URL. For example -

    /users
    /users/10
    /products
    /products/55
    /orders

The HTTP method tells what action we want to perform.

    GET     fetch data
    POST    create something
    PUT     replace something
    PATCH   partially update something
    DELETE  remove something

So in REST, the endpoint tells what resource we are dealing with, and the HTTP method tells what operation we want to perform.

# GRAPHQL API

GraphQL is an API query language. In GraphQL, the client asks for exactly the data it needs. So, instead of having many endpoints, GraphQL usually has one endpoint -

    /graphql

The client sends a query to this endpoint.

For example -

    {
        user(id: 10) {
            name
            email
        }
    }

This means -

    Give me the user with id 10, but only return name and email.

If the client also needs orders -

    {
        user(id: 10) {
            name
            email
            orders {
                id
                status
                totalAmount
            }
        }
    }

So, GraphQL gives more control to the client. The server defines what data is available, but the client decides what shape of data it wants.

# THE DIFFERENCES 

## OVER-FETCHING

Over-fetching means the client receives more data than it actually needs. This can happen in REST. For example, suppose the frontend only needs the user's name. It calls -

    GET /users/10

But, the server responds with -

    {
        "id": 10,
        "name": "Arjun",
        "email": "arjun@example.com",
        "phone": "9876543210",
        "address": "Delhi",
        "createdAt": "2026-01-01",
        "lastLogin": "2026-06-01"
    }

The frontend only needed name, but it got many extra fields. This is over-fetching.

In GraphQL, the client can ask only for name.

    {
        user(id: 10) {
            name
        }
    }

The response may be -

    {
        "data": {
            "user": {
                "name": "Arjun"
            }
        }
    }

So GraphQL can reduce over-fetching.

## UNDER-FETCHING

Under-fetching means one API call does not return enough data, so the client needs to make more API calls. This can also happen in REST.

For example, suppose we want to show an order details page. This page needs -

    1. Order details
    2. User details
    3. Product details
    4. Payment status

In REST, we may need multiple API calls -

    GET /orders/100
    GET /users/10
    GET /products/55
    GET /payments/900

So, one screen may require many round trips. This is under-fetching.

In GraphQL, the client can ask for all related data in one query -

    {
        order(id: 100) {
            id
            status
            user {
                name
                email
            }
            products {
                name
                price
            }
            payment {
                status
            }
        }
    }

So GraphQL can reduce under-fetching by allowing the client to fetch related data in one request.

## ENDPOINTS

REST usually has many endpoints. For example -

    GET /users
    GET /users/10
    GET /users/10/orders
    GET /products
    GET /products/55
    POST /orders
    PATCH /orders/100
    DELETE /orders/100

GraphQL usually has one endpoint -

    POST /graphql

## RESPONSE SHAPE

In REST, the server decides the response shape. For example -

    GET /users/10

The backend may always return the same user object. If the frontend wants fewer fields, the backend may need to create a new endpoint or add query parameters. For example -

    GET /users/10?fields=name,email

But this depends on whether the API supports it.

In GraphQL, the client decides the response shape. If the client wants name and email, it asks for name and email. If it wants name, email, and orders, it asks for those fields.

This is one of the biggest reasons GraphQL is useful for frontend-heavy applications.

## VERSIONING

REST APIs often use versioning. For example -

    /v1/users/10
    /v2/users/10

When the API changes significantly, a new version may be created. This allows old clients to keep using the old version while new clients use the new version.

GraphQL usually handles versioning differently. Instead of creating /v1 and /v2 endpoints, GraphQL schemas evolve over time.

For example, old fields can be marked as deprecated.

    oldField: String @deprecated(reason: "Use newField instead")

Clients can slowly move away from the old fields.

So REST often uses endpoint versioning, while GraphQL usually uses schema evolution.

## CACHING

REST works very well with HTTP caching. For example -

    GET /products/55

This is a normal HTTP GET request, so browsers, CDNs, and proxies can cache it easily. REST can use cache headers like -

    Cache-Control
    ETag
    Last-Modified

GraphQL caching is more complex.

Since many GraphQL requests go to the same endpoint -

    POST /graphql

traditional HTTP caching is not as straightforward.

Caching is still possible in GraphQL, but it often needs client-side caching, persisted queries, or special GraphQL-aware caching systems.

So, REST is usually simpler when we want to use standard HTTP caching.

## PERFORMANCE

GraphQL can improve performance by reducing over-fetching and under-fetching. The client can get exactly the data it needs in fewer requests.

This is very useful for mobile apps or complex frontend screens. But GraphQL can also create performance problems if not handled carefully. For example, a client may send a very expensive nested query -

    {
        users {
            orders {
                products {
                    reviews {
                        comments {
                            author {
                                profile {
                                    address
                                }
                            }
                        }
                    }
                }
            }
        }
    }

This kind of query can put heavy load on the backend. So GraphQL APIs often need query depth limits, complexity limits, rate limiting, and careful resolver design.

REST is usually more predictable because each endpoint has a fixed behavior.

For example -

    GET /products/55

The backend knows exactly what this endpoint does. So REST can be easier to optimize and monitor.

## ERROR HANDLING

REST uses HTTP status codes very directly -

    200 OK
    201 Created
    400 Bad Request
    401 Unauthorized
    404 Not Found
    500 Internal Server Error

GraphQL often returns HTTP 200 even when part of the query failed. The response may contain both data and errors -

    {
        "data": {
            "user": null
        },
        "errors": [
            {
                "message": "User not found"
            }
        ]
    }

This is because a GraphQL query may request many fields, and only some of them may fail. So error handling in GraphQL is different from REST.

# WHEN TO USE WHICH?

REST is a good choice when -

    1. The API is simple and resource-based
    2. We want to use standard HTTP features
    3. We need simple caching with CDNs or browsers
    4. The data requirements are predictable
    5. We want easier debugging and monitoring
    6. The team wants a simple and widely understood approach

For example, REST is very good for CRUD-based applications. If our application mostly deals with resources like users, products, orders, comments, and payments, REST is usually a natural fit.

GraphQL is a good choice when -

    1. Different clients need different data shapes
    2. The frontend has complex data requirements
    3. We want to reduce over-fetching and under-fetching
    4. Mobile clients need to reduce network calls
    5. The product has many frontend screens with different data needs
    6. We want a strongly typed schema for client-server communication

For example, suppose we have web app, Android app, and iOS app. Each one may need slightly different data. The web app may need more fields. The mobile app may need fewer fields. GraphQL can allow each client to ask for exactly what it needs.

# REST vs GraphQL IN MICROSERVICES

GraphQL is often used as an API layer in front of multiple backend services. For example -

    Client -> GraphQL API   -> User Service
                            -> Product Service
                            -> Order Service
                            -> Payment Service

The client sends one GraphQL query and then the GraphQL layer fetches data from multiple services and combines the response.                  

So GraphQL can act like a flexible API gateway for frontend data fetching.

But this also means the GraphQL layer can become complex.

It needs to know how to fetch data from different services, handle failures, optimize queries, and avoid performance problems like N+1 queries.

REST can also be used in microservices, where each service exposes its own REST API. For example -

    User Service exposes /users
    Product Service exposes /products
    Order Service exposes /orders

Then an API Gateway can route client requests to the correct service.

# N+1 PROBLEM IN GRAPHQL

One common issue in GraphQL is the N+1 problem. Suppose the client asks for a list of users and their orders -

    {
        users {
            name
            orders {
                id
                amount
            }
        }
    }

The GraphQL resolver may first fetch all users -

    1 query to fetch users

Then for each user, it may separately fetch all the orders -

    1 query for orders of user 1
    1 query for orders of user 2
    1 query for orders of user 3
    1 query for orders of user 4

This means, if there are 100 users, this may become 101 queries. This is called the N+1 problem. It can make GraphQL slow if resolvers are not designed carefully. A common solution is batching and caching using tools like DataLoader.

# SUMMARY

REST and GraphQL are both ways to design APIs.

REST exposes resources through multiple endpoints and uses HTTP methods like GET, POST, PUT, PATCH, and DELETE.

GraphQL exposes a query layer where the client asks for exactly the data it needs.

REST is simple, predictable, cache-friendly, and works very well for resource-based APIs.

GraphQL is flexible, frontend-friendly, and useful when different clients need different shapes of data.

REST is usually easier to understand and operate.

GraphQL gives more power to the client, but that power also needs more control on the backend.

In the last section, we went over the basics of an API. Now, when it comes to APIs, one very popular concept is the API Gateway.

Simply put, an API Gateway is a server that sits between the client and the backend services.

Instead of the client directly calling many different backend services, the client calls the API Gateway. The API Gateway then forwards the request to the correct backend service.

So, instead of -

    Client -> Backend Service

The flow now is -

    Client -> API Gateway -> Backend Service

For example, suppose we have an e-commerce application with many backend services -

    1. User Service
    2. Product Service
    3. Cart Service
    4. Order Service
    5. Payment Service
    6. Notification Service

Without an API Gateway, the frontend may need to know the address of every service. For example -

    /user-service/users/10
    /product-service/products/55
    /cart-service/cart
    /order-service/orders
    /payment-service/payments

This can become messy. The client now knows too much about the internal backend structure. If we later rename a service, move it to another server, change its port, or split it into multiple services, the client may also need changes.

With an API Gateway, the client only talks to one entry point. For example -

    GET /users/10
    GET /products/55
    GET /cart
    POST /orders
    POST /payments

The API Gateway receives these requests and internally routes them to the correct service.

# WHY AN API GATEWAY IS NEEDED?

An API Gateway is useful because it hides the internal structure of the backend system from the client.

The client does not need to know how many services exist behind the scenes.

It only needs to know the public API exposed by the gateway.

This is especially useful in microservices architecture.

In a monolithic application, there may be only one backend server. But in microservices, the backend is split into many small services.

If the client directly talks to all those services, the system becomes hard to manage.

An API Gateway solves this by acting as a common front door.

# RESPONSIBILITIES OF AN API GATEWAY

An API Gateway can do many things and some common responsibilities include -

    1. Request routing
    2. Authentication
    3. Authorization
    4. Rate limiting
    5. Load balancing
    6. Request transformation
    7. Response aggregation
    8. Logging and monitoring
    9. Caching
    10. SSL termination

## 1. REQUEST ROUTING

Request routing means sending the incoming request to the correct backend service. For example -

    GET /users/10       -> User Service
    GET /products/55    -> Product Service
    POST /orders        -> Order Service
    POST /payments      -> Payment Service

The client does not need to know where these services are running. The gateway handles that internally.

## 2. AUTHENTICATION

Authentication means checking who the user is. For example, when a request comes with a token -

    Authorization: Bearer abc123

The API Gateway can validate this token before forwarding the request to backend services. This is useful because every service does not need to repeat the same authentication logic. Instead, the gateway can handle authentication at the edge.

If the token is invalid, the gateway can reject the request immediately with '401 Unauthorized.'

## 3. AUTHORIZATION

Authorization means checking whether the user is allowed to perform a specific action.

If Authentication asks -

    Who are you?

Then Authorization asks -

    Are you allowed to do this?

For example, a normal user may be allowed to view their own order. But only an admin may be allowed to view all orders.

The API Gateway can sometimes perform basic authorization checks. However, deep business-specific authorization is often still handled inside the service itself.

For example, the Order Service may need to check whether this user really owns this order.

So, API Gateway can help with authorization, but not all authorization should blindly be moved there.

## 4. RATE LIMITING

Rate limiting means controlling how many requests a client can send in a given time. For example -

    100 requests per minute per user

If a user or client sends too many requests, the API Gateway can reject extra requests -

    429 Too Many Requests

This protects backend services from overload and abuse.

Rate limiting is very useful for public APIs. For example, if we expose an API to external developers, we may want to limit how many requests each API key can make.

## 5. LOAD BALANCING

Sometimes there are multiple instances of the same backend service. For example, we may have three Product Service instances -

    Product Service 1
    Product Service 2
    Product Service 3

The API Gateway can distribute requests across these instances. This helps improve performance and availability. If one instance is unhealthy, the gateway can stop sending requests to it.

## 6. REQUEST TRANSFORMATION

Sometimes the client sends data in one format, but the backend service expects another format. The API Gateway can transform the request before forwarding it.

For example, the client may send -

    {
        "userId": 10
    }

But, the backend may expect -

    {
        "id": 10
    }

The gateway can convert one format into another. Similarly, it can also transform responses before sending them back to the client. This is useful when we want to keep the public API stable even if internal services change.

## 7. RESPONSE AGGREGATION

Sometimes one client request needs data from multiple backend services. For example, suppose the frontend wants to show an order details page. It may need -

    1. Order details from Order Service
    2. User details from User Service
    3. Product details from Product Service
    4. Payment status from Payment Service

Without aggregation, the frontend may need to make many separate API calls. With an API Gateway, the client can call one endpoint -

    GET /order-summary/123

The API Gateway can internally call multiple services, combine the data, and return one response. This reduces the number of round trips between client and server.

But we should be careful.

If the gateway starts containing too much business logic, it can become a bottleneck or a mini-monolith.

So, response aggregation is useful, but it should be used carefully.

## 8. LOGGING AND MONITORING

The API Gateway is a good place to collect logs and metrics because all client requests pass through it. With it, we can track things like -

    1. Number of requests
    2. Error rates
    3. Latency
    4. Most used endpoints
    5. Rate-limited clients
    6. Authentication failures

This helps us understand how the system is behaving. For example, if the Product Service is slow, we may see higher latency for product-related APIs.

## 9. CACHING

The API Gateway can cache responses for frequently requested data. For example -

    GET /products/55

If product details do not change very often, the gateway can cache the response for some time. Then repeated requests can be served from the cache instead of going to the backend service every time.

This reduces load on backend services and improves response time.

But caching must be used carefully because stale data can become a problem. For example, we should be careful while caching prices, inventory, or user-specific data.

## 10. SSL TERMINATION

SSL termination means the API Gateway handles HTTPS encryption and decryption. The client talks to the gateway over HTTPS -

    Client -> HTTPS -> API Gateway

After that, the gateway may talk to internal services inside a private network.

This simplifies certificate management because certificates can be handled at the gateway level.

# EXAMPLE

Let's say we have an e-commerce application. Suppose, the frontend wants to place an order so the client sends -

    POST /orders

The request first reaches the API Gateway.

The API Gateway may do these steps -

    1. Check if the user is authenticated
    2. Check rate limits
    3. Route the request to the Order Service
    4. Log request information
    5. Return the response back to the client

The Order Service may then talk to other services like Payment Service, Inventory Service, and Notification Service. But from the client's point of view, it only called -

    POST /orders

The client does not need to know the full internal flow.

# API GATEWAY vs LOAD BALANCER

API Gateway and Load Balancer may look similar because both receive traffic and forward it somewhere else.

But they are not exactly the same.

A Load Balancer mainly distributes traffic across multiple instances of the same service. For example -

    Product Service Instance 1
    Product Service Instance 2
    Product Service Instance 3

Its main job is traffic distribution.

An API Gateway does more than that.

It can handle routing, authentication, rate limiting, request transformation, response aggregation, caching, and monitoring. So -

    Load Balancer distributes traffic
    API Gateway manages API traffic

In many real systems, both are used together.

# DOWNSIDES OF AN API GATEWAY

API Gateway is useful, but it also has some downsides. The first downside is that it can become a single point of failure. If the API Gateway goes down, clients may not be able to access backend services.

So, in production, we usually run multiple gateway instances behind a load balancer.

The second downside is added latency.

Since every request goes through the gateway, it adds one extra network hop. Usually this is acceptable, but it still matters for very latency-sensitive systems.

The third downside is complexity.

If we put too much logic inside the gateway, it can become hard to maintain.

For example, if authentication, routing, transformations, aggregations, business rules, and custom logic all live inside the gateway, the gateway itself becomes a complex system.

So, the API Gateway should handle cross-cutting concerns, but core business logic should usually stay inside the actual services.
